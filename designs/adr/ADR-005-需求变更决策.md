# ADR-005：v1 → v2 需求变更决策

**决策编号**：ADR-005
**决策标题**：基于RCR逆向校验的v1→v2系统性修正决策
**决策状态**：已采纳
**决策日期**：2026-06-22
**决策者**：A5-需求验证智能体
**影响范围**：项目v2版本的整体架构、接口契约、业务逻辑和前端体验

---

## 1. 上下文

### 1.1 背景

项目v1版本（基线BL-20260622-01）的代码已完成初始开发，涵盖了8个业务模块的基本CRUD操作。为了验证代码实现与设计文档的一致性，项目团队进行了系统性的逆向校验（RCR-001）和四维度质量校验（QAR-001），覆盖了架构层、模块层、拓扑层、契约层、代码工程层和需求溯源层六个校验维度。

### 1.2 校验结果

| 校验维度 | 满分 | 得分 | 判定 |
|---------|:----:|:----:|:----:|
| 架构层校验 | 20 | 12 | 不通过 |
| 模块层校验 | 15 | 12 | 通过 |
| 拓扑层校验 | 15 | 10 | 不通过 |
| 契约层校验 | 20 | 8 | 不通过 |
| 代码工程层校验 | 15 | 11 | 通过 |
| 需求溯源层校验 | 15 | 9 | 不通过 |
| **总分** | **100** | **62** | **不通过（62 < 75）** |

### 1.3 主要发现

逆向校验共发现15项设计-代码漂移（Drift），其中：
- **严重等级**：5项（33%）— 涉及事务缺失、Entity暴露、API路径不一致、魔法值、跨模块集成缺失
- **高等级**：5项（33%）— 涉及包名不一致、接口路径偏差、参数校验缺失、收据API未暴露
- **中等级**：3项（20%）— 涉及Entity替代DTO、审批流不完整、签到接口不完整
- **低等级**：2项（13%）— 涉及硬编码常量、类命名不规范

### 1.4 四维度质量评估

| 质量维度 | 满分 | 得分 | 判定 |
|---------|:----:|:----:|:----:|
| 完备性 | 25 | 10 | 不足 |
| 正确性 | 25 | 16 | 一般 |
| 一致性 | 25 | 8 | 严重不足 |
| 有效性 | 25 | 10 | 不足 |
| **总分** | **100** | **44** | **不合格** |

## 2. 决策问题

是否采纳基于RCR逆向校验和QAR质量校验发现的问题而整理的8条变更需求（CR-001至CR-008），对v1版本进行系统性修正，形成v2版本？

需要决定的子问题：
1. 8条CR的采纳范围（全部采纳/部分采纳）
2. 变更优先级排序和分阶段实施策略
3. 是否建立新的需求基线（BL-20260622-02）

---

## 3. 决策选项

### 选项A：全部采纳（推荐）

采纳全部8条CR（CR-001至CR-008），覆盖RCR发现的全部15项漂移，进行系统性修正。

**优势**：
- 一次性解决所有已知设计-代码漂移问题
- 建立完整的前后端契约一致性
- 补齐关键业务闭环（退费审批、优惠叠加、跨模块集成）
- 从根本上提升系统质量，从62分提升至预计85+分

**劣势**：
- 变更范围大（30+后端文件、20+前端文件）
- 工作量约32人日
- 破坏性变更集中，需谨慎管理兼容性
- 需要协调前后端同步上线

### 选项B：仅修复严重问题

仅采纳P0优先级的4条CR（CR-001至CR-004），暂缓P1和P2级别的变更。

**优势**：
- 变更范围可控（约20人日）
- 优先解决最严重的安全和契约问题
- 上线风险相对可控

**劣势**：
- 魔法值、枚举不一致等问题遗留
- 跨模块集成仍然缺失
- 需要后续再进行一轮变更，总成本更高
- 前端体验问题未解决

### 选项C：分批实施

将8条CR分为两个迭代：迭代1（P0+N0）修复严重问题；迭代2（P1+P2）做功能增强和优化。

**优势**：
- 风险可控，每次变更范围较小
- 每次上线可独立验证
- 团队可逐步适应变更节奏

**劣势**：
- 总周期延长
- 两次迭代之间可能产生新的不一致
- 需要维护中间状态的兼容性

---

## 4. 决策

**选择选项A：全部采纳**。

### 4.1 决策理由

1. **系统性问题的解决需要系统性方案**：15项漂移相互关联（如CR-001和CR-002共同影响API层），分批实施可能导致中间状态前后端不一致的窗口期延长
2. **避免重复工作**：如果分批实施，第一次变更后需要重新进行逆向校验，第二次变更后又需要再次校验，总校验成本更高
3. **基线冻结的完整性**：建立BL-20260622-02作为"全部已知问题已修复"的基线，为后续开发提供干净的起点
4. **达标要求**：v1版本综合得分62分（不合格），仅修复严重问题预计只能达到70-72分（仍低于75分通过线），需要全面修复才能达标

### 4.2 变更优先级排序

| 实施批次 | 变更编号 | 变更内容 | 优先级 | 预计工期 |
|:------:|:------:|---------|:------:|:------:|
| **第一批**<br>(基础设施) | CR-003 | 补充@Transactional事务注解 | P0 | 1.5天 |
| | CR-007 | 统一枚举值定义 | P1 | 3天 |
| **第二批**<br>(接口层) | CR-001 | 修正Controller暴露Entity（引入DTO/VO） | P0 | 7天 |
| | CR-002 | 统一API路径命名规范 | P0 | 3天 |
| **第三批**<br>(业务层) | CR-005 | 优化收费金额一致性（优惠叠加引擎） | P1 | 4天 |
| | CR-004 | 增加退费审批闭环 | P0 | 5天 |
| | CR-006 | 增加跨模块Service集成 | P1 | 5天 |
| **第四批**<br>(体验层) | CR-008 | 增强前端错误处理与状态展示 | P2 | 3天 |

**总工期估算**：约32人日（含测试和文档更新）

### 4.3 CR-001和CR-002同步实施说明

CR-001（Entity→DTO）和CR-002（API路径统一）是关联度最高的两项变更，均涉及API契约层。建议同步实施以最小化前后端不兼容窗口期：

- 后端先完成CR-001的DTO创建和CR-002的路径修改
- 前端同步适配新的响应结构和API路径
- 联调通过后统一上线

## 5. 后果分析

### 5.1 正面后果

| 后果 | 说明 |
|------|------|
| 数据安全性提升 | Entity不再暴露到前端，敏感字段（password、deleted、createBy等）得到系统性防护 |
| 前后端可正常联调 | API路径统一为复数RESTful规范后，前端请求能够正确到达后端Controller |
| 数据一致性保障 | 所有写操作均添加了事务注解，数据库操作失败时自动回滚 |
| 退费业务闭环 | 完整的退费审批流程（申请→审核→审批→退款）实现，满足REQ-FIN-013需求 |
| 费用计算准确性 | 优惠叠加引擎确保多优惠场景下金额计算准确；退费金额按课时消耗自动计算 |
| 报表数据真实可用 | Report模块集成各业务模块后，报表不再使用mock数据 |
| 前端用户体验提升 | Loading/Empty/Error状态全覆盖，关键操作有二次确认 |
| 技术债务清零 | 15项RCR漂移全部修复，v2基线从62分提升至预计85+分 |

### 5.2 负面后果

| 后果 | 缓解措施 |
|------|---------|
| 破坏性变更导致已有集成测试全部失效 | 在变更实施的同时编写新的集成测试，建立测试回归检查表 |
| 跨模块Service注入可能引入循环依赖 | 在设计中评估依赖方向，必要时使用@Lazy注解或事件驱动模式解耦 |
| 枚举值变更需数据迁移 | 提供枚举值映射兼容层；编写数据迁移脚本并验证 |
| 变更范围大可能导致遗漏 | 基于RCR漂移项制定逐项回归校验清单（CRR） |
| 前端需要同步大面积更新 | 制定前端升级指南，批次同步前后端变更 |

### 5.3 中立后果

| 后果 | 说明 |
|------|------|
| 模块接口层代码量增加 | 每个业务模块新增DTO/VO类和转换方法，代码总量增加约15-20%。但这是必要的架构修正，长期来看降低维护成本 |
| 包名（com.edufee）与MDS定义（com.edufeems）仍不一致 | 全局包名重构涉及所有文件的import路径变更，工作量巨大且风险高。暂时维持com.edufee，在MDS中更新约定或在下一次架构升级时统一修正 |

## 上下文（Context）

### 项目背景

教育培训机构教务收费管理系统（EduFeeMS）旨在为中小型教育培训机构提供一套完整的教务管理与收费一体化解决方案。系统需要覆盖学员管理、课程管理、班级管理、教师管理、收费管理、考勤管理和报表统计等核心业务场景。

### 项目规模与约束

| 约束维度 | 说明 |
|----------|------|
| **用户规模** | 预计支撑10,000+在册学员，日常活跃用户约200-500人 |
| **并发要求** | 峰值并发用户约50人，主要集中于开学报名季和考勤签到时段 |
| **开发资源** | 单人开发，需在有限时间内完成核心功能交付 |
| **部署环境** | 支持多校区独立部署，各校区数据需物理或逻辑隔离 |
| **技术约束** | 需运行于Linux服务器环境，支持主流浏览器访问 |
| **预算约束** | 有限预算，优先使用开源技术栈，避免商业授权费用 |

### 核心非功能需求

- **多校区数据隔离**：不同校区的业务数据（学员、课程、财务）必须相互隔离，避免跨校区数据泄露
- **可维护性**：单人开发模式下，代码结构需清晰、易于理解和维护
- **扩展性**：未来可能增加新的业务模块（如在线选课、家长端查询等）
- **安全性**：涉及财务数据，需严格的权限控制和操作审计

---

## 决策（Decision）

**采用「分层架构（Layered Architecture）+ 前后端分离」作为系统整体架构风格。**

### 架构描述

```
┌─────────────────────────────────────────────────────┐
│                    前端层 (Frontend)                  │
│              Vue.js SPA + Element Plus               │
│          Pages → Components → Base Components        │
├─────────────────────────────────────────────────────┤
│                   RESTful API (JSON)                 │
│               JWT 认证 + 权限拦截                      │
├─────────────────────────────────────────────────────┤
│                 Controller 层 (控制层)                 │
│         请求参数校验、路由分发、统一响应封装              │
├─────────────────────────────────────────────────────┤
│                  Service 层 (业务层)                   │
│         核心业务逻辑、事务管理、模块间调用               │
├─────────────────────────────────────────────────────┤
│               Repository 层 (数据访问层)               │
│            MyBatis-Plus ORM、数据库操作                │
├─────────────────────────────────────────────────────┤
│                 MySQL 8.0 + Redis                    │
│                    数据存储层                          │
└─────────────────────────────────────────────────────┘
```

### 核心设计原则

1. **严格分层**：Controller → Service → Repository，依赖方向单向向下
2. **前后端分离**：前端SPA与后端RESTful API通过HTTP/HTTPS通信
3. **模块化组织**：后端按功能领域划分为独立模块（auth、student、course、class、teacher、fee、attendance、report、common）
4. **接口契约**：模块间通过Service接口通信，降低耦合

## 备选方案与评估（Alternatives）

采用五维度加权评分法对3种候选架构风格进行评估：

### 评分权重说明

| 评估维度 | 权重 | 说明 |
|----------|------|------|
| 功能适用性 | 25% | 架构风格是否满足当前及可预见的功能需求 |
| 开发效率 | 25% | 单人开发模式下的开发速度和上手难度 |
| 可维护性 | 20% | 代码结构的清晰度、模块的可理解性和可修改性 |
| 性能与可扩展性 | 15% | 系统的并发处理能力和水平扩展潜力 |
| 部署运维复杂度 | 15% | 部署、监控、运维的复杂度和资源需求 |

### 方案对比矩阵

#### 方案A：分层架构（Layered Architecture）— 最终选择

| 评估维度 | 得分(1-10) | 加权得分 | 评分理由 |
|----------|:----------:|:--------:|----------|
| 功能适用性 | 9 | 2.25 | 分层架构完全满足业务需求，Controller处理HTTP请求、Service封装业务逻辑、Repository管理数据访问，职责清晰，与Vue.js前端通过REST API对接自然流畅 |
| 开发效率 | 10 | 2.50 | 分层架构学习成本最低，Spring Boot生态成熟，MyBatis-Plus简化数据访问，单人开发可快速完成功能迭代，无需处理分布式复杂性问题 |
| 可维护性 | 8 | 1.60 | 代码按层组织、职责明确，新增功能只需在各层添加对应代码；但模块间通过Service直接调用，可能导致隐式耦合，需通过编码规范约束 |
| 性能与可扩展性 | 6 | 0.90 | 单体部署模式下，水平扩展能力有限（只能整体扩容）；但通过多实例+负载均衡可满足50并发的性能需求；垂直扩展（升级服务器）可解决短期性能问题 |
| 部署运维复杂度 | 9 | 1.35 | 单体部署，一个JAR包启动即可，部署运维成本极低，适合单人运维模式；多校区可各部署一套独立实例实现数据隔离 |
| **加权总分** | | **8.60** | |

#### 方案B：微服务架构（Microservices）

| 评估维度 | 得分(1-10) | 加权得分 | 评分理由 |
|----------|:----------:|:--------:|----------|
| 功能适用性 | 7 | 1.75 | 微服务可在服务级别实现多校区隔离和数据独立部署，但服务的拆分粒度与业务模块并非一一对应，部分业务（如考勤和收费）需要跨服务事务协调 |
| 开发效率 | 3 | 0.75 | 微服务引入服务发现、配置中心、分布式事务、API网关等大量基础设施，单人开发需额外学习Spring Cloud/Dubbo生态，开发效率大幅降低 |
| 可维护性 | 5 | 1.00 | 服务拆分后各服务独立，理论上可维护性高；但分布式系统的调试、日志追踪、链路监控复杂度显著增加，对单人维护是巨大负担 |
| 性能与可扩展性 | 9 | 1.35 | 各服务可独立扩展，按需扩容高并发模块（如考勤签到）；但50并发用户场景下，微服务架构的扩展能力严重过剩 |
| 部署运维复杂度 | 2 | 0.30 | 需要Kubernetes/Docker Compose等多容器编排能力，CI/CD流水线复杂度高，监控和日志聚合需要ELK/Prometheus等额外组件，运维成本远超项目预算 |
| **加权总分** | | **5.15** | |

#### 方案C：模块化单体（Modular Monolith）

| 评估维度 | 得分(1-10) | 加权得分 | 评分理由 |
|----------|:----------:|:--------:|----------|
| 功能适用性 | 8 | 2.00 | 模块化单体与分层架构功能满足度相似，模块间通过明确接口通信，依赖关系清晰；但在多校区数据隔离上不如分层架构灵活（分层方案可通过部署多实例解决） |
| 开发效率 | 9 | 2.25 | Maven多模块项目结构清晰，模块间依赖可显式声明，编译隔离；但模块间接口设计和依赖管理增加了一定的前期设计工作 |
| 可维护性 | 9 | 1.80 | 模块边界明确，每个模块职责单一，符合高内聚低耦合原则；模块间通过接口依赖而非实现依赖，未来可平滑演进为微服务 |
| 性能与可扩展性 | 6 | 0.90 | 同样是单体部署，扩展能力与分层架构相当；但模块化设计为未来拆分为微服务提供了基础（只需将模块接口改为RPC/HTTP调用） |
| 部署运维复杂度 | 7 | 1.05 | 最终产出一个JAR包，部署运维简单；但多模块构建配置较复杂，模块间版本管理需额外关注 |
| **加权总分** | | **8.00** | |

### 方案对比总结

```
方案A（分层架构）:    ████████████████████ 8.60 ★ 最终选择
方案C（模块化单体）:  ████████████████░░░░ 8.00
方案B（微服务架构）:  ██████████░░░░░░░░░░ 5.15
```

### 最终决策理由

**方案A（分层架构）** 以最高加权总分当选，主要原因：

1. **开发效率最优**：分层架构的学习曲线最低，Spring Boot成熟生态可快速交付，最适合单人开发资源约束
2. **部署运维最简单**：单JAR包部署，运维成本趋近于零
3. **功能适用性足够**：50并发用户场景下，分层架构完全满足性能需求
4. **风险可控**：多校区通过多实例部署实现数据隔离，简单可靠
5. **演进路径清晰**：未来若业务增长需要，可从分层架构平滑演进到模块化单体，再演进到微服务

**注**：方案C（模块化单体）得分8.00，与方案A接近。在实际编码中，系统将采用Maven多模块项目结构（结合方案C的优点），即"模块化分层架构"，兼顾分层架构的开发效率和模块化单体的模块隔离优势。

## 后果（Consequences）

### 正面影响

| 影响 | 说明 |
|------|------|
| **高开发效率** | 分层架构是Spring Boot标准实践，开发工具链成熟，IDE插件丰富，可快速完成业务功能开发 |
| **低学习成本** | 开发人员无需掌握分布式系统知识，聚焦业务逻辑实现即可 |
| **快速部署** | 单JAR包启动，CI/CD流程简单，适合频繁的功能迭代和快速上线 |
| **调试便利** | 所有代码在同一进程运行，断点调试、日志追踪无跨进程复杂度 |
| **成本可控** | 无需额外的中间件（服务注册中心、配置中心等），服务器资源需求低 |

### 负面影响

| 影响 | 说明 | 缓解措施 |
|------|------|----------|
| **模块间耦合风险** | Service层模块间直接调用可能导致隐式耦合，修改一个模块可能影响其他模块 | 制定模块间接口契约规范，通过接口编程降低实现耦合；代码审查中重点关注跨模块调用 |
| **单体扩展受限** | 只能整体水平扩展，无法按模块粒度扩缩容 | 当前50并发的需求下，单实例足够；未来若业务增长，可按模块拆分为独立服务 |
| **部署耦合** | 修改任意模块需要全量打包部署，影响其他模块的可用性 | 采用蓝绿部署或滚动升级策略减少停机时间；非高频部署场景下影响可控 |
| **多校区数据隔离** | 单体部署模式下需通过campus_id字段实现逻辑隔离，安全性依赖代码实现正确性 | 在Service层统一注入campus_id过滤条件；定期进行数据隔离安全审计 |
| **代码膨胀风险** | 长期演进下，代码量增长可能导致编译和启动时间增加 | 定期进行代码重构；使用Maven多模块项目结构保持代码组织清晰 |

### 风险缓解计划

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:----:|:----:|----------|
| 模块间耦合导致修改成本上升 | 中 | 高 | 建立模块依赖拓扑图，明确允许和禁止的依赖方向；每季度进行架构合规审查 |
| 业务增长超出单体承载能力 | 低 | 中 | 保留模块化项目结构，为未来向微服务演进预留接口抽象层 |
| 多校区数据泄漏 | 低 | 极高 | Service层统一注入campus_id条件；敏感操作（退费、财务查询）强制二次鉴权；数据库层面启用行级安全策略（若MySQL版本支持） |
| 单人开发的知识孤岛 | 高 | 中 | 完善的代码注释和架构文档；关键设计决策通过ADR记录 |

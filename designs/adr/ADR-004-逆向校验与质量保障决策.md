## 1. 上下文

### 1.1 问题陈述

EduFeeMS项目v1版本已完成初步代码实现（9个后端模块 + 前端SPA），但面临以下质量保障挑战：

1. **设计-实现一致性未知**：MDS/DTS/OAS/TLCD等设计文档与v1源代码之间的一致性从未经过系统性校验
2. **代码质量不可见**：缺乏量化的代码质量标准，无法判断v1版本是否达到可交付状态
3. **漂移风险累积**：开发过程中的架构漂移（如Entity直接暴露、API路径不一致、跨模块集成缺失）未被及时发现和修复
4. **需求覆盖盲区**：98条功能需求与代码实现的映射关系不透明，无法评估交付完整度

### 1.2 利益相关者

| 角色 | 关注点 |
|------|--------|
| 架构师 | 架构分层是否被遵循，模块依赖是否违反拓扑 |
| 技术负责人 | 代码质量是否达标，是否存在技术债务 |
| 项目经理 | v1版本是否可交付，还差多少工作量 |
| 测试团队 | 哪些模块需要重点测试，哪些接口存在集成风险 |
| 前端开发 | 后端API是否与OAS契约一致，联调能否正常进行 |

---

## 2. 决策

### 2.1 核心决策

**采用"六级校验维度 + 四维度质量评估"双向校验机制，作为EduFeeMS项目v1及后续版本的强制性质量保障策略。**

### 2.2 六级校验维度（逆向校验）

从"正向设计图谱"到"逆向代码图谱"的对比校验：

| 维度 | 正向图谱 | 逆向图谱 | 校验重点 |
|:----:|---------|---------|---------|
| D1-架构层 | ASD + TLCD-C-ARCH | Controller/Service/Mapper调用链 | 分层违规、跨层穿透、事务缺失 |
| D2-模块层 | MDS包结构规范 | 实际目录结构和类命名 | 包名一致性、模块存在性、目录结构 |
| D3-拓扑层 | DTS依赖矩阵 | import语句和Spring注入 | 跨模块调用合规、禁止依赖违规 |
| D4-契约层 | OAS接口定义 | Controller方法注解 | 路径/方法/参数/响应一致性 |
| D5-代码工程层 | TLCD-C-CODE（15条） | Java源码逐行审查 | BigDecimal/日期/魔法值/Lombok/SQL等 |
| D6-需求溯源层 | SRS需求清单 | Controller+Service方法 | 需求-代码1:1映射覆盖 |

### 2.3 四维度质量评估

从软件质量属性维度独立评估：

| 维度 | 评估内容 | 评估方法 |
|:----:|---------|---------|
| Q1-完备性 | 所有需求是否有对应实现 | 需求-代码映射矩阵，覆盖率统计 |
| Q2-正确性 | 关键业务逻辑是否正确 | 代码走查：金额计算、状态流转、冲突检测 |
| Q3-一致性 | 前后端契约/数据库映射是否一致 | 路径对比、字段映射、枚举值对比 |
| Q4-有效性 | 代码是否能有效达成业务目标 | 可执行性、关键路径完整性、性能安全考量 |

## 3. 备选方案对比

### 方案A：仅做代码审查（Code Review）

| 维度 | 评估 |
|------|------|
| 实施成本 | 低——仅需人工审查代码 |
| 覆盖范围 | 窄——仅关注代码风格和明显缺陷，无法发现架构漂移和需求覆盖缺失 |
| 一致性保障 | 弱——没有设计文档作为参照基准，审查结果主观性强 |
| 可重复性 | 弱——人工审查标准因人而异 |

**结论：不采纳。** 仅做代码审查无法保证设计与实现的一致性，也无法量化评估代码质量。

### 方案B：仅做自动化测试（单元+集成测试）

| 维度 | 评估 |
|------|------|
| 实施成本 | 高——需要编写大量测试用例和维护测试代码 |
| 覆盖范围 | 中——可验证功能正确性，但无法发现架构层次违反和设计文档偏差 |
| 一致性保障 | 弱——测试只验证行为而非结构，不检测是否违反DTS依赖拓扑 |
| 可重复性 | 强——自动化执行 |

**结论：不采纳（作为主策略）。** 自动化测试与校验机制互补而非替代关系。测试无法回答"代码架构是否与设计一致"以及"是否所有需求都有对应实现"。

### 方案C：六级校验 + 四维度评估（本决策）

| 维度 | 评估 |
|------|------|
| 实施成本 | 中——需要编写RCR报告和QAR报告，v1首次投入约8-12人时 |
| 覆盖范围 | 广——覆盖架构/模块/拓扑/契约/代码/需求六大维度 |
| 一致性保障 | 强——以设计文档为基准，系统性发现漂移 |
| 可重复性 | 强——标准化校验维度，每个版本可重复执行 |
| 发现能力 | 本次校验发现15项漂移（5严重+5高+3中+2低），质量评分44/100 |

**结论：采纳。** 这是唯一能系统性回答"设计是否等于实现"和"代码质量究竟如何"的方案。

### 方案对比表

| 评估维度 | 方案A | 方案B | 方案C（采纳） |
|---------|:----:|:----:|:----:|
| 发现架构漂移 | 弱 | 弱 | **强** |
| 发现需求遗漏 | 弱 | 中 | **强** |
| API契约一致性 | 弱 | 中 | **强** |
| 代码规范合规 | 中 | 弱 | **强** |
| 业务逻辑正确性 | 中 | **强** | 中 |
| 可量化评分 | 弱 | 中 | **强** |
| 实施成本 | 低 | 高 | **中** |
| 自动化程度 | 弱 | **强** | 中 |
| 持续执行成本 | 中 | 高 | **中** |

## 4. 实施策略

### 4.1 执行节奏

| 阶段 | 时机 | 执行内容 |
|:----:|------|---------|
| v1 | 首次校验（本次） | 全量RCR + QAR，建立漂移基线 |
| v1.1 | 迭代修复后 | 增量RCR，仅重检漂移项和变更模块 |
| v2 | 重大版本 | 全量RCR + QAR，重新评分 |
| 持续 | 每个PR合入前 | 自动化架构约束检查（C-ARCH/C-CODE可自动化项） |

### 4.2 自动化路线

| 约束类型 | 当前方式 | 目标方式 |
|---------|---------|---------|
| C-ARCH-001（Controller→Mapper穿透） | 人工Grep | ArchUnit测试 |
| C-ARCH-004（@Transactional漏加） | 人工审查 | 自定义Checkstyle/PMD规则 |
| C-CODE-009（魔法值） | 人工审查 | SonarQube规则 |
| C-CODE-012（URL单复数） | 人工对比 | OpenAPI Lint + Spring MVC路径扫描 |
| DTS依赖拓扑 | 人工对比 | JDepend + 自定义拓扑检查脚本 |
| 需求覆盖 | 人工映射 | 需求ID-代码注解关联 + 覆盖报告生成 |

### 4.3 通过标准

| 版本阶段 | RCR最低分 | QAR最低分 | 严重漂移上限 |
|---------|:--------:|:--------:|:----------:|
| v1（MVP基线） | 60 | 40 | 无要求（建立基线） |
| v1.1（迭代修复） | 75 | 60 | 0项 |
| v2（正式版） | 85 | 75 | 0项 |
| v3+（生产版） | 90 | 85 | 0项 |

## 5. 后果分析

### 5.1 积极后果

1. **设计-实现一致性可见**：每次版本发布前可量化评估代码是否偏离设计意图
2. **技术债务显性化**：漂移清单将隐性架构问题转化为显性可追踪条目
3. **交付信心增强**：通过评分体系为项目经理提供是否可交付的客观依据
4. **知识传承**：RCR和QAR文档成为新人理解系统架构与实现差距的最佳入口
5. **持续改进循环**：每次校验→修复→再校验形成正向质量提升循环

### 5.2 消极后果与缓解措施

| 消极后果 | 风险等级 | 缓解措施 |
|---------|:--------:|---------|
| 首次全量校验投入较大（8-12人时） | 中 | v1基线建立后，后续版本仅需增量校验 |
| 可能产生大量漂移项导致团队焦虑 | 低 | 明确通过标准是渐进的，v1不要求满分 |
| 校验结论可能被用作"卡发布"的工具 | 低 | 通过标准由架构委员会共识确定，非单一个人裁决 |
| 部分校验依赖人工判断（如需求覆盖"部分覆盖"判定） | 中 | 制定覆盖度判断细则（见附录） |

### 5.3 与现有ADR的关联

| ADR | 关联方式 |
|-----|---------|
| ADR-001（架构选型-分层架构） | ASD中的分层规则是本校验使用的正向图谱基准 |
| ADR-002（技术栈-Spring Boot + Vue3） | 代码层约束中的Lombok/MyBatis-Plus/Swagger来源于技术栈选型 |
| ADR-003（数据库设计） | 数据库字段与实体属性一致性校验的数据参照基准 |
| 未来ADR-005（CI/CD流水线） | 自动化架构约束检查将集成到CI/CD流水线中 |

## 6. 附录

### 附录A：需求覆盖判定细则

| 覆盖状态 | 判定标准 |
|---------|---------|
| **完整覆盖** | Controller方法存在 AND Service实现完整 AND 核心业务逻辑无TODO |
| **部分覆盖** | Controller或Service存在骨架 BUT 关键流程标记TODO OR 缺少闭环（如审批/通知） |
| **未覆盖** | 无对应Controller方法 AND 无对应Service实现 |

### 附录B：漂移严重等级定义

| 等级 | 定义 | 示例 |
|:----:|------|------|
| **严重** | 直接导致系统不可用、数据不一致或安全漏洞 | 事务缺失、前后端路径完全不匹配、Entity暴露密码字段 |
| **高** | 导致集成困难、代码腐化加速或违反核心架构原则 | 魔法值过多、DTO规范违反、OAS接口缺失 |
| **中** | 影响可维护性或存在数据不一致风险但不立即导致故障 | 命名不规范、枚举值不一致、缺少审计日志 |
| **低** | 最佳实践违反，影响有限 | 硬编码前缀、目录结构微调 |

## 上下文（Context）

### 决策背景

在确立分层架构和技术栈后，需要为系统设计数据库层面的技术方案。数据库设计需满足以下核心需求：

- **多校区数据隔离**：不同校区的业务数据必须相互隔离，避免跨校区数据泄露
- **审计追溯**：所有数据变更需记录操作人、操作时间，支持审计追溯
- **财务数据精度**：涉及学费、退费等金额计算，需确保数值精度
- **数据可恢复性**：业务数据不可物理删除，需支持逻辑删除和误删恢复
- **查询性能**：支持按校区、时间范围、多条件组合的分页查询

### 约束条件

| 约束维度 | 说明 |
|----------|------|
| 数据库引擎 | MySQL 8.0 InnoDB（支持事务、行级锁、外键约束） |
| 部署模式 | 每校区独立数据库实例（物理隔离）或共享实例逻辑隔离 |
| 数据规模 | 单校区预计5年内学员数据<50,000条，核心业务表<500,000条 |
| 开发资源 | 单人开发，ORM框架为MyBatis-Plus |

## 决策（Decision）

### 1. 数据库引擎与字符集

| 决策项 | 选择 | 理由 |
|--------|------|------|
| **数据库** | MySQL 8.0 | LTS版本，InnoDB引擎成熟，Window Functions和CTE支持报表查询 |
| **存储引擎** | InnoDB | 支持事务（ACID）、行级锁、外键约束，适合财务数据场景 |
| **字符集** | utf8mb4 | 支持完整的Unicode字符集（含emoji），避免utf8mb3的编码陷阱 |
| **排序规则** | utf8mb4_unicode_ci | 基于Unicode标准的排序，不区分大小写，适合中文多语言场景 |
| **时区** | Asia/Shanghai (+08:00) | 统一使用中国标准时间，避免时区转换问题 |

### 2. 通用字段规范

所有数据表必须包含以下通用字段：

```sql
-- 所有表的通用字段结构
id          BIGINT       PRIMARY KEY AUTO_INCREMENT  COMMENT '主键ID',
create_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
update_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
create_by   BIGINT       DEFAULT NULL COMMENT '创建人ID',
update_by   BIGINT       DEFAULT NULL COMMENT '更新人ID',
is_deleted  TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '逻辑删除：0-未删除，1-已删除'
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT | 自增主键，使用分布式ID策略（雪花算法或数据库自增） |
| `create_time` | DATETIME | 记录创建时间，默认当前时间，不可手动修改 |
| `update_time` | DATETIME | 记录最后更新时间，每次UPDATE自动更新 |
| `create_by` | BIGINT | 创建人用户ID，用于审计追溯 |
| `update_by` | BIGINT | 最后更新人用户ID，用于审计追溯 |
| `is_deleted` | TINYINT(1) | 逻辑删除标记，0=正常，1=已删除。所有查询默认过滤is_deleted=1的记录 |

### 3. 多校区数据隔离方案

**决策：优先采用逻辑隔离（campus_id字段），预留物理隔离能力。**

| 隔离层级 | 实现方式 | 适用场景 |
|----------|----------|----------|
| **逻辑隔离** | 所有校区相关表添加`campus_id`字段，查询时通过MyBatis-Plus自动注入campus_id过滤条件 | 中小规模部署，校区数量<10 |
| **物理隔离** | 每校区独立数据库实例，通过Spring Boot多数据源动态路由 | 大规模部署，校区数量>10或对数据隔离有严格合规要求 |

**实现策略**：
- 所有业务表（student、course、class、teacher、fee、attendance等）均包含`campus_id`字段
- Service层通过ThreadLocal获取当前请求的campus_id，自动注入查询条件
- 超级管理员可切换校区视角查看数据
- 物理隔离通过修改数据源配置切换，应用层代码无需改动

### 4. 金额字段类型选择

**决策：所有金额字段统一使用 `DECIMAL(10,2)`。**

| 方案 | 精度 | 存储 | 风险 |
|------|:----:|------|------|
| **DECIMAL(10,2)** ★ | 精确 | 5字节 | 无精度丢失 |
| FLOAT | 近似 | 4字节 | 浮点运算误差，财务计算不可接受 |
| DOUBLE | 近似 | 8字节 | 浮点运算误差，财务计算不可接受 |
| BIGINT(分) | 精确 | 8字节 | 需额外转换，代码复杂度增加 |

**DECIMAL(10,2) 覆盖范围**：-99,999,999.99 ~ 99,999,999.99，满足教育培训机构的收费金额范围需求。

### 5. 逻辑删除策略

**决策：使用 `is_deleted` 字段实现逻辑删除，禁止物理删除业务数据。**

**实现方式**：

```
MyBatis-Plus 逻辑删除配置：
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: is_deleted   # 逻辑删除字段名
      logic-delete-value: 1             # 已删除值
      logic-not-delete-value: 0         # 未删除值
```

| 操作 | 实际SQL效果 |
|------|------------|
| `deleteById(id)` | `UPDATE table SET is_deleted = 1 WHERE id = ?` |
| `selectList(query)` | 自动追加 `WHERE is_deleted = 0` |
| 恢复删除 | 手动执行 `UPDATE table SET is_deleted = 0 WHERE id = ?` |

**适用范围**：所有业务数据表。系统级配置表（如系统参数）可物理删除。

### 6. 索引设计原则

| 原则 | 说明 |
|------|------|
| **主键索引** | 所有表使用自增BIGINT主键 |
| **campus_id索引** | 所有校区相关表必须建立`idx_{表名}_campus_id`索引 |
| **外键索引** | 所有外键字段建立索引，加速关联查询 |
| **时间索引** | 高频时间范围查询的字段（如create_time、pay_time）建立索引 |
| **唯一约束** | 业务唯一字段（身份证号、用户名、收据号）建立唯一约束 |
| **联合索引** | 高频组合查询条件（如campus_id + create_time）建立联合索引 |
| **覆盖索引** | 高频查询字段尽量使用覆盖索引，避免回表 |

## 命名规范

### 表命名规范

| 规范 | 说明 | 示例 |
|------|------|------|
| 前缀 `t_` | 所有业务表统一前缀 | `t_student_info` |
| snake_case | 小写+下划线分隔 | `t_fee_record`, `t_course_type` |
| 模块名_表名 | 体现所属模块 | `t_fee_record`（fee模块）, `t_student_info`（student模块） |

### 各模块核心表清单

| 模块 | 核心表 |

| common | （无业务表，仅提供基类字段标准） |

| student | `t_student_info`, `t_student_enrollment`, `t_student_class`, `t_student_status_log` |
| course | `t_course_category`, `t_course_info`, `t_course_hour_config`, `t_course_fee_standard` |
| class | `t_class_info`, `t_schedule`, `t_classroom`, `t_class_student` |
| teacher | `t_teacher_info`, `t_teacher_course`, `t_teacher_schedule`, `t_teacher_salary` |
| fee | `t_fee_item`, `t_fee_record`, `t_fee_refund`, `t_fee_arrearage`, `t_receipt`, `t_promotion`, `t_fee_promotion_apply` |
| attendance | `t_attendance_record`, `t_leave_request`, `t_leave_approval`, `t_attendance_stats` |
| report | `t_report_template`, `t_report_export_log` |

### 字段命名规范

| 规范 | 示例 |
|------|------|
| 字段名 snake_case | `student_name`, `fee_amount`, `campus_id` |
| 主键统一 `id` | `id BIGINT PRIMARY KEY AUTO_INCREMENT` |
| 外键：`关联表名_id` | `student_id`, `course_id`, `class_id`, `campus_id` |
| 时间字段：`_time` 后缀 | `create_time`, `update_time`, `pay_time`, `enroll_date` |
| 金额字段：`_amount` / `_price` / `_fee` 后缀 | `fee_amount`, `unit_price`, `total_fee` |
| 状态字段：`_status` 后缀 | `student_status`, `class_status`, `leave_status` |

### 索引命名规范

| 索引类型 | 命名格式 | 示例 |
|----------|----------|------|
| 普通索引 | `idx_{表名}_{字段}` | `idx_t_student_phone` |
| 唯一约束 | `uk_{表名}_{字段}` | `uk_t_student_id_card` |
| 联合索引 | `idx_{表名}_{字段1}_{字段2}` | `idx_t_fee_record_campus_pay_time` |

## 备选方案对比

### 对比：MySQL 8.0 vs PostgreSQL 15

| 对比维度 | MySQL 8.0 ★ | PostgreSQL 15 | 分析 |
|----------|:----------:|:------------:|------|
| 学习成本 | ★★★★★ (低) | ★★★☆☆ (中) | MySQL是国内Java开发者的默认选择 |
| 社区生态(国内) | ★★★★★ (主流) | ★★★☆☆ (较少) | MySQL在国内的部署量和使用率远高于PostgreSQL |
| ORM兼容 | ★★★★★ (优秀) | ★★★★☆ (良好) | MyBatis-Plus对MySQL支持最完善 |
| 运维工具 | ★★★★★ (丰富) | ★★★★☆ (较丰富) | MySQL的监控、备份、迁移工具更成熟 |
| 行级安全 | ★★☆☆☆ (需插件) | ★★★★★ (内置) | PostgreSQL原生RLS更适合多校区行级隔离 |
| JSON支持 | ★★★★☆ (良好) | ★★★★★ (优秀) | PostgreSQL的JSONB索引和查询更强大 |
| Window Functions | ★★★★☆ (良好) | ★★★★★ (优秀) | 两者都支持，PostgreSQL的窗口函数更丰富 |
| 事务隔离 | ★★★★☆ (良好) | ★★★★★ (优秀) | PostgreSQL的MVCC实现更完善 |

**结论**：选择 **MySQL 8.0**。虽有PostgreSQL在行级安全和窗口函数方面的优势，但MySQL的国内生态、运维成熟度、MyBatis-Plus兼容性和团队熟悉度更符合单人开发的实际需求。行级安全通过应用层campus_id过滤实现。

## 后果（Consequences）

### 正面影响

| 影响 | 说明 |
|------|------|
| **审计追溯完整** | 通用字段（create_by、update_by、create_time、update_time）确保每次数据变更可追溯到具体操作人和时间 |
| **数据可恢复** | 逻辑删除（is_deleted）替代物理删除，误删数据可快速恢复 |
| **金额精度可靠** | DECIMAL(10,2)类型确保财务计算零误差，避免浮点数精度问题 |
| **多校区灵活性** | campus_id字段实现逻辑隔离，未来可平滑升级为物理隔离 |
| **查询性能可控** | 合理的索引设计支撑分页查询和报表统计需求 |
| **开发效率高** | MyBatis-Plus自动处理逻辑删除过滤、通用字段填充，减少重复代码 |

### 负面影响与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **逻辑删除数据膨胀** | 长期运行后is_deleted=1的数据占用存储空间 | 定期将超过1年的已删除数据归档到历史库；建立数据归档定时任务 |
| **campus_id漏过滤** | 开发者遗漏campus_id过滤条件导致跨校区数据泄露 | 在BaseService层封装通用查询方法，自动注入campus_id条件；Code Review重点检查 |
| **DECIMAL范围限制** | DECIMAL(10,2)最大值约1亿，极端场景可能不够 | 当前教育培训收费场景下足够使用；若未来需要更大范围，通过数据库迁移脚本调整精度 |
| **通用字段维护成本** | 新增表时可能遗漏通用字段 | 制定数据库DDL模板（见附录）；使用代码生成器自动生成建表语句 |

### 数据安全策略

| 策略 | 说明 |
|------|------|
| **定期备份** | 每日全量备份 + 每小时增量备份（binlog） |
| **敏感字段加密** | 身份证号、手机号等敏感信息使用AES加密存储 |
| **SQL注入防护** | MyBatis-Plus参数化查询 + 禁止XML中使用`${}`拼接 |
| **操作日志** | 关键表（fee_record、refund）的变更记录写入独立的操作日志表 |

## 数据库DDL模板

```sql
-- 示例：收费记录表
CREATE TABLE t_fee_record (
    id            BIGINT         NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    student_id    BIGINT         NOT NULL                 COMMENT '学员ID',
    fee_item_id   BIGINT         NOT NULL                 COMMENT '收费项目ID',
    campus_id     BIGINT         NOT NULL                 COMMENT '校区ID',
    amount_payable DECIMAL(10,2) NOT NULL                 COMMENT '应收金额',
    amount_paid   DECIMAL(10,2) NOT NULL                 COMMENT '实收金额',
    pay_method    VARCHAR(20)    NOT NULL                 COMMENT '支付方式：CASH/ALIPAY/WECHAT/BANK_TRANSFER',
    pay_time      DATETIME       DEFAULT NULL             COMMENT '支付时间',
    receipt_no    VARCHAR(50)    DEFAULT NULL             COMMENT '收据号',
    remark        VARCHAR(500)   DEFAULT NULL             COMMENT '备注',
    create_time   DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time   DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    create_by     BIGINT         DEFAULT NULL             COMMENT '创建人ID',
    update_by     BIGINT         DEFAULT NULL             COMMENT '更新人ID',
    is_deleted    TINYINT(1)     NOT NULL DEFAULT 0       COMMENT '逻辑删除：0-未删除，1-已删除',
    PRIMARY KEY (id),
    INDEX idx_t_fee_record_student_id (student_id),
    INDEX idx_t_fee_record_campus_id (campus_id),
    INDEX idx_t_fee_record_pay_time (pay_time),
    INDEX idx_t_fee_record_campus_pay_time (campus_id, pay_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='收费记录表';
```

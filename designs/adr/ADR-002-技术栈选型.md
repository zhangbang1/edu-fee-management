## 上下文（Context）

### 决策背景

在确立分层架构+前后端分离的架构风格（ADR-001）后，需要为每一层选择具体的技术实现栈。选型需满足以下约束：

- 单人开发资源，需选择学习成本低、社区活跃、文档完善的技术
- 项目预算有限，优先选择开源免费的技术方案
- 系统需长期维护（3-5年），技术栈需有稳定的版本支持周期
- 技术栈之间需良好兼容，避免版本冲突

### 选型原则

1. **成熟度优先**：优先选择经过大规模生产验证的技术
2. **社区活跃度**：GitHub Star、Issue响应速度、更新频率
3. **学习成本**：单人开发的掌握时间和上手难度
4. **生态兼容性**：与已选技术的集成难度

## 备选方案对比

### 对比一：ORM框架 — MyBatis-Plus vs JPA/Hibernate

| 对比维度 | MyBatis-Plus | JPA/Hibernate | 分析 |
|----------|:------------:|:------------:|------|
| 学习曲线 | ★★★★★ (低) | ★★★☆☆ (中) | MyBatis-Plus基于SQL，开发人员无需学习HQL/JPQL |
| SQL控制力 | ★★★★★ (强) | ★★☆☆☆ (弱) | MyBatis-Plus保留原生SQL灵活性，复杂查询直接写SQL |
| CRUD效率 | ★★★★★ (高) | ★★★★☆ (较高) | 两者都提供自动CRUD，MyBatis-Plus的LambdaQueryWrapper更直观 |
| 分页支持 | ★★★★★ (优秀) | ★★★☆☆ (一般) | MyBatis-Plus分页插件开箱即用，配置简单 |
| 逻辑删除 | ★★★★★ (内置) | ★★☆☆☆ (需手动) | MyBatis-Plus内置逻辑删除注解 |
| 多表关联 | ★★★★☆ (灵活) | ★★★★★ (自动) | JPA自动关联映射更优雅；MyBatis-Plus需手动编写关联查询XML |
| 社区生态(国内) | ★★★★★ (活跃) | ★★★☆☆ (一般) | MyBatis-Plus国内社区活跃，中文文档完善，问题响应快 |
| 迁移成本 | ★★★★★ (低) | ★★★☆☆ (中) | 团队通常已有SQL经验，MyBatis-Plus几乎零学习成本 |

**结论**：选择 **MyBatis-Plus**。在SQL控制力、国内社区支持、分页和逻辑删除等业务场景契合度方面优于JPA/Hibernate。虽然多表关联查询不如JPA自动映射优雅，但复杂查询场景通过XML Mapper可完全控制SQL执行计划，对于需要性能优化的财务统计查询尤为重要。

### 对比二：前端框架 — Vue 3 vs React 18

| 对比维度 | Vue 3 | React 18 | 分析 |
|----------|:-----:|:--------:|------|
| 学习曲线 | ★★★★★ (低) | ★★★☆☆ (中) | Vue模板语法接近HTML，直观易懂；React需要掌握JSX和Hook概念 |
| 官方生态 | ★★★★★ (完整) | ★★★☆☆ (碎片) | Vue Router、Pinia、Vite均为官方维护；React生态依赖社区选择 |
| UI组件库(国内) | ★★★★★ (Element Plus) | ★★★★☆ (Ant Design) | Element Plus是国内最流行的Vue3组件库，企业后台系统首选 |
| 响应式系统 | ★★★★★ (自动) | ★★★☆☆ (手动) | Vue3的ref/reactive自动追踪依赖；React需手动管理useState/useEffect |
| TypeScript支持 | ★★★★☆ (良好) | ★★★★★ (优秀) | React的TS支持更深入；Vue3 Composition API的TS支持持续改进 |
| 打包构建 | ★★★★★ (Vite) | ★★★★☆ (Vite/Next) | Vite同为二者提供构建，体验相当 |
| 国内招聘/社区 | ★★★★★ (广泛) | ★★★★☆ (较多) | 国内中小项目Vue使用率更高，与企业后台场景更匹配 |

**结论**：选择 **Vue 3**。对于企业后台管理系统场景，Vue 3配合Element Plus的学习成本最低，可快速交付高质量界面。模板语法直观、响应式系统自动追踪、官方生态完整，适合单人开发的快速迭代节奏。

### 对比三：构建工具 — Maven vs Gradle

| 对比维度 | Maven | Gradle | 分析 |
|----------|:-----:|:------:|------|
| 学习曲线 | ★★★★★ (低) | ★★★☆☆ (中) | Maven的XML配置直观，约定优于配置；Gradle的Groovy/Kotlin DSL有额外学习成本 |
| 构建速度 | ★★★☆☆ (一般) | ★★★★★ (快) | Gradle的增量构建和缓存机制更快；但对于中等规模项目差异不显著 |
| 依赖管理 | ★★★★★ (优秀) | ★★★★★ (优秀) | 两者都支持中央仓库，Maven的依赖解析机制更广泛被理解 |
| 插件生态 | ★★★★★ (丰富) | ★★★★☆ (较丰富) | Maven插件生态更成熟，Spring Boot官方Maven插件支持最好 |
| IDE集成 | ★★★★★ (无缝) | ★★★★☆ (良好) | IntelliJ IDEA对Maven的原生支持最完善 |
| 多模块项目 | ★★★★★ (成熟) | ★★★★☆ (较成熟) | Maven的多模块父子继承机制清晰，国内使用最广泛 |
| 国内社区 | ★★★★★ (主流) | ★★★☆☆ (较少) | Maven在国内Java项目中的使用率远高于Gradle |

**结论**：选择 **Maven**。Maven作为Java生态事实标准，学习成本最低，国内社区支持最广泛，Spring Boot官方Maven插件集成最完善。Gradle的增量构建优势在中型项目中体现不明显，而多模块项目管理和IDE集成方面Maven更为成熟稳定。

## 后果（Consequences）

### 正面影响

| 影响 | 说明 |
|------|------|
| **开发效率高** | Spring Boot自动配置 + MyBatis-Plus代码生成器可快速生成CRUD代码；Vue3 + Element Plus组件丰富，前端开发效率高 |
| **学习成本低** | 所选技术栈（Maven、MyBatis-Plus、Vue3）均为国内主流技术，中文文档完善，社区资源丰富 |
| **维护成本低** | Java 11 LTS和Spring Boot 2.7.x均为长期支持版本，3-5年内无需大版本升级 |
| **问题排查便利** | 技术栈普及率高，遇到问题可快速搜索到解决方案，降低单人开发的排查成本 |
| **工具链成熟** | Lombok减少样板代码，Hutool提供丰富工具类，FastJson2高性能序列化，全面提升开发体验 |

### 负面影响与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **Spring Boot 2.7.x EOL** | 2.7.x预计2025年结束开源支持，之后需迁移至3.x | 预留一个季度时间进行Spring Boot 3.x迁移评估；项目初期使用2.7.x确保稳定性，中期规划迁移 |
| **MyBatis-Plus SQL注入风险** | 动态SQL拼接不当可能导致SQL注入 | 严格使用LambdaQueryWrapper参数化查询；禁止在XML Mapper中使用 `${}` 拼接用户输入 |
| **Vue3 与 Element Plus 版本兼容** | Element Plus大版本更新可能引入Breaking Changes | 锁定依赖版本号，升级前进行充分回归测试 |
| **Java 11 后续版本升级** | 较新Spring Boot版本可能要求Java 17+ | 业务稳定后规划Java版本升级路径（11→17→21 LTS） |

### 技术债务管理

| 债务项 | 严重程度 | 预计清偿时间 | 说明 |
|--------|:--------:|:------------:|------|
| Spring Boot 2.7.x → 3.x 迁移 | 中 | 2027 H1 | 需适配Jakarta EE命名空间变更 |
| Java 11 → 17 升级 | 低 | 2027 H2 | 利用LTS版本窗口进行升级 |
| Element Plus 大版本升级 | 低 | 按需 | 关注Breaking Changes，及时适配 |

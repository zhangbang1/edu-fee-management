# 约束提示词构造指南

> 项目：教育培训机构教务收费管理系统（EduFeeMS）
> 版本：v1.0
> 日期：2026-06-22
> 用途：指导AI代码生成时如何注入三层约束，确保生成的代码符合项目规范

---

## 一、概述

在AI辅助代码生成场景中，必须将项目的设计约束以结构化方式注入提示词（Prompt），使生成的代码自动遵循架构、模块、代码三层约束。本指南定义四类记忆的注入策略及完整提示词模板。

### 1.1 四类记忆模型

```
┌─────────────────────────────────────────────────────────┐
│                    AI 代码生成上下文                       │
├───────────────┬──────────────┬──────────────┬────────────┤
│  知识图谱记忆  │  契约记忆     │  规则记忆     │  实现记忆   │
│  (MDS+DTS+   │  (OpenAPI    │  (TLCD三层   │  (已有代码   │
│   ASD)       │   YAML)      │   约束)      │   图谱)     │
├───────────────┴──────────────┴──────────────┴────────────┤
│                    具体任务提示词                           │
└─────────────────────────────────────────────────────────┘
```

## 二、知识图谱注入（项目记忆）

### 2.1 三类结构化文档

| 缩写 | 全称 | 存放位置 | 内容说明 |
|------|------|----------|----------|
| MDS | Module Dependency Structure | `designs/diagrams/` | 模块依赖关系图，定义模块间调用白名单 |
| DTS | Data Type Specification | `designs/diagrams/` | 数据类型规范，定义实体、DTO、VO的数据结构 |
| ASD | Architecture Structure Diagram | `designs/diagrams/` | 架构层次图，定义Controller-Service-Repository分层 |

### 2.2 注入方式

在提示词中注入MDS+DTS+ASD时，采用以下格式：

```markdown
## 项目知识图谱

### 模块依赖结构（MDS）
本项目包含以下模块及其依赖关系：
- common：无依赖
- auth：依赖 common
- student：依赖 common, auth, course, class
- course：依赖 common, auth
- class：依赖 common, auth, course
- fee：依赖 common, auth, student, course
- attendance：依赖 common, auth, student, class

### 架构分层（ASD）
- Controller层：接收HTTP请求，参数校验，调用Service
- Service层：业务逻辑，事务管理
- Repository/Mapper层：数据库访问
- Client层：外部API调用封装

### 数据类型规范（DTS）
- 数据库实体（Entity）：com.xxx.edu.entity.*
- 数据传输对象（DTO）：com.xxx.edu.dto.*
- 视图对象（VO）：com.xxx.edu.vo.*
- 统一响应：com.xxx.common.R<T>
```

## 三、契约记忆注入（OpenAPI YAML）

### 3.1 契约文件

契约文件路径：`designs/contracts/openapi.yaml`

该文件定义了完整的RESTful API契约，包括：
- 6个接口分组（auth、students、courses、classes、fees、attendances）
- 每个端点的请求/响应Schema
- 可复用的数据模型定义（15+ Schema）
- JWT认证方案

### 3.2 注入方式

在提示词中注入契约记忆时，需要指定生成目标对应的端点定义：

```markdown
## 接口契约

请严格遵循以下API契约定义：

### 端点定义
[从 openapi.yaml 中提取目标端点的 path、method、parameters、requestBody、responses]

### 数据模型
[从 openapi.yaml 中提取相关的 Schema 定义]

### 约束
- 所有响应必须使用 R<T> 包装
- 分页查询使用 IPage<T> 返回
- JWT Token 从 Authorization: Bearer <token> 获取
```

## 四、规则记忆注入（TLCD三层约束）

### 4.1 约束文件

约束文件路径：`designs/diagrams/三层约束设计-TLCD.md`

### 4.2 按目标模块筛选约束

生成特定模块代码时，只需注入与该模块相关的约束：

**架构层约束（C-ARCH）**：始终全部注入（全局规则）

**模块层约束（C-MOD）**：仅注入目标模块的约束

**代码层约束（C-CODE）**：始终全部注入（编码规范）

### 4.3 注入格式

```markdown
## 三层约束（TLCD）

### 架构层约束（必须遵守）
- C-ARCH-001：Controller层禁止直接操作数据库（必须通过Service层）
- C-ARCH-002：所有数据库操作必须通过Repository/Mapper接口
- C-ARCH-003：Controller层只做参数校验和路由，不包含业务逻辑
- C-ARCH-004：Service层方法必须包含事务注解（@Transactional），除只读查询外
- C-ARCH-005：所有外部API调用必须通过独立的Client类封装
- C-ARCH-006：跨模块调用只能通过Service接口，不能直接调用Repository
- C-ARCH-007：前端不能直接访问数据库，所有数据请求通过API
- C-ARCH-008：敏感数据（密码、手机号）在日志中必须脱敏
- C-ARCH-009：所有异常统一由GlobalExceptionHandler处理，禁止Controller内try-catch吞异常
- C-ARCH-010：分页查询统一使用IPage<T>作为返回类型
- C-ARCH-011：所有API响应必须使用统一响应格式R<T>（code/message/data）
- C-ARCH-012：数据库实体类禁止被直接暴露到Controller层，必须通过DTO/VO转换

### student模块约束
- C-MOD-011：依赖common、auth、course、class模块
- C-MOD-012：StudentService只能通过ClassService接口操作班级数据
- C-MOD-013：学员分班/转班/退学操作必须校验班级状态
- C-MOD-014：学员信息中的手机号在查询列表时默认脱敏显示
- C-MOD-015：学员退学操作必须关联退费流程

### 代码层约束
- C-CODE-001~C-CODE-015：全部编码规范（详见TLCD文档）
```

## 五、实现记忆注入（已有代码图谱）

### 5.1 目的

当项目中已存在部分代码时，需要向AI提供已有代码的上下文，避免生成冲突代码，同时可以引用已有的工具类、常量、异常类。

### 5.2 应提供的已有代码信息

| 信息类型 | 说明 | 示例 |
|----------|------|------|
| 已有Entity | 目标模块引用的实体类 | Student、ClassInfo、Course |
| 已有Service接口 | 目标模块依赖的Service | ClassService、CourseService |
| 已有工具类 | 可用工具方法 | DateUtil、StringUtil、BeanCopyUtil |
| 已有常量 | Constants中的常量定义 | Constant.STUDENT_STATUS_ENROLLED |
| 已有异常 | BusinessException使用方式 | throw new BusinessException(ErrorCode.XXX) |
| 已有配置 | 项目配置项 | 分页默认值、Token过期时间等 |

### 5.3 注入格式

```markdown
## 已有代码上下文

### 已有实体（仅列出目标模块需要引用的）
- Student（com.xxx.edu.entity.Student）
  - 字段：id, name, gender, phone, birthday, parentName, parentPhone, address, source, status, createTime, updateTime
  - 主键策略：自增ID
  - 自动填充：createTime, updateTime

- ClassInfo（com.xxx.edu.entity.ClassInfo）
  - 字段：id, courseId, name, capacity, enrolledCount, startDate, endDate, teacherId, classroom, status

### 已有Service接口（目标模块需要调用的）
- ClassService.findByStudentId(Long studentId)：查询学员当前班级

### 已有工具类
- R<T>：统一响应，方法：R.ok(T data), R.ok(), R.error(int code, String msg)
- BeanCopyUtil：实体与DTO转换，BeanCopyUtil.copy(source, targetClass)

### 已有异常类
- BusinessException(errorCode)：抛出业务异常，由GlobalExceptionHandler统一处理
- ErrorCode：错误码枚举，如 ErrorCode.STUDENT_NOT_FOUND
```

## 六、完整提示词模板示例

以下为生成"学员管理Controller"的完整提示词模板。

### 6.1 模板

```markdown
# 代码生成任务：学员管理 Controller

## 1. 任务描述

请为教育培训机构教务收费管理系统（EduFeeMS）的 student 模块生成 Controller 层代码。

要求生成以下端点：
1. GET /api/students — 分页查询学员列表（支持姓名、手机号、状态筛选）
2. GET /api/students/{id} — 获取学员详情（含班级、缴费摘要）
3. POST /api/students — 新增学员
4. PUT /api/students/{id} — 修改学员信息
5. POST /api/students/{id}/enroll — 学员分班
6. POST /api/students/{id}/transfer — 学员转班
7. POST /api/students/{id}/withdraw — 学员退学

## 2. 接口契约

请严格遵循以下API定义，该定义来自 designs/contracts/openapi.yaml：

### GET /api/students
- 参数：pageNum(int,默认1), pageSize(int,默认10), name(string), phone(string), status(StudentStatus枚举)
- 响应：StudentPageResponse { pageInfo, records: StudentDTO[] }

### GET /api/students/{id}
- 参数：id(long,路径参数)
- 响应：StudentDetailResponse { ...StudentDTO, currentClass, enrollmentHistory, recentPayments }

### POST /api/students
- 请求体：StudentCreateRequest { name(必填), gender, phone(必填), birthday, parentName, parentPhone, address, source, remark }
- 响应：StudentResponse { code, message, data: StudentDTO }

### PUT /api/students/{id}
- 请求体：StudentUpdateRequest（全部可选）
- 响应：StudentResponse

### POST /api/students/{id}/enroll
- 请求体：EnrollRequest { classId(必填), effectiveDate }
- 响应：EnrollResponse

### POST /api/students/{id}/transfer
- 请求体：TransferRequest { targetClassId(必填), transferDate, reason }
- 响应：TransferResponse

### POST /api/students/{id}/withdraw
- 请求体：WithdrawRequest { withdrawDate(必填), reason(必填), requireRefund }
- 响应：WithdrawResponse

## 3. 架构层约束（必须全部遵守）

- C-ARCH-001：Controller层禁止直接操作数据库（必须通过Service层）
- C-ARCH-002：所有数据库操作必须通过Repository/Mapper接口
- C-ARCH-003：Controller层只做参数校验和路由，不包含业务逻辑
- C-ARCH-004：Service层方法必须包含事务注解（@Transactional），除只读查询外
- C-ARCH-005：所有外部API调用必须通过独立的Client类封装
- C-ARCH-006：跨模块调用只能通过Service接口，不能直接调用Repository
- C-ARCH-007：前端不能直接访问数据库，所有数据请求通过API
- C-ARCH-008：敏感数据（密码、手机号）在日志中必须脱敏
- C-ARCH-009：所有异常统一由GlobalExceptionHandler处理，禁止Controller内try-catch吞异常
- C-ARCH-010：分页查询统一使用IPage<T>作为返回类型，分页参数使用Page<T>
- C-ARCH-011：所有API响应必须使用统一响应格式R<T>（code/message/data）
- C-ARCH-012：数据库实体类禁止被直接暴露到Controller层，必须通过DTO/VO转换

## 4. student模块约束

- C-MOD-011：依赖 common、auth、course、class 模块
- C-MOD-012：StudentService 只能通过 ClassService 接口操作班级数据
- C-MOD-013：学员分班/转班/退学操作必须校验班级状态（开课中/已结课）
- C-MOD-014：学员信息中的手机号在查询列表时默认脱敏显示（如 138****8000）
- C-MOD-015：学员退学操作必须关联退费流程，无退费记录不允许标记退学

## 5. 代码层约束（必须全部遵守）

- C-CODE-001：所有日期时间字段使用 LocalDateTime 类型
- C-CODE-002：所有日期格式统一为 yyyy-MM-dd HH:mm:ss
- C-CODE-003：所有金额字段使用 BigDecimal，禁止 float/double
- C-CODE-004：Controller方法使用 @Valid 注解进行参数校验
- C-CODE-005：DTO类使用 Lombok @Data 注解
- C-CODE-006：数据库实体类使用 @TableName 和 @TableId 注解
- C-CODE-007：日志使用 @Slf4j 注解，日志级别：debug/info/warn/error
- C-CODE-008：异常使用自定义 BusinessException，统一由 GlobalExceptionHandler 处理
- C-CODE-009：禁止使用魔法值，常量定义在 Constants 类中
- C-CODE-010：SQL查询返回多表数据使用 VO 类封装
- C-CODE-011：实体类使用 @TableField(fill = FieldFill.INSERT/UPDATE) 自动填充 createTime/updateTime
- C-CODE-012：RESTful URL命名为资源名复数形式，层级不超过3层
- C-CODE-013：每个 Controller 方法必须有 @Operation 注解
- C-CODE-014：SQL IN 查询参数个数不超过1000，超过时需分批查询
- C-CODE-015：禁止在循环中执行数据库查询，必须使用批量操作

## 6. 已有代码上下文

### 模块依赖（MDS）
student 模块依赖：
- common：使用 R<T> 统一响应、BusinessException 异常、Constants 常量
- auth：使用 @PreAuthorize 权限注解、JWT Token 验证（由全局拦截器处理）
- course：调用 CourseService 查询课程信息
- class：调用 ClassService 进行班级相关操作

### 已有实体
- Student：id, name, gender, phone, birthday, parentName, parentPhone, address, source, status, createTime, updateTime
- EnrollmentRecord：id, studentId, classId, enrollDate, status, createTime
- TransferRecord：id, studentId, fromClassId, toClassId, transferDate, reason
- WithdrawRecord：id, studentId, withdrawDate, reason, requireRefund, refundId

### 已有Service接口
- StudentService：待实现
- ClassService.findByStudentId(Long studentId)：查询学员当前班级
- ClassService.checkAvailability(Long classId)：检查班级容量
- ClassService.checkStatus(Long classId)：检查班级状态
- CourseService.getById(Long courseId)：查询课程

### 已有工具
- R.ok(T data)：返回成功响应
- R.error(int code, String msg)：返回错误响应
- BeanCopyUtil.copy(source, targetClass)：对象拷贝
- PhoneUtil.maskPhone(String phone)：手机号脱敏

## 7. 生成要求

1. 包名：com.example.edufeems.controller
2. 类名：StudentController
3. 使用 @RestController 和 @RequestMapping("/api/students")
4. 所有方法使用 @Operation 注解描述
5. 写操作的接口使用 @PreAuthorize 控制权限
6. 不要实现 Service 层逻辑，仅在 Controller 中调用 Service 方法
7. 使用 @Valid 进行参数校验
8. 日志使用 log.info() / log.error()，敏感信息脱敏

## 8. 输出格式

请输出完整的 Java 类文件代码，包含：
- package 声明
- import 语句
- 类注解
- 全部7个端点方法（每个方法只做参数校验、Service调用、结果包装）
```

## 七、提示词复用策略

### 7.1 按层级拆分

| 层级 | 复用方式 | 说明 |
|------|----------|------|
| 架构层约束 | 固定模板 | 所有代码生成任务共享同一份 C-ARCH 列表 |
| 模块层约束 | 按模块选择 | 根据生成目标所属模块，仅注入该模块的 C-MOD 约束 |
| 代码层约束 | 固定模板 | 所有代码生成任务共享同一份 C-CODE 列表 |
| 契约记忆 | 按端点选择 | 从 openapi.yaml 中提取目标端点定义 |
| 实现记忆 | 按上下文选择 | 根据目标模块的依赖关系，注入已有的 Entity/Service/Util |

### 7.2 按模块生成的提示词速查

| 目标模块 | 注入的 C-MOD | 注入的依赖模块 | 关联 Entity |
|----------|-------------|---------------|-------------|
| auth | C-MOD-006~010 | common | User, Role, Permission |
| student | C-MOD-011~015 | common, auth, course, class | Student, EnrollmentRecord, TransferRecord, WithdrawRecord |
| course | C-MOD-016~019 | common, auth | Course |
| class | C-MOD-020~024 | common, auth, course | ClassInfo, Schedule |
| fee | C-MOD-025~030 | common, auth, student, course | FeeRecord, RefundRecord |
| attendance | C-MOD-031~035 | common, auth, student, class | AttendanceRecord, LeaveRecord |

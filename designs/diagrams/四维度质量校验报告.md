## 2. 正确性（Correctness）

### 2.1 收费金额计算正确性审查

#### 审查对象：`FeeServiceImpl.createFeeRecord()` 和 `makePayment()`

**BigDecimal使用审查**：

```java
// FeeServiceImpl.createFeeRecord() - 金额计算链路
BigDecimal amount = feeRecord.getAmount() != null ? feeRecord.getAmount() : BigDecimal.ZERO;
BigDecimal discountAmount = feeRecord.getDiscountAmount() != null ? feeRecord.getDiscountAmount() : BigDecimal.ZERO;
BigDecimal actualAmount = amount.subtract(discountAmount);  // ✓ 使用subtract
actualAmount.setScale(2, RoundingMode.HALF_UP);             // ✓ 四舍五入到2位小数
```

| 检查项 | 结果 | 说明 |
|--------|:----:|------|
| 金额类型为BigDecimal | 通过 | FeeRecord.amount, paidAmount, unpaidAmount均为BigDecimal |
| 使用subtract/add而非运算符 | 通过 | 全程使用BigDecimal方法 |
| 设置精度的scale和rounding | 通过 | 使用HALF_UP舍入模式 |
| 负数金额校验 | 通过 | `actualAmount.compareTo(BigDecimal.ZERO) < 0` 校验 |
| 超额缴费校验 | 通过 | `payAmount.compareTo(unpaidAmount) > 0` 校验 |

**判定：收费核心计算链路正确。**

#### 审查对象：`FeeServiceImpl.applyRefund()`

```java
BigDecimal refundAmount = amount.setScale(2, RoundingMode.HALF_UP);
if (refundAmount.compareTo(payment.getAmount()) > 0) {
    throw BusinessException.badRequest("退款金额不能超过缴费金额");
}
```

| 检查项 | 结果 | 说明 |
|--------|:----:|------|
| 退费金额校验 | 通过 | 校验不超过缴费金额 |
| 已退费记录拦截 | 通过 | 检查 PAYMENT_STATUS_REFUNDED |
| 退款后金额回写 | **未实现** | TODO状态——退费后未更新paidAmount/unpaidAmount |

**判定：退费申请逻辑基本正确，但事务闭环不完整（未回写收费记录的实际金额字段）。**

## 3. 一致性（Consistency）

### 3.1 前后端API契约一致性

#### 3.1.1 URL路径一致性

| 前端API调用 | 前端请求路径 | 后端Controller路径 | 一致性 |
|------------|------------|-------------------|:------:|
| `getStudentList()` | GET `/students` | GET `/api/student/list` | **不一致** |
| `getStudentDetail(id)` | GET `/students/{id}` | GET `/api/student/{id}` | **不一致**（前缀差异） |
| `createStudent(data)` | POST `/students` | POST `/api/student` | **不一致**（前缀差异） |
| `updateStudent(id, data)` | PUT `/students/{id}` | PUT `/api/student` | **不一致**（缺少id+前缀差异） |
| `getFeeRecords(params)` | GET `/fees` | GET `/api/fee/records` | **不一致** |
| `createFeeRecord(data)` | POST `/fees` | POST `/api/fee/records` | **不一致** |
| `refundFee(id, data)` | POST `/fees/{id}/refund` | POST `/api/fee/refund/apply` | **不一致** |

**结论：前端的API封装层路径与后端Controller路径存在系统性不一致。** 前端假设OAS定义的路径规范（如`/students`），但后端Controller使用了不同结构（`/api/student/list`）。在当前版本下，前后端无法正常通信。

#### 3.1.2 请求参数一致性

| 前端参数 | 后端接收参数 | 一致性 |
|---------|------------|:------:|
| `{ page, pageSize, keyword, status, courseId, classId }` | `{ page, size, name, status, campusId }` | **不一致**（pageSize→size, keyword→name, 缺courseId/classId） |
| `{ studentId, items, totalAmount, discountAmount, paidAmount, paymentMethod, remark }` | `{ feeRecordId, amount, paymentMethod, transactionNo, operatorId, operatorName }` | **不一致**（结构完全不同） |

#### 3.1.3 响应格式一致性

| 前端期望 | 后端返回 | 一致性 |
|---------|---------|:------:|
| OAS StudentDTO结构 | `R<StudentDTO.DetailResponse>` | 一致（StudentController做了DTO转换） |
| OAS FeeRecordDTO结构 | `R<FeeRecord>` (Entity) | **不一致**（字段结构和命名不同） |
| OAS LoginResponse结构 | `R<LoginResponse>` | 基本一致 |

### 3.3 跨模块数据一致性

| 检查项 | 结果 | 说明 |
|--------|:----:|------|
| 学员状态常量 | 一致 | Constants中定义了STUDYING/SUSPENDED/TRANSFERRED/WITHDRAWN/GRADUATED |
| 缴费状态常量 | 不一致 | OAS使用PAID/PARTIAL/UNPAID/REFUNDED；Constants额外定义了REFUNDING |
| 班级状态常量 | 不一致 | Constants使用PENDING/IN_PROGRESS/FINISHED/CANCELED；OAS使用RECRUITING/IN_PROGRESS/COMPLETED |
| 考勤状态常量 | 部分一致 | Constants有PRESENT/LATE/ABSENT/LEAVE/EARLY_LEAVE；OAS仅定义了ON_TIME/LATE/ABSENT |

**关键发现**：同一业务概念在OAS、Constants、MDS之间使用不同的枚举值，将导致前后端数据交换时出现解析错误。

**示例冲突**：
- 班级状态：后端使用`PENDING`（待开课），前端OAS预期`RECRUITING`（招生中）
- 缴费状态：后端额外定义了`REFUNDING`（退款中）状态，前端OAS不包含此状态

### 5.2 质量总结

v1版本代码的核心优点：
1. **代码结构清晰**：9个业务模块+1个引导模块，分层架构基本遵循
2. **核心计算正确**：收费金额使用BigDecimal精确计算，校验逻辑完备
3. **基础安全到位**：JWT认证、密码加密、接口鉴权已实现
4. **代码风格统一**：Lombok、MyBatis-Plus、Swagger注解使用规范

v1版本代码的主要问题：
1. **前后端契约断裂**：API路径系统性不一致，导致前后端无法集成联调
2. **Entity直接暴露**：6个Controller返回Entity而非DTO/VO，存在数据安全风险
3. **业务流程不完整**：退费审批、请假审批、收据生成、通知机制均为TODO
4. **枚举值不一致**：同一概念在OAS、Constants、MDS中使用不同值
5. **跨模块集成缺失**：各模块独立开发，未建立Service层跨模块调用

### 5.3 与逆向校验报告（RCR）的关系

| RCR维度 | 质量维度关联 | 说明 |
|---------|------------|------|
| 架构层校验 | 正确性、一致性 | 分层违反影响架构正确性和数据流一致性 |
| 契约校验 | 一致性 | 前后端路径/参数/响应不一致 |
| 需求溯源校验 | 完备性 | 需求覆盖评估 |
| 代码工程层校验 | 正确性 | 魔法值、事务缺失影响业务正确性 |
| 拓扑校验 | 有效性 | 跨模块依赖缺失导致业务闭环不完整 |

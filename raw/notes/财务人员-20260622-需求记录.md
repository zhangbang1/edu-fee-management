# 财务人员 — 需求记录

**涉众角色**：财务管理人员（财务主管/财务专员）
**记录日期**：2026-06-22
**优先级**：P0（最高优先级）
**需求编号前缀**：REQ-FIN

---

## 一、功能需求

### 1.1 收费标准设置

| 编号 | 需求描述 | 优先级 | 业务价值 |
|------|----------|--------|----------|
| REQ-FIN-001 | 系统需支持按课程设置收费标准，支持多种收费模式：按期收费（学期/季度）、按课时包收费（10节/20节/50节等）、按月收费 | P0 | 收费灵活性的基础支撑 |
| REQ-FIN-002 | 系统需支持阶梯定价，根据购买课时数量自动匹配不同单价 | P0 | 鼓励多买多优惠，提升客单价 |
| REQ-FIN-003 | 系统需支持优惠政策配置（早鸟优惠、多科联报折扣、老带新减免、全款优惠等），支持设置有效期、适用课程和叠加规则 | P0 | 支撑营销活动的合规落地 |
| REQ-FIN-004 | 系统需支持教材费、杂费（材料费、活动费、考试费等）的独立配置和管理 | P1 | 非学费收入的规范化管理 |
| REQ-FIN-005 | 收费标准修改需留痕，记录修改人、修改时间、修改前后内容 | P1 | 收费调整可追溯，满足审计要求 |

### 1.2 学费收取管理

| 编号 | 需求描述 | 优先级 | 业务价值 |
|------|----------|--------|----------|
| REQ-FIN-006 | 系统需支持学费收取登记，录入学号/姓名后自动调出应缴费信息（课程、课时包、金额、优惠），支持部分收款和多笔收款 | P0 | 收费业务的日常高效操作 |
| REQ-FIN-007 | 系统需支持多种支付方式的记录（现金、银行转账、微信支付、支付宝、POS刷卡），记录交易流水号 | P0 | 支付方式多样化的现实需求 |
| REQ-FIN-008 | 系统需自动生成标准格式的电子收据，支持打印和PDF导出，收据包含机构名称、学员信息、收费项目、金额、日期、收款人 | P0 | 财务凭证的标准化和合规化 |
| REQ-FIN-009 | 系统需支持在线支付对接（微信支付/支付宝），家长端可直接在线完成缴费并自动生成电子收据 | P1 | 提升缴费便捷度和家长体验 |
| REQ-FIN-010 | 系统需支持收费记录的组合查询（按日期、校区、学员、收费项目、支付方式等），支持导出Excel | P0 | 日常对账和查询的基础功能 |

### 1.3 退费处理

| 编号 | 需求描述 | 优先级 | 业务价值 |
|------|----------|--------|----------|
| REQ-FIN-011 | 系统需支持退费申请发起（由教务或财务发起），记录退费原因、退费类型（全额退/部分退）、关联原始收费记录 | P0 | 退费业务的标准入口 |
| REQ-FIN-012 | 系统需支持退费金额的自动计算：根据已消耗课时和单价计算应退金额，扣除已产生的教材费等不可退项目 | P0 | 杜绝人工计算错误导致的资金损失 |
| REQ-FIN-013 | 系统需实现退费的多级审批流程：教务初审→财务复核→管理员终审（大额退费），每级审批记录意见和时间 | P0 | 退费风控和合规管理 |
| REQ-FIN-014 | 系统需记录退款的支付方式、退款金额、退款日期、退款流水号，与原收费记录关联 | P0 | 退费全流程留痕可追溯 |

### 1.4 欠费催缴管理

| 编号 | 需求描述 | 优先级 | 业务价值 |
|------|----------|--------|----------|
| REQ-FIN-015 | 系统需自动识别欠费学员：已报名未缴费、分期付款逾期、课时消耗完未续费等情况，生成欠费列表 | P0 | 减少漏收少收，保障机构收入 |
| REQ-FIN-016 | 系统需支持催缴通知的发送：可手动或自动向欠费学员家长发送缴费提醒（App推送/短信/微信模板消息） | P1 | 提升催缴效率，降低沟通成本 |
| REQ-FIN-017 | 系统需支持欠费学员的状态标记（正常/欠费提醒/欠费停课），欠费停课自动锁定学员签到功能 | P1 | 欠费管理与教学管理的联动 |

### 1.5 财务报表

| 编号 | 需求描述 | 优先级 | 业务价值 |
|------|----------|--------|----------|
| REQ-FIN-018 | 系统需提供收费日报/月报/季报/年报，包含收费总额、笔数、按收费项目分类统计、按校区分类统计 | P0 | 经营分析的日常数据支撑 |
| REQ-FIN-019 | 系统需提供退费统计报表，包含退费总额、退费笔数、退费率、按退费原因分类统计 | P0 | 退费分析和风险监控 |
| REQ-FIN-020 | 系统需提供应收应付统计报表：应收学费（已报名未缴费）、应付课酬（已授课未结算），支持按时间段汇总 | P0 | 资产负债管理的决策依据 |
| REQ-FIN-021 | 所有财务报表支持按校区、课程品类、时间段等维度筛选和下钻，支持图表可视化展示 | P1 | 多维度分析能力 |
| REQ-FIN-022 | 所有报表支持导出为Excel格式（含公式）和PDF格式（含水印），满足不同使用场景 | P0 | 报表输出和归档需求 |

## 二、非功能需求

| 编号 | 需求描述 | 类型 | 优先级 |
|------|----------|------|--------|
| NF-FIN-001 | 所有涉及金额的计算需使用高精度数据类型（Decimal），禁止使用浮点数计算 | 准确性 | P0 |
| NF-FIN-002 | 每笔财务操作（收费、退费、审批、修改）需记录不可篡改的操作日志，包含操作人、时间、IP、操作前后数据快照 | 可审计性 | P0 |
| NF-FIN-003 | 财务数据需加密存储（AES-256），数据库备份文件同样加密 | 安全性 | P0 |
| NF-FIN-004 | 大额退费（可配置阈值，默认5000元）操作需二次身份验证 | 安全性 | P0 |
| NF-FIN-005 | 财务报表在大数据量（年报表、全校区）下查询和导出响应时间不超过10秒 | 性能 | P1 |
| NF-FIN-006 | 每日收费记录需支持按日自动对账（与在线支付平台的交易流水比对），差异自动标记 | 合规性 | P1 |

---

## 三、约束条件

1. 收费标准和退费规则需支持按校区独立配置
2. 收据格式需符合税务部门要求，预留电子发票对接接口
3. 退费审批流程的级数和每级审批人需可配置
4. 财务数据需定期自动备份，备份周期不低于每日一次
5. 涉及退费的课时消耗计算需与教务模块的考勤数据联动

## 1. 引言

### 1.1 目的

本文档旨在为"教育培训机构教务收费管理系统"提供完整、精确的软件需求规格说明。本文档依据IEEE 830-1998标准编写，详细描述了系统的功能需求、非功能需求、外部接口需求和设计约束，为后续的软件设计、编码实现、测试验证和项目验收提供统一的参考基准。

本文档的预期读者包括：
- **软件开发团队**：理解系统功能边界和实现要求
- **项目管理人员**：评估项目范围和工作量
- **测试人员**：制定测试计划和测试用例
- **涉众代表**：确认需求是否完整、准确反映业务需求
- **CCB（变更控制委员会）**：作为需求变更评估的基线依据

### 1.2 范围

#### 1.2.1 系统名称
教育培训机构教务收费管理系统（Educational Institution Academic & Fee Management System，简称EduFeeMS）

#### 1.2.2 系统概述
EduFeeMS是面向中小型教育培训机构的综合教务收费管理平台，覆盖学员管理、课程管理、班级管理、教师管理、收费管理、考勤管理、报表统计和系统管理八大核心业务模块。系统旨在帮助教育培训机构实现教务流程的信息化、收费管理的规范化、数据统计的自动化，全面提升机构运营效率和管理水平。

#### 1.2.3 系统边界
**系统包含的功能**：
- 学员全生命周期管理（报名→分班→转班→退学）
- 课程和班级的配置与调度管理
- 教师信息与授课安排管理
- 学费、教材费、杂费等各项费用的收取与退费管理
- 欠费检测与自动催缴
- 学员考勤（签到、请假、缺勤）管理
- 多维度业务报表统计与分析
- 用户权限与系统配置管理
- 多个校区的统一管理与数据隔离
- 收费发票的开具与管理

**系统不包含的功能**：
- 在线教学与直播授课功能
- 学员学习进度与成绩管理（由独立的学习管理系统负责）
- 人力资源管理与薪酬发放（仅提供课时统计数据供HR系统使用）
- 财务管理中的税务申报功能

## 2. 总体描述

### 2.1 产品视角

#### 2.1.1 系统定位

EduFeeMS在教育培训机构的整体信息化架构中处于核心位置。系统通过Web浏览器提供统一的操作入口，服务于机构内部的各类用户角色。系统采用前后端分离的B/S架构，后端提供RESTful API服务，前端提供响应式Web界面。系统可独立部署，同时预留与第三方系统（如微信支付、支付宝、短信平台、邮件服务）的集成接口。

#### 2.1.2 系统与外部接口

EduFeeMS与以下外部系统存在接口关系：

| 外部系统 | 接口类型 | 数据流向 | 说明 |
|---------|---------|---------|------|
| 微信支付/支付宝 | API调用 | 双向 | 学费在线支付与退款 |
| 短信平台 | API调用 | 单向（发送） | 缴费提醒、课程通知、催缴通知 |
| 邮件服务 | SMTP | 单向（发送） | 报表邮件发送、通知邮件 |
| 打印服务 | 本地驱动 | 单向（输出） | 收据打印、发票打印、报表打印 |
| 学习管理系统（LMS） | REST API | 双向 | 学员课程进度数据同步（预留） |
| 人力资源系统（HR） | REST API | 单向（输出） | 教师课时统计数据（预留） |
| 税务系统 | REST API | 单向（输出） | 发票数据上报（预留） |

#### 2.1.3 系统总体架构

系统采用前后端分离的Web应用架构：
- **前端层**：Vue.js单页应用（SPA），提供响应式用户界面
- **后端层**：Spring Boot RESTful API服务，处理业务逻辑
- **数据层**：MySQL关系数据库，存储所有业务数据
- **缓存层**：Redis，提供高频访问数据的缓存
- **文件存储**：本地文件系统或OSS，存储收据、报表、发票等文件

#### 2.1.4 部署架构

系统支持单校区部署和多校区集中部署两种模式：
- **单校区模式**：所有服务部署在一台服务器上
- **多校区模式**：后端服务集中部署，各校区通过浏览器访问，支持校区间数据隔离

### 2.2 产品功能

EduFeeMS提供以下八大核心功能模块：

1. **学员管理**：学员信息的增删改查、学员档案管理、分班与转班、退学处理
2. **课程管理**：课程类型设置、课程信息维护、课时配置、费用标准制定
3. **班级管理**：班级创建与维护、排课管理、班级学员管理、教室资源管理
4. **教师管理**：教师信息维护、授课安排、课时统计、课酬计算
5. **收费管理**：收费项目设置、学费收取、退费处理、欠费催缴、收据打印、台账记录、优惠管理、发票管理
6. **考勤管理**：学员签到/签退、请假申请与审批、缺勤记录、考勤统计、考勤补签、考勤预警
7. **报表统计**：财务报表、课时统计报表、学员统计报表、欠费统计报表、运营分析仪表盘、教师绩效报表
8. **系统管理**：用户管理、角色权限管理、系统参数配置、操作日志、数据备份、校区管理

## 3. 具体需求

### 3.1 学员管理模块

#### 3.1.1 功能描述

学员管理模块负责学员从报名到结业/退学的全生命周期管理。该模块是系统的核心入口，所有教务操作均以学员为中心展开。模块涵盖学员信息的增删改查、学员档案管理、分班与转班操作、退学处理以及关键状态变更的自动通知功能，确保学员在机构的整个学习周期均可被准确追踪和管理。

#### 3.1.2 功能需求

##### REQ-STU-001：学员信息录入

- **功能描述**：教务人员录入新学员的基本信息，创建学员档案
- **输入**：学员姓名、性别、出生日期、联系电话、联系人姓名、联系人电话、联系地址、来源渠道、备注
- **处理过程**：
  1. 教务人员进入学员管理模块，点击"新增学员"按钮，系统展示学员信息录入表单页面，表单包含所有必填字段和可选的扩展信息字段，界面采用分区布局（基本信息区、联系信息区、来源信息区），便于教务人员高效填写。
  2. 教务人员逐项填写学员姓名、性别、出生日期、联系电话、联系人姓名、联系人电话、联系地址等基本信息，其中姓名字段支持中英文输入，出生日期通过日期选择器填写，联系电话字段实时检测格式是否合法。
  3. 教务人员从下拉列表中选择学员来源渠道（如：网络推广、老学员推荐、地推活动、到店咨询、合作渠道等），并在备注栏填写补充说明信息，如特殊需求、家长关注点等辅助备忘内容。
  4. 系统在教务人员提交表单时进行前端校验：检查必填字段（姓名、联系电话、联系人姓名）是否完整填写，校验手机号格式是否符合11位数字的要求，校验出生日期是否为有效日期且不能晚于当前日期。
  5. 前端校验通过后，系统在后台执行重复学员检测：根据姓名和联系电话的组合查询现有学员库，若发现同名同电话的潜在重复记录，系统弹出提示框列出可能的重复学员，由教务人员确认是新增还是返回修改，系统不强制阻止录入。
  6. 确认无误后，系统在数据库事务中执行以下操作：生成全局唯一的学员编号（格式：STU-YYYYMMDD-四位数序号，如STU-20260622-0001），将学员初始状态设置为"待分班"，记录创建时间（精确到秒）和创建人（当前登录的教务人员账号），创建学员档案记录并返回操作结果。
  7. 系统向前端返回创建成功的响应，包含生成的学员编号、学员姓名和初始状态，页面跳转至学员详情页或保持在学员列表页并显示成功提示消息。
- **输出**：学员档案创建成功提示，显示生成的学员编号
- **业务规则**：
  - BR-STU-001-01：学员编号格式固定为 STU-YYYYMMDD-NNNN，其中YYYYMMDD为创建日期，NNNN为当日四位数递增序号，编号一经生成不可修改。
  - BR-STU-001-02：联系电话字段必须为11位手机号，系统不允许录入非手机号的联系方式作为主联系电话。
  - BR-STU-001-03：出生日期字段用于自动计算学员年龄，年龄小于3岁或大于80岁时系统提示确认。
  - BR-STU-001-04：新创建的学员初始状态必须为"待分班"，只有经过分班操作后方可变更状态。
  - BR-STU-001-05：来源渠道字段为必填项，用于后续学员来源分析报表的统计。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-STU-002：学员信息查询

- **功能描述**：支持多条件组合查询学员信息
- **输入**：学员姓名（模糊）、联系电话、学员编号、状态、报名日期范围、所在班级、所在校区
- **处理过程**：
  1. 用户进入学员列表页面，页面顶部显示多条件搜索栏，包含学员姓名（文本输入框，支持模糊匹配）、联系电话（文本输入框）、学员编号（文本输入框，精确匹配）、状态（下拉多选框）、报名日期范围（日期区间选择器）、所在班级（下拉搜索框）和所在校区（下拉选择框）等查询字段。
  2. 用户根据需求填入一个或多个查询条件，系统采用"AND"逻辑组合所有非空条件。学员姓名字段使用LIKE模糊匹配（支持部分姓名检索），联系电话和学员编号使用精确匹配，状态字段支持多选（可选择多个状态值进行OR组合），报名日期范围使用BETWEEN范围查询。
  3. 用户点击"搜索"按钮或按下回车键后，系统将查询条件封装为请求参数发送至后端API，后端构建动态SQL查询语句，使用参数化查询方式防止SQL注入攻击，所有用户输入均经过安全过滤处理。
  4. 系统查询数据库并返回匹配的学员记录列表，列表默认按报名日期降序排列（最新报名的学员排在最前面），支持用户点击表头切换排序字段和排序方向（升序/降序）。
  5. 查询结果采用分页展示，每页默认显示20条记录。页面底部显示分页导航控件，包含总记录数、总页数、当前页码显示，以及首页、上一页、下一页、末页和跳转到指定页的功能按钮。
  6. 每条学员记录以表格行形式展示学员编号、姓名、性别、状态（带颜色标签）、联系电话、所在班级名称、报名日期等关键信息。状态字段使用不同颜色区分：待分班（灰色）、在读（绿色）、已退学（红色）。
  7. 用户点击任意学员行或"查看详情"按钮，系统跳转至该学员的完整档案页面，展示学员的详细信息、班级历史、缴费记录、考勤记录等汇总数据。
- **输出**：学员列表（分页），每页显示学员编号、姓名、状态、所在班级、联系电话
- **业务规则**：
  - BR-STU-002-01：默认分页大小为每页20条记录，用户可在10/20/50/100条间切换。
  - BR-STU-002-02：搜索结果最多返回前1000条匹配记录（超过时提示用户缩小查询范围）。
  - BR-STU-002-03：用户只能查询其所辖校区范围内的学员数据（数据权限隔离）。
  - BR-STU-002-04：查询条件支持URL参数传递，用户可通过浏览器书签保存常用查询条件。
- **优先级**：高
- **关联涉众**：教务人员、机构管理员

##### REQ-STU-003：学员信息修改

- **功能描述**：修改已存在学员的基本信息
- **输入**：学员编号（定位）、需修改的字段（除编号外均可修改）
- **处理过程**：
  1. 教务人员通过学员搜索功能或学员列表页面定位到需要修改信息的学员，点击学员行或"编辑"按钮进入学员信息编辑页面。
  2. 系统加载该学员当前的完整信息并填充至编辑表单中，学员编号字段以只读方式显示（灰色背景，不可修改），其余字段（姓名、性别、出生日期、联系电话、联系人姓名、联系人电话、联系地址、来源渠道、备注等）均处于可编辑状态。
  3. 教务人员根据实际情况修改需要变更的字段，例如更新联系人电话、修正联系地址、补充备注信息等。修改过程中系统实时进行前端输入校验，确保修改后的数据格式合法。
  4. 教务人员完成修改后点击"保存"按钮，系统再次进行后端数据校验，包括必填字段检查、手机号格式校验、出生日期有效性校验，校验不通过则返回具体的错误提示定位到对应字段。
  5. 校验通过后，系统在数据库事务中执行更新操作：将新值写入学员信息表，同时在变更历史表中创建一条变更记录，记录修改时间、修改人（当前登录教务人员的账号）、变更字段名、变更前的旧值和变更后的新值。
  6. 系统检查变更是否涉及关键信息（如联系电话变更可能影响欠费催缴通知的发送目标、联系人变更可能影响紧急联系），若涉及关键信息变更，系统在保存成功后提示教务人员关注相关影响。
  7. 更新成功后，系统刷新学员信息编辑页面，显示最新的学员数据，并在页面顶部显示"修改成功"的绿色提示消息，该提示3秒后自动消失。
- **输出**：修改成功提示，更新后的学员信息
- **业务规则**：
  - BR-STU-003-01：学员编号为系统自动生成的不可变标识，任何情况下均不允许修改。
  - BR-STU-003-02：每次修改操作必须在变更历史表（audit_log）中记录完整的前后对比，不可省略。
  - BR-STU-003-03：若学员当前状态为"已退学"，其基本信息仍可查看但不可修改，如需修改需先由管理员解锁。
  - BR-STU-003-04：联系电话修改后，系统需检查新号码是否与其他在读学员重复，若重复则提示但不阻止。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-STU-004：学员分班

- **功能描述**：将"待分班"状态的学员分配到指定班级
- **输入**：学员编号、目标班级编号
- **处理过程**：
  1. 教务人员在学员列表页面筛选状态为"待分班"的学员，或在学员详情页面看到"待分班"状态时，点击"分班"操作按钮，系统进入分班操作流程。
  2. 系统展示可选班级列表页面，列表以表格形式显示当前校区下所有状态为"待开课"或"进行中"的班级信息，每行包含班级名称、关联课程名称、授课教师姓名、上课时间描述、当前学员人数、最大学员人数（容量）、剩余名额数等信息。
  3. 剩余名额数采用醒目的视觉提示：名额充足（剩余>=5）显示绿色、名额紧张（剩余1-4）显示橙色、名额已满（剩余0）显示红色且默认不可选（除非管理员开启了超额分班权限）。
  4. 教务人员根据学员的课程需求和班级剩余名额情况，选择一个合适的目标班级。点击目标班级行后，系统弹出分班确认对话框，显示学员姓名、目标班级名称、课程名称、开班日期等关键信息供教务人员二次确认。
  5. 教务人员点击"确认分班"后，系统在后台执行以下校验：检查目标班级的剩余名额是否大于0（或当前人数是否未超过容量上限），检查学员当前状态是否仍为"待分班"（防止并发操作导致的分班冲突），检查学员是否已被分配到其他班级（若已有有效班级，则提示教务人员应使用"转班"功能）。
  6. 校验通过后，系统在数据库事务中执行分班操作：更新学员状态为"在读"，记录分班日期为当前日期，在学员-班级关联表中创建关联记录（记录学员编号、班级编号、入班日期），同时更新目标班级的当前学员人数（递增1）。
  7. 系统创建报名记录（Enrollment），包含学员编号、班级编号、报名日期、报名操作人。同时触发通知流程：向学员/家长发送分班成功通知（包含班级名称、课程名称、开班日期、上课时间等信息），通知方式为系统内消息通知，若学员联系方式中有手机号则同步发送短信通知。
- **输出**：分班成功提示，学员状态更新
- **业务规则**：
  - BR-STU-004-01：只能对状态为"待分班"的学员执行分班操作，其他状态学员需通过其他流程处理。
  - BR-STU-004-02：目标班级的当前学员人数必须小于最大学员人数（容量），超额分班需管理员权限。
  - BR-STU-004-03：同一学员在同一时间段只能属于一个有效班级（转班后原班级关联自动失效）。
  - BR-STU-004-04：分班操作成功后，系统自动为学员生成首期缴费账单（根据课程费用标准）。
  - BR-STU-004-05：分班日期默认为操作当日，教务人员不可修改为未来日期。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-STU-005：学员转班

- **功能描述**：将在读学员从一个班级转到另一个班级
- **输入**：学员编号、当前班级编号、目标班级编号、转班原因
- **处理过程**：
  1. 教务人员通过学员搜索或班级学员列表定位到需要转班的学员，点击"转班"操作按钮，系统启动转班操作流程。
  2. 系统首先展示当前班级的基本信息摘要，包括班级名称、课程名称、授课教师、学员入班日期、已上课次数等，同时展示学员在该班的缴费情况摘要。
  3. 教务人员填写转班原因（从下拉列表中选择或手动输入，如：时间冲突、课程难度不匹配、家长要求、班级合并等），转班原因为必填项，用于后续运营分析时统计转班原因分布。
  4. 系统加载并展示可转班的目标班级列表，与分班操作类似，展示每个可选班级的名称、课程、剩余名额、上课时间等信息。系统优先推荐与学员当前课程类型相同或相近的班级，若课程类型不同则标记"跨课程转班"以提醒教务人员注意费用差异。
  5. 教务人员选择目标班级后，系统计算费用差异：比较当前班级和目标班级的课程费用标准，若目标班级费用高于当前班级，系统提示需补缴差额费用（显示差额金额），并在转班确认后自动生成补缴账单；若目标班级费用低于当前班级，系统提示需进行退费处理（按退费规则计算可退金额）。
  6. 教务人员确认转班操作后，系统在数据库事务中执行：将学员当前班级关联记录标记为"已转出"并记录转出日期，创建新的学员-班级关联记录标记为"已转入"并记录转入日期，记录转班历史（原班级编号、新班级编号、转班日期、转班原因、操作人），更新原班级和新班级的学员人数计数。
  7. 若存在费用差异，系统自动创建费用调整待办任务通知财务人员处理。系统同时触发通知流程：向学员/家长发送转班成功通知（包含原班级、新班级、上课时间变化等信息），向新班级授课教师发送新学员入班通知，向原班级授课教师发送学员转出通知。
- **输出**：转班成功提示，费用调整提醒（如适用）
- **业务规则**：
  - BR-STU-005-01：只能对状态为"在读"的学员执行转班操作。
  - BR-STU-005-02：转班原因必须填写，且不少于5个字符，用于后续统计分析的完整性保障。
  - BR-STU-005-03：同一学员每月转班次数限制为不超过2次（防止频繁转班影响教学秩序）。
  - BR-STU-005-04：若原班级已完成课程超过总课时的70%，系统提示不建议转班，但仍允许操作（需教务人员二次确认）。
  - BR-STU-005-05：跨课程类型转班时，费用差额计算需纳入已消耗课时因素，按比例计算应补/应退金额。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-STU-006：学员退学

- **功能描述**：办理学员退学手续，计算退费金额
- **输入**：学员编号、退学原因、退学日期
- **处理过程**：
  1. 教务人员通过学员搜索或班级学员列表定位到需要办理退学的学员，点击"退学"操作按钮，系统弹出退学处理对话框。
  2. 系统展示该学员的当前信息摘要：学员姓名、当前班级名称、入班日期、已上课次数/总课时数、已缴费用总额、剩余预付费用余额等，帮助教务人员全面了解学员当前状态。
  3. 教务人员从下拉列表中选择退学原因（如：学员个人原因、转学、课程不适合、经济困难、教学质量不满意等），选择退学日期（默认为当天日期，不可选择未来日期），并在备注栏补充详细说明。
  4. 系统根据学员所在班级的课程费用标准和已消耗课时数，自动计算应退金额。计算公式为：总缴费金额 -（已消耗课时 × 单课时单价）- 已产生的杂费 - 手续费（如有）。其中已消耗课时数据来源于考勤模块中该学员的实到签到次数。
  5. 系统根据退费规则（可在系统参数中配置）进一步调整退费金额。典型退费规则包括：已上1/3课时以内退还已缴费用的60%，已上1/3至2/3退还已缴费用的30%，已上超过2/3不予退费。退费规则可配置，系统应用当前生效的规则版本。
  6. 教务人员确认退学信息和退费金额后提交，系统在数据库事务中执行：将学员状态更新为"已退学"，记录退学日期和退学原因，保存退费计算明细，创建一条退费待办任务并推送给财务人员（任务类型为"退费处理"，关联学员编号和应退金额）。
  7. 系统触发通知流程：向学员/家长发送退学确认通知和退费金额说明，向班级授课教师发送学员退学通知，向教务主管发送退学汇总通知。退费待办任务进入财务人员的待办工作列表，待财务人员执行实际退款操作（参见REQ-FEE-004）。
- **输出**：退学处理完成提示，退费金额显示，财务退费待办
- **业务规则**：
  - BR-STU-006-01：只能对状态为"在读"的学员执行退学操作，"待分班"学员无需退学流程可直接删除。
  - BR-STU-006-02：退学日期不得早于学员入班日期，不得晚于当前日期。
  - BR-STU-006-03：退学操作不可逆，一旦确认退学，学员状态不可恢复为"在读"。
  - BR-STU-006-04：退费金额计算必须精确到分（0.01元），采用四舍五入方式。
  - BR-STU-006-05：若学员存在欠费，退费金额应先抵扣欠费部分，再将剩余金额退还给学员。
- **优先级**：高
- **关联涉众**：教务人员、财务人员

##### REQ-STU-007：学员档案查看

- **功能描述**：查看学员完整的档案信息
- **输入**：学员编号
- **处理过程**：
  1. 用户通过任意入口（学员列表、班级学员列表、搜索结果等）点击学员姓名或"查看档案"按钮，系统加载学员档案页面。
  2. 系统查询学员基本信息表，加载并展示学员的个人信息区域，包括学员编号、姓名、性别、出生日期、年龄、联系电话、联系人信息、联系地址、来源渠道、当前状态（带颜色标签）、创建日期等完整字段。
  3. 系统查询学员-班级关联表，加载班级历史记录区域，以时间线形式展示学员在机构的班级变动历程：首次分班记录、历次转班记录（显示原班级→新班级、转班日期、转班原因）、当前所在班级信息（班级名称、课程名称、授课教师、入班日期、上课时间表）。
  4. 系统查询缴费记录表和欠费信息，加载缴费记录区域，展示学员的收费历史明细（收费日期、收费项目、应收金额、优惠金额、实收金额、支付方式、收据编号），以及当前欠费状态（欠费金额、欠费天数、催缴记录）。
  5. 系统查询考勤记录表，加载考勤统计区域，以表格和图表结合的形式展示：课程应到总次数、实到次数（签到）、请假次数（已审批）、缺勤次数（未签到且未请假）、出勤率百分比、近30日考勤趋势折线图。
  6. 页面采用分Tab标签页布局（基本信息Tab、班级历史Tab、缴费记录Tab、考勤统计Tab），用户可自由切换查看不同维度的档案信息，各Tab数据独立加载，提升页面响应速度。
  7. 档案页面提供快捷操作按钮区域，根据学员当前状态动态显示可用操作：若学员状态为"待分班"则显示"分班"按钮，若为"在读"则显示"转班"和"退学"按钮，所有状态下均显示"打印档案"按钮用于导出学员档案的PDF版本。
- **输出**：学员完整档案页面，包含基本信息、班级历史、缴费记录、考勤统计
- **业务规则**：
  - BR-STU-007-01：学员档案页面只读展示，不提供直接编辑功能，修改需通过各子功能入口操作。
  - BR-STU-007-02：缴费记录按时间倒序展示，最近一笔记录排在最前。
  - BR-STU-007-03：考勤出勤率低于60%时，系统在档案页面用红色高亮标记，提醒教务人员关注。
  - BR-STU-007-04：档案页面数据实时同步，不得使用超过5分钟的缓存数据。
- **优先级**：中
- **关联涉众**：教务人员、机构管理员

##### REQ-STU-008：学员状态变更通知

- **功能描述**：学员关键状态变更时自动通知相关人员
- **输入**：状态变更事件（分班、转班、退学、缴费完成等）
- **处理过程**：
  1. 系统在各业务模块的关键操作完成后（如分班、转班、退学、缴费成功等），发布对应的状态变更事件到内部事件总线。
  2. 通知服务监听事件总线中的状态变更事件，根据事件类型从通知规则配置中查找对应的通知策略，确定需要通知的对象列表（学员/家长、教师、教务人员等）和通知方式（系统内消息、短信、邮件）。
  3. 对于需要通知学员/家长的场景（如分班成功、缴费成功、欠费提醒），系统优先使用系统内消息通知，若学员档案中有有效的手机号且事件级别为"重要"，则同步发送短信通知。
  4. 对于需要通知教师或教务人员的场景（如新学员入班、学员退学、学员转出），系统在对应角色的"消息中心"中生成通知消息，同时在页面顶部导航栏的消息图标上显示未读红点计数。
  5. 系统根据通知模板生成通知内容，模板中包含变量占位符，运行时替换为具体的学员姓名、班级名称、日期等实际信息，确保通知内容个性化且准确。
  6. 系统调用短信平台API发送短信通知（如有需要），记录短信发送状态（发送中/已发送/发送失败），若发送失败则自动重试一次，两次均失败则记录失败日志并通知系统管理员。
  7. 所有通知发送记录均持久化到通知记录表（notifications），包含通知ID、接收人、通知类型、通知内容、发送渠道、发送时间、发送状态等字段，供后续审计和统计使用。
- **输出**：通知发送记录
- **业务规则**：
  - BR-STU-008-01：系统内消息通知保留期为90天，超期自动归档。
  - BR-STU-008-02：短信通知仅在8:00-21:00时段发送，避免夜间打扰学员/家长。
  - BR-STU-008-03：同一学员同一天内同一类型通知最多发送3次，防止通知骚扰。
  - BR-STU-008-04：学员/家长可在个人设置中选择关闭短信通知渠道（系统内消息不可关闭）。
- **优先级**：中
- **关联涉众**：学生/家长、教师

### 3.2 课程管理模块

#### 3.2.1 功能描述

课程管理模块负责培训机构所有课程类型的定义和维护，包括课程基本信息、课时设置、费用标准等。课程是班级创建的基础模板，一个课程类型可对应多个班级实例。课程管理模块还负责管理课程的上下架状态，确保只有已上架的课程才能被用于创建新班级。

#### 3.2.2 功能需求

##### REQ-CRS-001：课程类型管理

- **功能描述**：创建和维护课程类型（如：数学辅导、英语培训、编程入门等）
- **输入**：课程名称、课程类别、适用年龄段、课程描述、课程图片
- **处理过程**：
  1. 教务人员进入课程管理模块，点击"新增课程"按钮，系统展示课程类型创建表单页面，表单包含课程基本信息区域（课程名称、课程类别、适用年龄段）和扩展信息区域（课程描述、课程图片、教学大纲等）。
  2. 教务人员填写课程名称（如："小学数学培优班"、"少儿英语启蒙"），从预设类别中选择课程类别（如：学科辅导、语言培训、艺术培训、编程教育、体育培训等），选择适用年龄段范围（如：6-8岁、9-12岁、13-15岁等）。
  3. 教务人员在课程描述编辑器中撰写课程的详细介绍，包括课程目标、教学内容概述、适合学员群体、教学特色等，编辑器支持富文本格式（粗体、列表、段落等），提供不低于200字的介绍内容空间。
  4. 教务人员上传课程封面图片（可选，支持JPG/PNG格式，大小不超过2MB），系统自动将图片裁剪为统一尺寸的缩略图用于列表展示，原始图片保留用于详情页展示。
  5. 教务人员提交表单后，系统首先校验课程名称的唯一性：在同一校区范围内，课程名称必须唯一（不区分大小写）。若发现重名课程，系统提示"该课程名称已存在，请使用其他名称或检查是否重复创建"。
  6. 校验通过后，系统在数据库中创建课程类型记录，自动生成课程编号（格式：CRS-YYYYMMDD-四位序号），记录创建时间和创建人。课程初始状态设为"未上架"，需经后续配置（课时、费用标准等）并人工上架后方可用于创建班级。
  7. 创建成功后，系统返回课程类型详情页面，教务人员可继续进行课时配置和费用标准设置操作。系统同时记录课程创建操作日志。
- **输出**：课程类型创建成功提示
- **业务规则**：
  - BR-CRS-001-01：课程名称在同一校区内必须唯一，不同校区可使用相同名称。
  - BR-CRS-001-02：课程编号格式固定为 CRS-YYYYMMDD-NNNN，编号一经生成不可修改。
  - BR-CRS-001-03：新创建的课程默认状态为"未上架"，需单独执行上架操作后方可用于创建班级。
  - BR-CRS-001-04：课程类别从系统预置的类别字典中选择，教务人员不可自定义新增类别。
  - BR-CRS-001-05：课程图片上传是可选的，不上传时使用系统默认的课程占位图片。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-CRS-002：课程信息维护

- **功能描述**：修改已有课程的基本信息
- **输入**：课程编号、需修改的字段
- **处理过程**：
  1. 教务人员在课程列表页面通过搜索或浏览找到需要维护的课程，点击课程行或"编辑"按钮进入课程编辑页面。
  2. 系统加载课程的当前完整信息填充至编辑表单，课程编号字段以只读方式显示。教务人员可修改课程名称、课程类别、适用年龄段、课程描述、课程图片等信息。
  3. 若教务人员修改了课程名称，系统在提交时重新执行名称唯一性检查，确保新名称不与同校区其他课程重复。同时系统检测是否有在进行的班级引用了该课程，并在页面顶部显示"该课程下有N个进行中班级，信息修改将同步影响这些班级的展示信息"的提醒。
  4. 教务人员完成修改后提交表单，系统执行数据校验（与新增时的校验规则一致），校验通过后将更新后的数据写入课程信息表。
  5. 系统在课程变更历史表中创建一条修改记录，包含课程编号、修改时间、修改人、变更字段列表和变更内容摘要，用于追溯课程信息的历史变化。
  6. 若课程的描述信息发生重大变更（描述长度变化超过50%），系统可选地向该课程下所有在读学员发送课程信息更新通知，由教务人员在提交前勾选是否发送通知。
  7. 更新成功后，系统返回课程详情页面展示最新信息，页面顶部显示成功提示消息。若有进行中的班级引用了该课程，班级学员端的课程信息展示将在页面刷新后同步更新。
- **输出**：修改成功提示
- **业务规则**：
  - BR-CRS-002-01：课程编号不可修改，其他字段均可编辑。
  - BR-CRS-002-02：课程信息修改不影响已关联班级的课时和费用配置（班级创建时已快照）。
  - BR-CRS-002-03：修改课程适用年龄段后，已有班级不受影响，仅影响后续新班级创建时的推荐逻辑。
  - BR-CRS-002-04：所有修改操作需记录完整的变更日志，用于审计追溯。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-CRS-003：课时配置

- **功能描述**：为课程配置总课时数和单次课时长
- **输入**：课程编号、总课时数、每课时分钟数、课程有效期（月）
- **处理过程**：
  1. 教务人员在课程详情页面点击"课时配置"标签，进入课时配置页面，系统展示当前的课时配置信息（若已配置过）或空白配置表单（首次配置）。
  2. 教务人员设置总课时数，即该课程类型下每个班级的标准总课时数（如：48课时），总课时数必须为正整数，系统提供常用的课时数快捷选项（24、36、48、60、72、96课时）。
  3. 教务人员设置每课时分钟数，即每次上课的标准时长（如：45分钟/课时或60分钟/课时），每课时分钟数影响课堂考勤的时间计算和课酬统计的课时折算。
  4. 教务人员设置课程有效期，即从学员入班之日起计算的有效学习时长（以月为单位，如：6个月或12个月），超过有效期的课程安排需要教务人员手动延期。
  5. 系统校验课时配置的合理性：总课时数不能超过999课时（超过需分拆为多个课程），每课时分钟数需在30-180分钟范围内，课程有效期需在1-36个月范围内。
  6. 教务人员确认并保存课时配置后，系统将配置信息写入课程课时配置表。若课程之前已有配置，系统创建新版本的课时配置记录并标记旧版本为"已失效"，保留完整的版本历史。
  7. 保存成功后，系统返回成功提示。若该课程已有进行中的班级，系统提示"课时配置变更不会影响已有班级的课时设置，仅对新创建的班级生效"，确保教务人员了解配置的生效范围。
- **输出**：课时配置保存成功
- **业务规则**：
  - BR-CRS-003-01：课时配置采用版本化管理，每次修改创建新版本，旧版本保留供历史查询。
  - BR-CRS-003-02：课时配置变更仅对修改后新创建的班级生效，已有班级沿用班级创建时的快照配置。
  - BR-CRS-003-03：每课时分钟数一经设置，在同一版本内不可修改为不同类型的值（如不能混用45分钟和60分钟课时）。
  - BR-CRS-003-04：课程有效期到期后，系统自动提醒教务人员处理该课程下仍在进行中的班级。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-CRS-004：费用标准设置

- **功能描述**：为课程设置收费标准和优惠方案
- **输入**：课程编号、标准单价（元/课时）、总价、优惠方案（早鸟优惠、多科联报优惠、老带新优惠等）
- **处理过程**：
  1. 教务人员在课程详情页面点击"费用标准"标签，进入费用配置页面。系统展示当前生效的费用标准版本（如有）和费用标准历史版本列表。
  2. 教务人员设置课程的标准收费模式，可在两种模式中选择：按课时单价模式（设置每课时的价格，总价=课时数×单价）或按课程总价模式（设置一个固定的课程总价格，不按课时拆分），系统根据选择的模式调整后续输入字段。
  3. 教务人员配置适用的优惠方案，可从系统已定义的优惠方案库中选择（如：早鸟优惠-开课前30天报名享9折、多科联报优惠-同时报名2科及以上享8.5折、老带新优惠-老学员推荐新学员双方各减200元），可为一个课程关联多个优惠方案。
  4. 系统计算优惠后的参考价格范围，展示给教务人员预览：原价、应用各优惠方案后的价格、优惠叠加后的最低价格（若允许多优惠叠加），帮助教务人员评估定价合理性。
  5. 教务人员确认费用标准后提交，系统执行校验：标准单价不能为负数或零，优惠方案的有效期需在合理范围内，同一课程关联的优惠方案之间不能存在相互冲突的规则。
  6. 系统保存费用标准配置到课程费用标准表，创建新版本记录。若课程之前已有费用标准，旧版本标记为"已失效"并记录失效日期和原因。费用标准变更需记录完整的版本历史和变更人信息。
  7. 保存成功后，系统检查是否有已关联该课程但尚未收费的学员，若有则提示教务人员"费用标准变更可能影响N名学员的待缴费金额，请核查是否需要为这些学员同步更新账单"，并提供"同步更新学员账单"和"仅对新报名学员生效"两个选项。
- **输出**：费用标准保存成功
- **业务规则**：
  - BR-CRS-004-01：费用标准采用版本化管理，每次修改创建新版本并保留完整历史。
  - BR-CRS-004-02：标准单价必须大于0，单位精确到0.01元（分）。
  - BR-CRS-004-03：费用标准变更需经过教务主管审批（若机构配置了审批流程），审批通过后方可生效。
  - BR-CRS-004-04：已上架且有在读学员的课程，费用标准修改需在弹窗中二次确认后方可生效。
  - BR-CRS-004-05：同一课程最多可关联5个优惠方案，超出数量提示教务人员整合优惠策略。
- **优先级**：高
- **关联涉众**：教务人员、财务人员

##### REQ-CRS-005：课程状态管理

- **功能描述**：管理课程的上架/下架状态
- **输入**：课程编号、目标状态（上架/下架）
- **处理过程**：
  1. 教务人员在课程列表或课程详情页面看到课程当前的状态标识（绿色"已上架"标签或灰色"未上架"标签），点击状态切换按钮或在详情页点击"上架"或"下架"按钮触发状态变更。
  2. 当执行上架操作时，系统首先执行上架前置条件检查：课程是否已完成课时配置（总课时数和每课时分钟数均已设置），是否已设置费用标准（标准单价或总价已配置），课程是否有关联的适用班级（已有班级或可创建班级）。若任一条件不满足，系统提示具体缺失项并引导教务人员完成配置。
  3. 当执行下架操作时，系统检查该课程是否有关联的进行中班级。若有进行中班级，系统弹出提示对话框显示"该课程下有N个进行中班级，下架后将无法为该课程创建新班级，但已有班级不受影响。确认继续？"，教务人员确认后继续。
  4. 对于下架操作，系统还检查是否有学员处于"待分班"状态且选择了该课程类型，若存在则提示教务人员这些学员需要重新分配课程，并提供待分班学员列表供教务人员批量处理。
  5. 教务人员确认状态变更后，系统更新课程状态字段并记录状态变更日志（变更时间、变更人、变更前状态、变更后状态、变更原因备注）。
  6. 上架操作完成后，该课程出现在课程列表的"已上架"筛选项中，可被用于创建新班级。系统可向相关教务人员发送课程上架通知。
  7. 下架操作完成后，该课程不出现在可选课程列表中，但已有班级不受影响仍然正常进行。在创建新班级时，已下架的课程不出现在可选课程列表中。
- **输出**：状态变更成功提示
- **业务规则**：
  - BR-CRS-005-01：课程上架前必须完成课时配置和费用标准设置，否则不可上架。
  - BR-CRS-005-02：课程下架后，已有班级不受影响但不可创建新班级。
  - BR-CRS-005-03：课程状态变更操作需记录完整的操作日志。
  - BR-CRS-005-04：已下架的课程可在完成必要调整后重新上架，上架后恢复所有正常功能。
- **优先级**：中
- **关联涉众**：教务人员

### 3.3 班级管理模块

#### 3.3.1 功能描述

班级管理模块负责班级的创建、排课、学员管理和教室资源分配。班级是课程的具体实例，关联了课程、教师、学员和教室等核心业务实体。一个课程类型可以开设多个班级实例，每个班级有独立的学员名单、上课时间安排和生命周期状态。班级管理模块还负责处理排课冲突检测和教室资源的调度管理。

#### 3.3.2 功能需求

##### REQ-CLS-001：班级创建

- **功能描述**：基于课程创建具体的班级实例
- **输入**：关联课程编号、班级名称、开班日期、预计结业日期、最大学员数、校区、教室编号、授课教师编号
- **处理过程**：
  1. 教务人员进入班级管理模块，点击"创建班级"按钮，系统展示班级创建表单页面。页面首先要求选择课程类型，系统以卡片或列表形式展示当前校区下所有"已上架"的课程。
  2. 教务人员选择目标课程后，系统自动填充课程关联的默认配置信息：根据课程的课时配置自动计算并显示建议的总课时数和课程有效期，根据课程费用标准显示参考学费价格。班级名称自动生成建议值，格式为"{课程名}-{当前年份}年第{N}期班"，教务人员可修改。
  3. 教务人员设置班级核心参数：开班日期（通过日期选择器选择，须为未来日期），预计结业日期（系统根据总课时数、每课时分钟数和假设的每周上课频率自动计算建议值，教务人员可调整），最大学员数（默认值为教室容量，教务人员可调低但不可超过教室容量）。
  4. 教务人员选择校区（下拉选择框，展示所有启用的校区），选择教室（根据所选校区动态加载该校区下所有"可用"状态的教室列表，每个教室显示名称和容量），选择授课教师（从教师列表中选择，教师列表按所授课程类型过滤，只显示能教授该课程的教师）。
  5. 教务人员提交创建表单后，系统执行校验：开班日期必须在当前日期之后，预计结业日期必须晚于开班日期，最大学员数必须为正整数且不超出教室容量，所选教师必须关联了该课程类型的授课能力标签。
  6. 校验通过后，系统在数据库事务中创建班级记录：生成班级编号（格式：CLS-YYYYMMDD-四位序号，如CLS-20260622-0001），设置班级状态为"待开课"，保存所有班级配置参数到班级信息表，同时在教师课表预留表中创建占位记录（具体时间在排课环节填写）。
  7. 创建成功后，系统返回班级详情页面，展示班级完整信息，并提供"去排课"快捷按钮引导教务人员进入排课管理流程。系统记录班级创建操作日志（操作人、操作时间、创建参数）。
- **输出**：班级创建成功提示
- **业务规则**：
  - BR-CLS-001-01：班级名称在同一校区内必须唯一（同一课程可有多期班级，靠期号区分）。
  - BR-CLS-001-02：开班日期必须在课程有效期起始日期之后，预计结业日期不可超过课程有效期截止日期。
  - BR-CLS-001-03：最大学员数必须为正整数，取值范围为1到所选教室的容量上限。
  - BR-CLS-001-04：班级创建后默认状态为"待开课"，需在开班日期前完成排课操作。
  - BR-CLS-001-05：同一教室在同一时间段（日期+时段）只能被一个班级占用，创建时不检测时间冲突（排课时检测）。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-CLS-002：排课管理

- **功能描述**：为班级安排具体的上课时间和教室
- **输入**：班级编号、上课日期、开始时间、结束时间、教室编号、教师编号
- **处理过程**：
  1. 教务人员在班级详情页面点击"排课"标签或"新增排课"按钮，进入排课管理页面。页面左侧显示排课日历视图（周视图或月视图可切换），右侧显示排课表单和历史排课列表。
  2. 教务人员在排课表单中选择上课日期（通过点击日历视图中的日期格子或使用日期选择器）、上课开始时间和结束时间（通过时间选择器，支持15分钟粒度），系统根据课程配置中的每课时分钟数自动建议结束时间。
  3. 系统在教务人员选择日期和时间后自动执行三级冲突检测：第一级检测所选教师在该时段是否已有其他排课安排，若冲突则在日历视图上以红色标记冲突时段并显示冲突的班级名称；第二级检测所选教室在该时段是否已被其他班级占用，若冲突则列出占用该教室的班级信息；第三级检测该班级学员中是否有部分学员在该时段已被安排了其他课程（跨班级时间冲突检测）。
  4. 若存在任何冲突，系统在排课确认前展示完整的冲突详情列表：每条冲突显示冲突类型（教师冲突/教室冲突/学员冲突）、冲突对象、冲突时段和建议替代时段。系统自动推荐该班级所有可用时段（基于教室空闲、教师空闲、最少学员冲突三个维度综合评分），以绿色标记可用时段。
  5. 教务人员确认排课安排无冲突或接受冲突后，点击"确认排课"。系统在数据库事务中创建课程安排记录（Schedule），包含排课ID、班级编号、上课日期、开始时间、结束时间、教室编号、授课教师编号、排课状态（"已排课"）等信息。
  6. 排课记录创建后，系统同步更新三个关联表：教师课程表（teacher_schedule）中添加该时段的授课记录，教室占用表（room_occupancy）中添加该时段的占用记录，班级学员课表（student_schedule）中为每个班级学员添加一条课程安排通知记录。
  7. 系统触发排课通知流程：向授课教师发送新排课通知（包含班级名称、上课日期和时间、教室、学员人数），向所有班级学员/家长发送课程安排通知（包含课程名称、上课日期和时间、教室）。通知发送记录保存到通知表。
- **输出**：排课成功提示，冲突检测结果
- **业务规则**：
  - BR-CLS-002-01：上课开始时间必须早于结束时间，单次排课时长不少于60分钟、不超过240分钟。
  - BR-CLS-002-02：同一教师在同一天同一时段最多只能安排一个班级的课程。
  - BR-CLS-002-03：同一教室在同一天同一时段最多只能被一个班级占用。
  - BR-CLS-002-04：排课日期必须在班级的开班日期之后、预计结业日期之前。
  - BR-CLS-002-05：系统记录排课的冲突检测结果，若教务人员在有冲突的情况下强制排课，需填写强制排课原因。
- **优先级**：高
- **关联涉众**：教务人员、教师、学生/家长

##### REQ-CLS-003：班级学员管理

- **功能描述**：查看和管理班级中的学员列表
- **输入**：班级编号
- **处理过程**：
  1. 教务人员或教师在班级详情页面点击"学员管理"标签，系统加载并展示该班级的学员信息管理页面。
  2. 页面顶部展示班级基本信息摘要横幅：班级名称、关联课程、授课教师姓名、教室名称、当前学员人数/最大学员人数（进度条展示）、班级状态标识。
  3. 系统查询班级-学员关联表，加载该班级下所有学员的名单列表，以表格形式展示：学员编号、姓名、性别、联系电话、入班日期、缴费状态（已缴清/部分缴费/未缴费，带颜色标签）、出勤率（百分比+进度条）、最近一次签到日期。
  4. 学员列表上方提供搜索框和筛选器，支持按学员姓名搜索、按缴费状态筛选（全部/已缴清/部分缴费/未缴费）、按出勤率排序（升序/降序），帮助教务人员快速定位特定学员。
  5. 对于每行学员记录，提供操作按钮："查看档案"（跳转学员档案页）、"转班"（触发转班流程，参见REQ-STU-005）、"退学"（触发退学流程，参见REQ-STU-006）。操作按钮根据学员状态和用户权限动态显示。
  6. 页面提供"添加学员"快捷按钮，点击后弹出分班对话框（参见REQ-STU-004），教务人员可选择"待分班"状态且课程类型匹配的学员添加到当前班级，操作成功后班级学员列表实时刷新。
  7. 教务人员可从班级中移除学员，点击学员行的"移出班级"按钮，系统弹出确认对话框让教务人员选择移出原因（转班/退学/数据修正），确认后系统执行相应的转班或退学流程。
- **输出**：班级学员管理页面
- **业务规则**：
  - BR-CLS-003-01：只有状态为"待开课"或"进行中"的班级才能添加新学员。
  - BR-CLS-003-02：班级学员人数已达上限后，添加学员按钮变为不可用状态。
  - BR-CLS-003-03：从班级移除学员必须选择移除原因并填写备注说明。
  - BR-CLS-003-04：学员列表支持导出为Excel格式，用于线下统计或点名表打印。
- **优先级**：高
- **关联涉众**：教务人员、教师

##### REQ-CLS-004：教室资源管理

- **功能描述**：管理培训机构的教室资源
- **输入**：教室名称、容量、所在校区、设施配置、状态
- **处理过程**：
  1. 管理人员（机构管理员或教务人员）进入教室资源管理页面，页面展示所有教室的列表视图，每条记录显示教室名称、容量、所在校区、当前状态和占用率（当日）。
  2. 管理人员点击"新增教室"按钮，系统展示教室信息表单：教室名称（如"301多媒体教室"）、容量（座位数，正整数）、所在校区（下拉选择）、设施配置（多选框：投影仪、白板/黑板、电脑、音响、空调等）、教室位置描述（楼层和房间号说明）。
  3. 管理人员填写完成后提交，系统校验教室名称在所在校区内的唯一性，容量需在1-500人范围内，所在校区必须为系统已配置的启用校区。
  4. 校验通过后，系统创建教室资源记录并自动生成教室编号（格式：ROOM-校区代码-序号），初始状态设为"可用"。系统记录创建操作日志。
  5. 在教室详情页，系统提供教室使用日历视图，以日历热力图或时间线形式展示该教室在各时段（日/周/月视图）的占用情况：已被排课占用的时段标记为红色（已占用），空闲时段标记为绿色（可用），维护中时段标记为灰色。点击已占用时段可查看占用该教室的班级和课程详情。
  6. 管理人员可修改教室状态：设置为"可用"（正常使用）、"维护中"（设备维修或清洁中，暂时不可排课）、"停用"（长期不使用，不参与排课）。切换为"维护中"或"停用"时，系统检测该教室是否有未来的排课安排，若有则提示管理人员需先将这些排课转移到其他教室。
  7. 管理人员可编辑教室信息（名称、容量、设施配置等），修改容量时系统检测是否影响已有班级（若修改后容量小于某班级的当前学员数，则提示容量不足并阻止保存）。所有修改操作记录完整的变更日志。
- **输出**：教室管理列表，教室使用日历
- **业务规则**：
  - BR-CLS-004-01：教室名称在同一校区内必须唯一。
  - BR-CLS-004-02：教室容量修改后，不影响已有班级的最大学员数（班级创建时已快照教室容量）。
  - BR-CLS-004-03：状态为"停用"的教室不出现在排课可选教室列表中。
  - BR-CLS-004-04：删除教室前需确保该教室无历史排课记录，否则只能停用不可删除。
- **优先级**：中
- **关联涉众**：教务人员

##### REQ-CLS-005：班级状态管理

- **功能描述**：管理班级的生命周期状态
- **输入**：班级编号、目标状态
- **处理过程**：
  1. 教务人员在班级详情页面看到班级当前状态的标识和状态流转操作按钮。班级状态流转路径为：待开课 → 进行中 → 已结业。每个状态对应不同的可用操作集合。
  2. 当班级状态为"待开课"时，教务人员可执行"开班"操作将状态变更为"进行中"。系统校验当前日期是否已达到或超过开班日期，若尚未到达开班日期，系统提示"开班日期为YYYY-MM-DD，当前尚不能开班"并阻止操作。
  3. 教务人员执行"开班"操作时，系统确认班级已完成排课（至少有一条课程安排记录），确认班级已有学员（学员人数大于0），若两个条件均满足则执行状态变更：将班级状态更新为"进行中"，记录开班实际日期和操作人。
  4. 当班级状态为"进行中"时，教务人员可执行"结业"操作。系统首先执行结业前置检查：确认所有计划排课是否已完成（所有排课的考勤均已登记完毕），确认班级内所有学员的处理方案（在读学员需全部转班或退学处理完毕，不可有遗留的在读学员）。
  5. 结业检查通过后，系统弹出结业确认对话框，展示班级汇总信息：总排课次数、已完成次数、学员总数、已结业学员数、总课时消耗数等。教务人员确认后执行结业操作。
  6. 结业操作在数据库事务中执行：将班级状态更新为"已结业"，记录结业日期和操作人。结业后的班级在系统中保留所有历史数据（排课记录、学员记录、考勤记录等）供查询和统计使用。
  7. 班级结业后，所有与该班级相关的操作（添加学员、排课、转班入该班等）均被禁用。班级在列表中的"已结业"筛选条件下可见，在"进行中"筛选条件下不可见。
- **输出**：状态变更成功提示
- **业务规则**：
  - BR-CLS-005-01：班级状态只能沿"待开课→进行中→已结业"单向流转，不可逆向操作。
  - BR-CLS-005-02："待开课"的班级在开班日期前不可变更为"进行中"。
  - BR-CLS-005-03：班级结业前需确认无在读学员遗留，否则阻止结业操作。
  - BR-CLS-005-04：已结业的班级所有数据只读保留，不可新增或修改任何关联数据。
  - BR-CLS-005-05：班级结业后自动触发教师课酬结算待办任务。
- **优先级**：中
- **关联涉众**：教务人员

### 3.4 教师管理模块

#### 3.4.1 功能描述

教师管理模块负责教师信息的维护、授课安排管理以及课时统计。系统支持全职和兼职教师的管理，每位教师可关联多个可教授课程类型。教师管理模块为排课冲突检测提供教师可用时段数据，同时为财务结算提供课时统计和课酬计算的基础数据。

#### 3.4.2 功能需求

##### REQ-TCH-001：教师信息管理

- **功能描述**：维护教师基本信息
- **输入**：教师姓名、性别、联系电话、教授课程类型、教师类型（全职/兼职）、课酬标准（元/课时）、可授课时间、所属校区
- **处理过程**：
  1. 教务人员进入教师管理模块，点击"新增教师"按钮，系统展示教师信息录入表单。表单分为基本信息区（姓名、性别、联系电话等）和授课信息区（教授课程类型、教师类型、课酬标准、可授课时间等）。
  2. 教务人员填写教师基本个人信息：姓名、性别、联系电话、身份证号（可选，用于全职教师管理）、入职日期（全职教师）或合作开始日期（兼职教师）、所属校区。
  3. 教务人员配置教师的授课能力信息：从课程类型列表（多选）中选择教师能够教授的课程类型（如：数学辅导、物理辅导等），一个教师可勾选多个课程类型。该配置用于排课时过滤可选教师列表。
  4. 教务人员设置教师的课酬标准：填写每课时的报酬金额（元/课时），课酬标准可根据教师类型（全职/兼职）、授课课程类型分别设置不同的金额（如全职教师80元/课时，兼职教师100元/课时）。系统支持按课程类型设置差异化课酬。
  5. 教务人员配置教师的可授课时间段：通过时间选择网格设置教师每周的可用时段（如：周一至周五14:00-20:00、周六周日9:00-18:00），该配置用于排课冲突检测时判断教师是否有时间接受新的排课安排。
  6. 系统校验必填字段（姓名、联系电话、至少一个教授课程类型、课酬标准）是否完整填写，联系电话格式是否正确。校验通过后，系统生成教师编号（格式：TCH-YYYYMMDD-四位序号），创建教师档案记录。
  7. 创建成功后，该教师自动获得系统的"教师"角色账号（若尚未创建系统账号），系统生成临时密码并通过短信发送给教师，教师首次登录后需修改密码。
- **输出**：教师档案创建成功
- **业务规则**：
  - BR-TCH-001-01：教师编号格式固定为 TCH-YYYYMMDD-NNNN，编号一经生成不可修改。
  - BR-TCH-001-02：每位教师至少关联一个可教授课程类型，否则无法参与排课。
  - BR-TCH-001-03：课酬标准必须大于0，支持按课程类型设置不同课酬。
  - BR-TCH-001-04：教师联系电话需唯一，同一手机号不能关联多个教师记录。
  - BR-TCH-001-05：全职和兼职教师在课酬计算和可授课时段配置上有不同的默认策略。
- **优先级**：高
- **关联涉众**：教务人员

##### REQ-TCH-002：授课安排查看

- **功能描述**：教师查看自己的课程安排
- **输入**：教师编号、查询日期范围
- **处理过程**：
  1. 教师登录系统后，系统根据其登录账号关联的教师编号自动识别身份，首页默认展示"我的课程"仪表盘页面。
  2. 系统查询该教师的所有排课安排（schedule表），按上课日期筛选并加载课程详细信息（班级名称、课程名称、教室名称、学员人数、上课时间等）。
  3. 页面以可切换的视图模式展示课程安排：日视图（显示当天的课程时间线，精确到小时）、周视图（7天网格，每格显示该时段是否有课和课程名称）、月视图（月历网格，有课日期以圆点标记，点击弹出当日课程列表），教师可通过标签页切换视图模式。
  4. 每节课程卡片展示详细信息：课程名称、班级名称、上课时间段（开始-结束时间）、教室名称（含楼层和房间号）、当前学员人数、班级状态标签。教师点击课程卡片可进入该节课的详细页面。
  5. 课程详细页面展示该节课的完整学员名单（姓名、出勤状态），教师可在此页面进行签到操作（参见REQ-ATT-001），查看考勤历史记录，以及查看该班级在前几次课的教学备注。
  6. 系统在课程开始前30分钟向教师推送上课提醒（系统内消息通知和可选短信通知），提醒内容包含课程名称、上课时间、教室和学员人数。
  7. 教师可导出个人课程表为PDF或打印格式，用于线下参考。课程表导出内容包含日期、时间、课程、班级、教室等完整信息。
- **输出**：教师课程表视图
- **业务规则**：
  - BR-TCH-002-01：教师只能查看自己的课程安排，不可查看其他教师的排课数据。
  - BR-TCH-002-02：课程安排默认显示本周数据，支持前后翻页查看历史或未来安排。
  - BR-TCH-002-03：已结业班级的课程安排在教师课表中以灰色文字展示，区别于进行中的活跃课程。
  - BR-TCH-002-04：上课提醒通知仅对进行中班级的课程安排生效。
- **优先级**：高
- **关联涉众**：教师

##### REQ-TCH-003：课时统计

- **功能描述**：统计教师的授课课时数
- **输入**：教师编号、统计时间段
- **处理过程**：
  1. 财务人员或教师进入课时统计页面，选择统计的时间段范围（本月、上月、本季度、本年度或自定义日期范围），系统默认展示当前月份的数据。
  2. 系统查询该教师在选定时间段内的所有课程安排记录（schedule表），筛选状态为"已完成"且教师已完成授课考勤登记的排课记录，排除因故取消的课程安排。
  3. 系统逐条计算每条课程安排对应的课时数（根据班级关联课程的每课时分钟数进行折算：实际上课分钟数除以每课时分钟数得到课时数，不满一课时按比例折算），汇总得到总授课课时数。
  4. 系统根据教师的课酬标准（从教师信息表中获取，支持按课程类型差异化课酬）乘以各类课程的总课时数，分别计算每类课程的课酬金额，最后汇总得到应发课酬总额。
  5. 系统检查是否存在考勤异常情况：教师迟到（签到时间晚于上课时间超过15分钟）、早退（签退时间早于下课时间超过15分钟）、缺课（有排课但未签到），根据系统配置的扣款规则计算扣款金额，在课酬总额中扣除后得到实发课酬。
  6. 课时统计结果以表格形式展示：每行一条授课记录，包含上课日期、班级名称、课程名称、上课时间、课时数、课酬单价、课酬金额。表格下方显示汇总行：总课时数、应发课酬、扣款明细、实发课酬。
  7. 财务人员确认课时统计数据无误后，可点击"导出"将统计结果导出为Excel或PDF格式。若机构配置了与HR系统的对接，系统自动将课时统计数据推送到HR系统的接口。
- **输出**：课时统计表，包含总课时、课酬金额
- **业务规则**：
  - BR-TCH-003-01：课时统计仅计算状态为"已完成"且教师已完成签到/签退的课程安排。
  - BR-TCH-003-02：课酬单价从教师信息表中实时获取，若教师在统计期间有课酬标准变更，按变更生效日期分段计算。
  - BR-TCH-003-03：课时数计算精确到0.5课时（如实际上课40分钟，标准课时45分钟，则计为0.9课时，四舍五入到0.5课时）。
  - BR-TCH-003-04：教师可在每月5日前对上月课时统计数据提出异议，超过5日未反馈则默认确认。
  - BR-TCH-003-05：课时统计数据作为课酬发放的依据，导出后自动锁定不可修改（除非管理员解锁）。
- **优先级**：高
- **关联涉众**：教师、财务人员

##### REQ-TCH-004：教师课程学员查看

- **功能描述**：教师查看所授课程班级的学员名单
- **输入**：班级编号或课程安排编号
- **处理过程**：
  1. 教师通过"我的课程"页面或"课程详情"页面进入指定班级的学员管理视图，系统自动根据教师编号验证该教师是否为该班级的授课教师。
  2. 系统查询班级-学员关联表，加载该班级所有在读学员的完整名单，以列表形式展示每位学员的核心信息。
  3. 每条学员记录展示：学员编号、姓名、性别、年龄（根据出生日期自动计算）、联系电话、入班日期、累计出勤率（百分比+进度条）、当前缴费状态标识（已缴清/部分缴费/未缴费）。
  4. 教师可点击学员姓名进入该学员的简要档案页（与教务人员看到的完整档案有所区别，教师仅可查看学员的基本信息和在本班级的考勤记录，不可查看其他班级的信息和缴费明细）。
  5. 学员列表支持按姓名搜索、按出勤率排序、按缴费状态筛选，帮助教师在上课前快速了解本班学员的基本情况。列表还提供"导出点名表"按钮，可将学员名单导出为可打印的点名表格式。
  6. 在课程安排详情页（上课当天），系统自动展示该节课的学员签到面板，每个学员行旁边提供"签到"、"请假"、"缺勤"三个快捷状态按钮，教师可一键完成考勤登记（参见REQ-ATT-001）。
  7. 教师可查看每个学员在本班级的历史出勤记录摘要：应到次数、实到次数、请假次数、缺勤次数，以及最近5次课的出勤状态明细。对于连续缺勤3次以上的学员，系统用黄色高亮标记提醒教师关注。
- **输出**：班级学员名单
- **业务规则**：
  - BR-TCH-004-01：教师仅可查看自己授课班级的学员名单，不可查看其他班级或全局学员数据。
  - BR-TCH-004-02：教师在简版学员档案中不可查看学员的详细缴费金额和财务信息。
  - BR-TCH-004-03：导出点名表功能仅在课程开始前24小时至课程结束后24小时内可用。
  - BR-TCH-004-04：教师对学员信息的查看权限仅限于在读学员，已退学学员自动从教师视图隐藏。
- **优先级**：中
- **关联涉众**：教师

### 3.5 收费管理模块

#### 3.5.1 功能描述

收费管理模块是系统的核心功能之一，负责培训机构的全部收费、退费、催缴和台账管理。该模块涉及财务人员、教务人员、学生/家长的共同使用。系统支持多种支付方式（现金、微信支付、支付宝、银行转账），自动计算应收费用并生成收据和发票，提供完整的欠费检测与催缴机制，以及严谨的退费处理流程。

#### 3.5.2 功能需求

##### REQ-FEE-001：收费项目设置

- **功能描述**：设置机构的收费项目类型
- **输入**：收费项目名称（学费、教材费、杂费、活动费等）、费用类型（一次性/按期/按课时）、默认金额、是否必收
- **处理过程**：
  1. 财务人员进入收费管理模块的"收费项目设置"页面，系统展示当前已配置的所有收费项目列表，每条记录显示项目名称、费用类型、默认金额、是否必收、关联课程数等关键信息。
  2. 财务人员点击"新增收费项目"按钮，系统弹出收费项目配置表单，包含收费项目名称（从预设的项目类型中选择或手动输入自定义名称）、费用类型（单选：一次性费用、按期费用、按课时费用）、默认金额（元）、是否必收（开关按钮）、适用课程范围（多选课程类型）以及备注说明。
  3. 财务人员配置"是否必收"选项：若设为必收项，则该收费项目将自动加入所有关联课程类型的学员首期账单中，无需教务人员手动选择。必收项通常为学费和教材费等核心收费项目。
  4. 财务人员选择该收费项目的适用课程范围（可多选），系统记录课程-收费项目的关联关系。同一课程可关联多个收费项目（如：数学辅导课关联"学费"必收项和"教材费"选收项）。
  5. 财务人员提交配置后，系统校验收费项目名称的唯一性（同一校区内不可重名）、默认金额合法性（必须为非负数，精确到分）、适用课程范围至少选择一项课程类型。
  6. 校验通过后，系统保存收费项目配置并更新课程-收费项目关联表。若财务人员修改了已有收费项目的"是否必收"属性或默认金额，系统提示该变更可能影响后续新建班级的收费账单生成。
  7. 财务人员可对收费项目进行启用/停用操作，停用的收费项目不出现在新建账单时的可选收费项列表中，但历史已使用的记录不受影响。
- **输出**：收费项目配置保存成功
- **业务规则**：
  - BR-FEE-001-01：收费项目名称在同一校区内必须唯一。
  - BR-FEE-001-02：至少保留一个"学费"类型的收费项目且该项目的"是否必收"属性必须为"是"。
  - BR-FEE-001-03：默认金额不得为负数，若为0则表示该收费项目金额由教务人员在具体账单中手动填写。
  - BR-FEE-001-04：收费项目的修改和停用操作需记录完整的操作日志。
  - BR-FEE-001-05：停用的收费项目在重新启用前不出现在任何账单生成流程中。
- **优先级**：高
- **关联涉众**：财务人员

##### REQ-FEE-002：学费收取

- **功能描述**：向学员收取学费及各项费用
- **输入**：学员编号、收费项目（可多选）、实收金额、支付方式、优惠金额、备注
- **处理过程**：
  1. 财务人员在收费管理模块点击"新建收费"按钮，通过学员搜索框查找并选择待收费的学员，系统加载该学员的基本信息和当前缴费状态摘要。
  2. 系统自动计算应收费用的详细明细：首先加载学员所在班级关联的课程费用标准，然后查询学员的班级入班日期和已缴费用记录，区分首期费用（入班时的一次性交费）和续期费用（按课程周期分期交费）。系统自动列出所有适用于该学员的收费项目（包含必收项和选收项）。
  3. 系统加载该学员可适用的优惠方案列表，自动计算每个优惠方案的优惠金额。若学员满足多个优惠条件（如早鸟优惠+老带新优惠），系统根据优惠叠加规则计算最终的优惠金额（参见REQ-FEE-008优惠叠加规则）。
  4. 系统计算优惠后的应收总额：应收总额 = 各项收费项目金额之和 - 优惠金额。同时检查学员是否有历史欠费，若有则在账单明细中单独列出"历史欠费"项目并合并计算总应收金额。
  5. 财务人员在收费确认页面上核对收费项目、各项金额、优惠方案和最终应收金额，可手动调整优惠金额（调整原因需备注），选择支付方式（下拉选择：现金、微信支付、支付宝、银行转账），填写备注信息。
  6. 财务人员点击"确认收款"后，系统在数据库事务中执行以下操作：创建缴费记录（FeeRecord表）包含缴费编号、学员编号、应收金额、优惠金额、实收金额、支付方式等字段，生成全局唯一且连续的收据编号，更新学员的缴费状态（若全额缴清则标记为"已缴清"，部分缴费则标记为"部分缴费"），记录完整的收费操作日志。
  7. 收费成功后，系统跳转至收据预览页面，展示收据的完整信息（参见REQ-FEE-003）。若支付方式为微信支付或支付宝，系统调用对应的支付平台API进行在线收款，收款成功后回调确认。系统向学员/家长发送缴费成功通知。
- **输出**：收款成功提示，收据信息，学员缴费状态更新
- **业务规则**：
  - BR-FEE-002-01：每次收费操作必须在数据库事务中完成，确保缴费记录、收据编号、学员状态三者的数据一致性。
  - BR-FEE-002-02：实收金额必须等于应收总额减去优惠金额，不允许存在无原因的金额差异。
  - BR-FEE-002-03：收据编号格式为 RCPT-YYYYMMDD-NNNNN（5位数递增序号），全局唯一不可重复。
  - BR-FEE-002-04：现金收费超过5000元时，系统提示财务人员进行双人复核（若机构配置了此规则）。
  - BR-FEE-002-05：在线支付（微信/支付宝）超时时间为5分钟，超时未支付则自动取消该笔收费请求。
- **优先级**：高
- **关联涉众**：财务人员、学生/家长

##### REQ-FEE-003：收据打印

- **功能描述**：打印收费收据
- **输入**：缴费记录编号
- **处理过程**：
  1. 财务人员在收费完成后自动跳转至收据预览页面，或在收费台账中选择一条缴费记录点击"打印收据"按钮，系统加载该缴费记录的完整信息。
  2. 系统根据收据模板生成收据预览内容，包含以下信息区域：机构抬头（机构名称、Logo、地址、联系电话）、收据编号（RCPT-YYYYMMDD-NNNNN）、学员信息（姓名、联系电话）、收费项目明细表（项目名称、数量、单价、金额）、优惠信息（优惠类型、优惠金额）、金额合计（大写和小写）、支付方式、收款日期、收款人签名区。
  3. 财务人员在收据预览页面上核对所有信息是否准确无误，可调整打印设置（纸张大小默认A5、打印份数等）。系统支持A5热敏纸打印和A4激光打印两种模板。
  4. 财务人员点击"打印"按钮，系统调用浏览器打印功能或本地打印服务，将收据内容发送到打印机。系统记录打印操作日志（打印时间、打印人、缴费记录编号）。
  5. 对于补打收据的场景（原收据丢失或损坏），财务人员在收费台账中选择需要补打的记录，点击"补打收据"按钮。系统在原收据模板基础上添加醒目的"补打"水印字样和补打日期，以区分原始收据和补打收据。
  6. 补打操作需填写补打原因（如：学员丢失、原收据损坏等），系统记录补打日志。同一缴费记录的补打次数上限为3次，超过需管理员审批。
  7. 系统支持将收据导出为PDF格式并发送至学员的电子邮箱（若学员档案中有邮箱信息），作为电子收据留存。
- **输出**：收据打印输出
- **业务规则**：
  - BR-FEE-003-01：收据编号与缴费记录一一对应，一张缴费记录只能生成一个收据编号。
  - BR-FEE-003-02：补打收据必须显示"补打"水印标识，以区别于原始收据。
  - BR-FEE-003-03：收据打印后不可修改缴费记录中的金额和项目，如需修改需先作废原收据。
  - BR-FEE-003-04：打印失败的收据需重新触发打印，不计入补打次数统计。
- **优先级**：高
- **关联涉众**：财务人员

##### REQ-FEE-004：退费处理

- **功能描述**：处理学员退费申请
- **输入**：学员编号、退费类型（退学退费/课程退费）、退费项目、退费金额
- **处理过程**：
  1. 教务人员发起退费申请（通常关联学员退学流程REQ-STU-006或课程取消），系统自动生成退费待办任务并推送至财务人员的待办工作列表。财务人员在"退费待办"页面看到所有待处理的退费申请。
  2. 财务人员打开退费待办项，系统展示退费申请的完整信息：学员姓名、退费类型、关联的学员退学记录或课程取消记录、学员的缴费历史明细、已消耗课时数等关键数据。
  3. 系统根据退费规则自动计算应退金额的参考值：计算总缴费金额，减去已消耗课时对应的费用（已消耗课时×单课时单价），减去已发生且不可退的杂费（如已发放的教材费），应用当前生效的退费比例规则。
  4. 财务人员审核系统计算的退费金额，可根据实际情况手动调整退费金额（调整原因必须备注说明）。若退费金额超过2000元（可配置阈值），系统自动将退费申请升级为"需管理员审批"状态，推送至机构管理员的审批待办列表。
  5. 财务人员（和管理员，如需审批）确认退费金额后，选择退款方式：原路退回（通过原始支付渠道退款，如微信支付退款到原支付账户、银行转账退款到原银行账户）或现金退款。退款方式需与原始收费的支付方式匹配，若原始为在线支付则优先原路退回。
  6. 财务人员执行退款操作，系统在数据库事务中创建退款记录（RefundRecord表），包含退款编号、关联的原始缴费记录编号、退费金额、退款方式、退款日期、审批人、退款操作人等信息。若为原路退回，系统调用第三方支付平台的退款API完成在线退款。
  7. 退款成功后，系统更新学员的缴费状态（重新计算已缴和应退差额），记录完整的退款操作日志。系统向学员/家长发送退款成功通知（包含退款金额、退款方式、预计到账时间），向教务人员发送退费处理完成通知。
- **输出**：退费处理完成，退款记录
- **业务规则**：
  - BR-FEE-004-01：退费金额不能超过学员实际缴费总额，超额退费需特殊审批。
  - BR-FEE-004-02：退费金额大于2000元（可配置）时需机构管理员审批，审批通过后方可执行退款。
  - BR-FEE-004-03：每笔退费必须关联一条原始缴费记录，用于财务对账和审计追溯。
  - BR-FEE-004-04：原路退回的到账时间取决于第三方支付平台的处理速度，系统不做硬性承诺。
  - BR-FEE-004-05：退款操作不可逆，一旦执行退款不可撤销（如操作失误需重新收费处理）。
- **优先级**：高
- **关联涉众**：教务人员、财务人员、机构管理员

##### REQ-FEE-005：欠费检测与催缴

- **功能描述**：自动检测学员欠费并生成催缴通知
- **输入**：检测时间（系统自动或手动触发）
- **处理过程**：
  1. 系统每日凌晨2:00自动执行欠费检测定时任务，扫描所有状态为"在读"的学员，加载每个学员的缴费记录和应缴费用标准，计算应缴费用的总额与已缴费用的差额。
  2. 对于每个在读学员，系统判断其应缴费用是否已超过应缴日期（根据班级的开班日期和课程周期自动计算应缴费日期）。若超过应缴日期且仍存在未缴费用，则将该学员标记为"欠费"状态，记录欠费金额和欠费起始日期。
  3. 系统根据欠费天数对欠费学员进行分级处理：欠费1-7天的学员触发"初级提醒"级别（系统内消息通知），欠费8-15天的学员触发"中级催缴"级别（系统内消息+短信通知），欠费16-30天的学员触发"高级催缴"级别（系统内消息+短信通知+教务人员跟进标记），欠费超过30天的学员标记为"严重欠费"并生成专项报告推送至财务人员和教务主管。
  4. 系统根据催缴模板自动生成催缴通知内容，模板中包含学员姓名、欠费金额、欠费天数、缴费截止日期、缴费方式说明等信息。通知内容通过消息推送和短信平台发送给学员/家长。
  5. 系统在催缴记录表（dunning_records）中记录每次催缴的详细信息：催缴日期、催缴级别、通知方式、通知内容、发送状态（成功/失败/待发送）。
  6. 财务人员可通过"欠费管理"页面查看当前的欠费情况汇总：欠费总金额、欠费学员总数、按欠费天数分级的分布统计、Top N欠费学员明细列表。财务人员可对特定学员手动触发催缴通知或添加跟进备注记录。
  7. 学员完成缴费后（通过REQ-FEE-002收费流程），系统自动更新该学员的欠费状态，清除欠费标记，关闭所有未完成的催缴记录。若学员部分缴费（未全额缴清），系统重新计算剩余欠费金额并继续催缴流程。
- **输出**：欠费学员清单，催缴通知记录
- **业务规则**：
  - BR-FEE-005-01：欠费检测定时任务的默认执行时间为每日凌晨2:00，可在系统参数中修改。
  - BR-FEE-005-02：同一学员在同一欠费周期内同级别催缴通知最多发送3次，避免重复骚扰。
  - BR-FEE-005-03：短信催缴通知仅在每日9:00-20:00时段内发送。
  - BR-FEE-005-04：欠费超过30天的学员，系统自动限制其部分功能（如不可转班、不可报名新课程）。
  - BR-FEE-005-05：财务人员可为个别学员设置"免催缴"标记（如已与机构达成特殊缴费协议），这些学员不纳入自动催缴范围。
- **优先级**：高
- **关联涉众**：财务人员、学生/家长

##### REQ-FEE-006：收费台账

- **功能描述**：查看和管理所有收费和退费记录
- **输入**：日期范围、收费类型、支付方式、操作用户、校区
- **处理过程**：
  1. 财务人员进入"收费台账"页面，页面顶部提供多条件筛选栏：日期范围（默认展示当月数据）、收费类型（全部/收费/退费）、支付方式（全部/现金/微信支付/支付宝/银行转账）、操作人（下拉搜索）、校区（下拉选择）等筛选维度。
  2. 系统根据筛选条件查询收费和退费记录表，返回符合条件的记录列表，默认按操作时间倒序排列。列表展示每条记录的关键信息：交易日期、学员姓名、交易类型（收费/退费，带颜色标签）、收费项目摘要、应收金额、优惠金额、实收金额、支付方式、收据编号、操作人。
  3. 点击某条记录可展开查看交易的详细信息，包括收费项目明细表（每个收费项目的名称和金额）、优惠方案详情（优惠类型、优惠值、计算过程）、退费关联的原始收费记录（针对退费记录）等完整信息。
  4. 页面底部展示当日或当月的交易汇总统计：收费笔数、收费总金额、退费笔数、退费总金额、净收入（收费总额减退费总额），以及按支付方式的分类汇总。
  5. 财务人员可点击"导出Excel"按钮将当前筛选结果导出为Excel格式文件，导出的内容包含所有展示列表的字段加上展开的明细数据。导出文件名默认格式为"收费台账_YYYYMMDD-YYYYMMDD.xlsx"。
  6. 台账页面提供"日结"功能，财务人员可对当天（或指定日期）的所有收费和退费记录进行日结操作，系统生成日结报表（包含当日交易汇总和明细），日结后的记录标记为"已日结"状态。
  7. 已完成日结的记录默认不允许修改或删除（保证财务数据的完整性），如需修改需由管理员先执行"反日结"操作解除日结锁定。
- **输出**：收费台账列表，汇总统计
- **业务规则**：
  - BR-FEE-006-01：收费台账中的所有记录为只读状态，不可直接编辑或删除，如需修改需通过退费或冲正流程。
  - BR-FEE-006-02：日结操作每日最多执行一次，日结后不可修改当日记录。
  - BR-FEE-006-03：导出的Excel文件包含完整的操作人和操作时间信息，用于财务审计。
  - BR-FEE-006-04：收费台账数据保留期至少5年，满足税务稽查要求。
- **优先级**：高
- **关联涉众**：财务人员

##### REQ-FEE-007：优惠管理

- **功能描述**：管理机构的优惠政策
- **输入**：优惠名称、优惠类型（折扣/满减/赠课）、优惠值、适用范围、有效期
- **处理过程**：
  1. 财务人员进入"优惠管理"页面，系统展示当前所有已配置的优惠方案列表，每条记录显示优惠名称、类型、优惠值、适用范围、有效期、状态（启用/停用）、关联课程数。
  2. 财务人员点击"新增优惠方案"创建新的优惠策略：选择优惠类型（折扣：按百分比减免如9折、8.5折；满减：满一定金额减免固定金额如满5000减300；赠课：报名即赠送N课时），设置优惠值（折扣类型的值为百分比数值，满减类型的值为减免金额，赠课类型的值为赠送课时数）。
  3. 财务人员设置优惠的适用范围：选择适用的课程类型（可多选，留空表示适用所有课程），设置适用的学员类型（全部/新学员/老学员续费），设置最低消费金额门槛（如满5000元以上才可享受该优惠），设置有效期的起止日期。
  4. 财务人员可选择是否允许该优惠与其他优惠叠加使用（参见REQ-FEE-008），设置优惠的优先级（数值越小优先级越高，当多个优惠冲突时按优先级取优），以及设置每人可使用次数上限（如每人限用1次/不限次数）。
  5. 财务人员提交配置后，系统校验优惠值的合法性：折扣值需在1%-99%之间，满减金额必须小于最低消费金额，赠课数必须为正整数，有效期起始日期不能晚于结束日期。
  6. 校验通过后系统保存优惠方案记录，优惠状态默认为"启用"。已启用的优惠方案在收费环节自动被系统加载并计算优惠金额，应用于符合适用范围的学员账单。
  7. 财务人员可对优惠方案进行编辑、停用或删除操作。停用的优惠方案不影响已使用该优惠的历史缴费记录。删除优惠方案时需确认无进行中且适用的缴费账单，否则提示不可删除。
- **输出**：优惠方案保存成功
- **业务规则**：
  - BR-FEE-007-01：同一时间内最多允许启用10个优惠方案，超过时提示财务人员清理过期方案。
  - BR-FEE-007-02：折扣类型的优惠值必须在1%-99%之间，不可设置0折或免费。
  - BR-FEE-007-03：满减金额必须小于该优惠的最低消费金额门槛。
  - BR-FEE-007-04：已过期超过30天的优惠方案系统自动标记为"已过期"并不再出现在可选列表中。
  - BR-FEE-007-05：优惠方案删除为逻辑删除（软删除），历史数据保留关联关系不受影响。
- **优先级**：中
- **关联涉众**：财务人员

##### REQ-FEE-008：优惠叠加规则

- **功能描述**：定义和管理多个优惠方案同时适用时的叠加规则
- **输入**：优惠方案列表、叠加规则类型（允许叠加/互斥/取最优）、叠加顺序
- **处理过程**：
  1. 财务人员在优惠管理页面中为每个优惠方案设置"叠加策略"属性，可选项为：允许叠加（该优惠可与其他优惠同时生效）、互斥（该优惠不可与任何其他优惠同时使用）、部分互斥（不可与指定类型的优惠叠加）。
  2. 财务人员可配置全局的优惠叠加规则：设置叠加时的计算模式，可选"顺序叠加模式"（优惠A先计算，优惠B在优惠A的结果基础上再计算）或"独立计算取最优模式"（各优惠独立计算，取对学员最有利的结果）。
  3. 在收费环节（REQ-FEE-002），系统加载学员所有适用的优惠方案后，根据优惠叠加规则引擎自动判断哪些优惠可以同时生效。首先过滤掉互斥组中优先级较低的优惠，保留优先级最高的一个，然后将允许叠加的优惠按优先级从高到低排序。
  4. 若计算模式为"顺序叠加模式"，系统按优先级从高到低依次应用每个优惠：第一个优惠在原始金额上计算折扣或减免，第二个优惠在上一步计算结果上继续计算，依此类推。每个优惠的应用过程均产生中间计算明细记录。
  5. 若计算模式为"独立计算取最优模式"，系统分别计算每个优惠方案独立应用后的最终金额，以及各优惠组合叠加后的金额，从中选出最终实付金额最低（对学员最有利）的方案作为推荐优惠组合。
  6. 系统在收费确认页面上展示优惠叠加的详细计算过程，包含：原始应收金额、每个优惠方案的名称和优惠值、叠加计算步骤（每步后的中间金额）、最终优惠总额和应收金额。财务人员和学员可清晰了解优惠的计算逻辑。
  7. 财务人员可在特殊情况下手动调整最终的优惠金额，手动调整需填写调整原因并记录调整日志。手动调整的优惠金额不能超过系统自动计算的最优优惠金额。
- **输出**：优惠叠加规则保存成功，收费时自动应用叠加规则
- **业务规则**：
  - BR-FEE-008-01：同一笔收费最多允许3个优惠方案同时叠加生效，超出则取优先级最高的3个。
  - BR-FEE-008-02：折扣类优惠之间存在互斥关系时，系统自动选择折扣力度最大（金额最低）的方案。
  - BR-FEE-008-03：满减类优惠必须满足各自的最低消费门槛才能参与叠加计算。
  - BR-FEE-008-04：赠课类优惠不参与金额叠加计算，独立生效，但需与其他优惠的适用条件分别校验。
  - BR-FEE-008-05：优惠叠加的最终优惠金额不得超过原始应收金额的50%（即最低缴费比例为50%），防止过度优惠。
- **优先级**：中
- **关联涉众**：财务人员

##### REQ-FEE-009：发票管理

- **功能描述**：为学员的缴费记录开具和管理发票
- **输入**：缴费记录编号、发票类型（增值税普通发票/增值税专用发票）、发票抬头、纳税人识别号、发票内容
- **处理过程**：
  1. 学员/家长或财务人员在缴费完成后，可在收费详情页面或收费台账中点击"申请发票"按钮，系统跳转至发票申请页面。
  2. 发票申请页面要求填写以下信息：发票类型（下拉选择：增值税普通发票-个人/增值税普通发票-企业/增值税专用发票）、发票抬头（个人姓名或企业全称）、纳税人识别号（企业发票必填，系统自动校验税号格式的合法性）、发票内容（从预设选项中选择，如：培训费、教育服务费等）、接收邮箱（电子发票接收地址，必填）。
  3. 系统校验发票申请的合法性：一张缴费记录只能开具一张发票（不可重复开票），若该缴费记录已开具过发票则提示"该缴费记录已开票"；增值税专用发票需额外校验纳税人识别号是否为有效的18位统一社会信用代码。
  4. 财务人员在"发票管理"页面查看待处理的发票申请列表，审核发票信息是否准确。对于增值税专用发票，财务人员需核对纳税人识别号与企业名称的匹配性（通过内置的纳税人信息验证或人工核对）。
  5. 财务人员审核通过后，系统生成发票记录：包含发票编号（格式：INV-YYYYMMDD-NNNNN，全局唯一）、关联的缴费记录编号、发票类型、发票抬头、纳税人识别号、发票金额、开票日期、发票状态（"已开具"）。
  6. 系统调用电子发票服务（或手动录入发票信息），生成电子发票PDF文件并自动发送至申请人填写的接收邮箱。系统在发票记录中保存电子发票文件的存储路径和发送状态。
  7. 发票开具完成后，系统在缴费记录上标记"已开票"状态，若学员后续发生退费（REQ-FEE-004），系统自动检查该缴费记录是否已开票，若已开票则提示财务人员需先进行发票冲红操作才能完成退费。
- **输出**：发票开具成功，电子发票PDF发送
- **业务规则**：
  - BR-FEE-009-01：一张缴费记录只能开具一张发票，不可重复开票。若发票信息有误需作废原发票后重新开具。
  - BR-FEE-009-02：发票金额必须等于缴费记录中的实收金额，不可多开或少开。
  - BR-FEE-009-03：已开具发票的缴费记录若发生退费，必须先执行发票冲红操作（开具红字发票），冲红完成后方可执行退款。
  - BR-FEE-009-04：电子发票的存储期限为10年，满足税务法规对发票留存的要求。
  - BR-FEE-009-05：系统记录完整的发票开具日志，包含申请时间、审核时间、开票时间、操作人等。
- **优先级**：高
- **关联涉众**：财务人员、学生/家长

### 3.6 考勤管理模块

#### 3.6.1 功能描述

考勤管理模块负责学员上课的签到、请假、缺勤管理以及考勤统计。教师通过系统或移动端在课堂上进行考勤登记，学员/家长可通过系统提交请假申请。考勤数据将用于计算学员的出勤率、为教务人员提供学员学习状态参考、以及为退费计算提供已消耗课时依据。

#### 3.6.2 功能需求

##### REQ-ATT-001：学员签到

- **功能描述**：教师对上课学员进行签到登记
- **输入**：课程安排编号、班级编号、签到学员列表、签到时间
- **处理过程**：
  1. 教师在上课前或上课期间登录系统，进入"今日课程"页面，系统自动展示该教师当天的所有课程安排列表，每节课程显示课程名称、班级名称、上课时间、教室和学员人数。
  2. 教师点击某节课程的"签到"按钮，系统加载该班级的完整学员名单（仅包含状态为"在读"的学员），展示在签到面板中。每个学员行旁边提供三个状态按钮："签到"（绿色）、"请假"（黄色）、"缺勤"（红色），默认为"未签到"状态。
  3. 教师对到课学员逐个点击"签到"按钮（或使用"全部签到"批量操作按钮快速将所有未处理学员标记为签到），系统记录每个签到学员的签到时间和签到操作教师。
  4. 若教师在课程开始后30分钟内未进行任何签到操作，系统自动向教师发送签到提醒通知。若课程结束后2小时仍有学员未标记任何状态，系统自动将这些学员标记为"缺勤"并发送确认通知给教师。
  5. 教师的签到操作具有时效性：课程开始前15分钟至课程结束后2小时内可进行签到修改，超过此时间窗口则签到状态锁定。若需修改锁定后的签到记录，教师需提交考勤修正申请由教务人员审批。
  6. 签到操作完成后，系统更新该课程安排的考勤统计缓存数据（已签到人数、请假人数、缺勤人数），并同步更新每个学员的个人出勤统计累计数据。
  7. 对于未签到的学员，系统自动向学员/家长发送"未出勤提醒"通知（若家长端开启了通知），提醒家长关注学员出勤情况。
- **输出**：签到完成提示，签到状态显示
- **业务规则**：
  - BR-ATT-001-01：签到操作必须在课程开始前15分钟至课程结束后2小时内完成。
  - BR-ATT-001-02：每次课程安排只有一次签到机会，不可对同一学员在同一课程安排中重复签到。
  - BR-ATT-001-03：课程结束2小时后未手动标记的学员自动改为"缺勤"状态。
  - BR-ATT-001-04：签到时间记录精确到分钟，用于判断学员是否迟到（签到时间晚于上课时间）。
  - BR-ATT-001-05："全部签到"操作仅对当前状态为"未签到"的学员生效，已标记为请假或缺勤的学员不受影响。
- **优先级**：高
- **关联涉众**：教师

##### REQ-ATT-002：请假申请

- **功能描述**：学员/家长提交请假申请
- **输入**：学员编号、请假日期（或课程安排编号）、请假原因
- **处理过程**：
  1. 学员/家长登录系统后，在"我的课程"或"课表"页面中看到即将到来的课程安排，每节课程旁边显示"请假"按钮（仅对未开始的课程可用）。
  2. 学员/家长点击"请假"按钮后，系统弹出请假申请表单，自动填充学员姓名、课程信息、上课日期和时间等只读信息。学员/家长需选择请假原因（从下拉列表中选择：病假、事假、其他，或手动输入详细原因），并可上传证明材料（如医院证明图片，可选）。
  3. 学员/家长提交请假申请后，系统校验请假申请的时限规则：常规请假需在课程开始前至少2小时提交（可在系统参数中配置），若距离课程开始不足2小时则标记为"紧急请假"并提示学员/家长联系教师说明情况。
  4. 系统创建请假申请记录（LeaveRequest表），状态初始设为"待审批"。系统向该班级的授课教师推送请假审批通知（消息中心+可选的短信通知），教师可在系统中查看请假申请的详细信息。
  5. 教师登录系统后在"待办事项"中看到请假审批通知，点击进入审批页面。教师查看请假原因和附件（如有），可选择"同意"或"拒绝"。若拒绝，需填写拒绝原因。审批结果实时更新请假申请状态。
  6. 若教师在课程开始前未处理请假申请，系统在课程开始后自动将状态为"待审批"的请假申请默认视为"同意"（避免因教师未及时审批导致学员考勤异常），但需在审批记录中标注"系统自动审批"。
  7. 请假申请审批通过后，系统自动将该学员当次课程的考勤状态从"未签到"改为"请假"，无需教师在签到面板上再手动处理。系统向学员/家长发送审批结果通知。
- **输出**：请假申请提交成功，审批结果通知
- **业务规则**：
  - BR-ATT-002-01：请假申请需在课程开始前提交，课程开始后不可提交请假申请。
  - BR-ATT-002-02：同一学员在同一课程安排中只能提交一次请假申请。
  - BR-ATT-002-03：若教师在课程开始前未处理审批，系统默认视为同意。
  - BR-ATT-002-04：已审批通过的请假记录不可修改或撤回。
  - BR-ATT-002-05：请假次数纳入学员出勤率计算（请假不计入缺勤，但会降低出勤率）。
- **优先级**：中
- **关联涉众**：学生/家长、教师

##### REQ-ATT-003：缺勤记录管理

- **功能描述**：管理学员的缺勤记录
- **输入**：课程安排编号、学员编号、缺勤原因
- **处理过程**：
  1. 系统在课程结束2小时后自动扫描该课程安排下所有状态仍为"未签到"且无有效请假审批的学员，将他们的考勤状态自动更新为"缺勤"。系统记录缺勤产生时间和产生方式（系统自动或教师手动标记）。
  2. 教师在课程结束后的48小时内可对缺勤记录进行补充编辑：为缺勤学员添加缺勤原因（从预设原因列表中选择或手动输入，如：无故缺勤、交通延误、忘记上课等）。
  3. 教务人员和教师在考勤管理页面可查看缺勤记录的汇总视图：按班级统计缺勤人数和缺勤率、按学员统计个人的缺勤次数和缺勤天数，以及最近缺勤明细列表。
  4. 系统持续监控每个学员的缺勤情况，当某学员连续缺勤次数达到预警阈值（可配置，默认为连续3次）时，系统自动触发缺勤预警通知（参见REQ-ATT-006）。
  5. 教务人员可通过"缺勤跟进"功能对频繁缺勤学员添加跟进备注记录，记录与家长/学员的沟通内容和结果，跟进记录关联缺勤记录保存。
  6. 缺勤记录关联至学员的考勤统计和出勤率计算，缺勤率 = 缺勤次数 / 应到次数 × 100%。高缺勤率学员在学员列表和报表中通过颜色标记突出显示。
  7. 缺勤记录数据作为退费处理（REQ-FEE-004）中已消耗课时计算的重要依据：只有签到记录计为已消耗课时，缺勤和请假时段的课时不计为已消耗。
- **输出**：缺勤记录
- **业务规则**：
  - BR-ATT-003-01：课程结束2小时后未签到且未请假的学员自动标记为缺勤。
  - BR-ATT-003-02：教师可在课程结束后48小时内补填缺勤原因，超时后缺勤记录锁定。
  - BR-ATT-003-03：缺勤记录不可删除，只能通过考勤修正流程（REQ-ATT-005）变更为其他状态。
  - BR-ATT-003-04：连续缺勤次数统计不包括请假记录，仅统计状态为"缺勤"的课程安排。
- **优先级**：中
- **关联涉众**：教师、教务人员

##### REQ-ATT-004：考勤统计

- **功能描述**：统计学员的出勤情况
- **输入**：学员编号或班级编号、统计时间段
- **处理过程**：
  1. 用户（教师、教务人员、机构管理员或学员/家长本人）进入考勤统计页面，选择统计范围：可按学员维度（查看单个学员的出勤统计）、按班级维度（查看整个班级的出勤概况）、按课程维度（查看某一课程下所有班级的出勤汇总）。
  2. 用户选择统计时间段（默认显示当前月份），系统查询该时间段内所选维度的所有课程安排记录和各学员的考勤状态（签到、请假、缺勤）。
  3. 系统计算以下核心指标：应到次数（该时间段内计划安排的课程总数）、实到次数（签到次数，含迟到但到课的情况）、请假次数（经审批通过的请假）、缺勤次数（既未签到也未请假的课程数）、出勤率（实到次数÷应到次数×100%，保留一位小数）。
  4. 系统以可视化图表展示考勤数据：出勤率趋势折线图（按日/周/月展示出勤率变化）、考勤状态饼图（签到/请假/缺勤的占比分布）、考勤热力图日历（以日历网格展示每天各课程的出勤情况，绿色为全员出勤、黄色为部分缺勤、红色为多人缺勤）。
  5. 考勤统计页面提供明细列表和汇总两种视图。明细列表显示每次课程安排的考勤情况（日期、课程、应到人数、实到人数、请假人数、缺勤人数、出勤率）。汇总视图显示时间段内的总计数据。
  6. 系统支持按多个维度交叉筛选考勤数据：按教师筛选（查看某教师所授课程的出勤情况）、按课程类型筛选、按日期范围筛选。筛选条件支持URL参数传递，方便用户收藏常用查询。
  7. 用户可将考勤统计结果导出为Excel格式文件，导出内容包括所有统计汇总数据和详细明细数据。导出的文件名默认包含统计维度、时间段等信息。
- **输出**：考勤统计报表
- **业务规则**：
  - BR-ATT-004-01：出勤率计算中，应到次数为已安排的所有课程数（含已取消课程？实际不含，仅含正常进行的课程）。
  - BR-ATT-004-02：请假记录计为"非出勤"，不计入实到次数但也不计入缺勤次数。
  - BR-ATT-004-03：学员维度考勤统计仅对"在读"状态学员开放，已退学学员的考勤数据仍保留供查询。
  - BR-ATT-004-04：考勤统计数据每日凌晨更新缓存，确保统计页面响应速度。
- **优先级**：中
- **关联涉众**：教务人员、教师、机构管理员

##### REQ-ATT-005：考勤补签

- **功能描述**：对因系统故障、教师失误或特殊原因遗漏签到的学员进行考勤补签
- **输入**：课程安排编号、补签学员编号列表、补签类型（签到/请假）、补签原因
- **处理过程**：
  1. 教师或教务人员在发现考勤记录有误（如学员实际到课但被误标为缺勤、学员口头请假但未在系统中提交申请等）时，可进入该课程安排的考勤详情页面，点击"补签"或"修正"按钮。
  2. 系统弹出考勤补签对话框，展示当前课程安排的原始考勤状态列表，每行学员旁边显示当前状态和"修正"操作按钮。补签操作受时间窗口限制：常规补签需在课程结束后48小时内完成，超时需提交考勤修正申请由教务主管审批。
  3. 教师/教务人员在考勤补签对话框中选择需要修正的学员（可多选），选择修正后的考勤状态（从"缺勤"改为"签到"、从"缺勤"改为"请假"、从"签到"改为"请假"等），从下拉列表中选择补签原因（如：教师操作遗漏、系统故障、学员特殊情况说明、数据修正等），并在备注栏补充详细说明。
  4. 系统校验补签操作的合法性：不可将"签到"状态改为"缺勤"（签到为已确认的到课事实），不可对课程结束超过30天的记录进行补签（超过时效性要求）。若补签类型为"缺勤→请假"，系统还检查该学员在该时段是否有提交请假申请记录。
  5. 若补签操作在常规时间窗口（48小时内）由授课教师提交，系统直接执行状态变更并记录补签日志。若超过时间窗口或由非授课教师提交，系统创建考勤修正审批待办任务，推送至教务主管审批。
  6. 教务主管在审批页面查看补签申请详情（原始状态、修正后状态、补签原因、提交人、提交时间），审批通过后系统执行状态变更；审批拒绝时需填写拒绝原因，补签申请退回提交人。
  7. 补签操作完成后，系统在考勤修正记录表中创建完整记录（包含修正前状态、修正后状态、修正原因、操作人、审批人、操作时间），同步更新学员的考勤统计数据（出勤率、出勤次数等）。
- **输出**：考勤补签完成，考勤记录更新
- **业务规则**：
  - BR-ATT-005-01：补签操作的常规时间窗口为课程结束后48小时内，超出需审批。
  - BR-ATT-005-02：不可将"签到"状态改为"缺勤"，到课事实不可被否认。
  - BR-ATT-005-03：课程结束后超过30天的考勤记录锁定，不允许任何形式的修正。
  - BR-ATT-005-04：每次补签操作必须填写补签原因，原因字段不能为空且不少于5个字符。
- **优先级**：中
- **关联涉众**：教师、教务人员

##### REQ-ATT-006：考勤预警

- **功能描述**：对学员的出勤异常情况进行自动检测和分级预警
- **输入**：预警触发条件（连续缺勤次数、出勤率阈值等，可配置）
- **处理过程**：
  1. 系统在每个课程日结束后自动执行考勤预警检测任务，扫描所有在读学员的近期考勤数据，根据预设的预警规则识别出需要关注的异常学员。
  2. 系统支持多级预警阈值配置（由机构管理员在系统参数中设置，默认值）：黄色预警（连续缺勤2次）、橙色预警（连续缺勤3次或月度出勤率低于70%）、红色预警（连续缺勤5次或月度出勤率低于50%）。不同级别的预警触发不同的响应策略。
  3. 当学员触发黄色预警时，系统自动向学员/家长发送"出勤关注提醒"通知，提示近期有缺勤情况，建议关注出勤。同时向授课教师推送预警通知，提醒教师关注该学员的学习状态。
  4. 当学员触发橙色预警时，系统向教务人员推送"学员出勤异常"待办任务，要求教务人员在48小时内与学员/家长取得联系，了解缺勤原因并进行跟进。教务人员需在系统中记录沟通结果和后续处理方案。
  5. 当学员触发红色预警时，系统在教务主管和机构管理员的仪表盘上以醒目的红色标记展示该预警项，同时自动生成"严重缺勤学员报告"。系统建议教务主管安排面谈或家访，并在系统中记录干预措施。
  6. 预警信息在系统中的展示位置包括：学员详情页的考勤区域（预警标签+预警级别颜色）、班级学员列表中该学员行的背景色变化（按照预警级别着色）、仪表盘页面的"考勤预警"专题卡片（展示各级别预警学员数量和Top问题学员）。
  7. 系统保留完整的预警历史记录：每次预警的触发时间、预警级别、触发条件（具体的考勤数据）、响应措施和处理结果。教务人员可查看每个学员的预警历史追溯，分析出勤问题的演变趋势。
- **输出**：考勤预警通知，预警待办任务，严重缺勤报告
- **业务规则**：
  - BR-ATT-006-01：考勤预警规则中的连续缺勤次数仅统计状态为"缺勤"的记录，请假不计入连续缺勤。
  - BR-ATT-006-02：预警级别递进规则：当学员同时满足多个级别条件时，取最高级别预警。
  - BR-ATT-006-03：预警处理完成后（教务人员记录处理结果），该级别预警标记为"已处理"，但后续若再次触发可重新激活。
  - BR-ATT-006-04：学员转班或退学后，其关联的未处理预警任务自动关闭。
- **优先级**：中
- **关联涉众**：教务人员、机构管理员、教师、学生/家长

### 3.7 报表统计模块

#### 3.7.1 功能描述

报表统计模块为机构管理层和财务人员提供多维度的数据统计和分析功能，支持数据驱动的运营决策。系统提供财务收支报表、课时统计报表、学员统计报表、欠费统计报表、运营分析仪表盘和教师绩效报表六大类统计功能，每类报表均支持图表可视化展示和Excel/PDF多格式导出。

#### 3.7.2 功能需求

##### REQ-RPT-001：财务日报/月报

- **功能描述**：生成指定时间段的财务收支报表
- **输入**：报表类型（日/月/季度/年）、日期、校区（可选）
- **处理过程**：
  1. 用户（财务人员或机构管理员）进入报表统计模块，选择"财务报表"类型，然后从下拉列表中选择报表周期（日报、月报、季报、年报），系统根据所选周期自动设置默认日期范围（日报默认昨日、月报默认上月、季报默认上季度、年报默认去年）。
  2. 用户可选择性地指定校区范围（全部校区或特定校区），系统根据数据权限自动限制财务人员只能查看其所辖校区的数据，机构管理员可查看所有校区。
  3. 系统统计选定时间段内的财务核心数据：收费总笔数和收费总金额（按收费项目分类统计）、退费总笔数和退费总金额、净收入（收费总额 - 退费总额）、按支付方式分类的交易统计（现金、微信支付、支付宝、银行转账各自的笔数和金额）、按课程类型分类的收费统计。
  4. 系统自动计算环比数据：将当前周期的各项指标与上一周期进行对比，计算环比增长率和绝对变化值，以绿色（增长）或红色（下降）箭头标识变化方向。
  5. 报表以可视化图表和表格结合的方式展示：顶部为关键指标卡片（收费总额、退费总额、净收入、环比变化），中部为趋势折线图（按日期展示每日收费金额变化），底部为分类饼图和分类明细表格（按收费项目、支付方式、课程类型维度）。
  6. 用户可切换报表的展示粒度：日报可下钻到按小时统计，月报可下钻到按日统计，年报可下钻到按月统计。下钻操作通过点击图表数据点触发，加载更细粒度的数据。
  7. 报表支持一键导出为Excel格式（含完整数据和图表图片）或PDF格式（A4排版打印就绪）。导出文件自动包含报表标题、统计周期、生成时间、制表人等元信息。
- **输出**：财务统计报表（含图表）
- **业务规则**：
  - BR-RPT-001-01：财务日报默认统计昨日0:00至23:59的数据，月报统计上月1日至月末最后一日。
  - BR-RPT-001-02：净收入 = 收费总额 - 退费总额，不包括优惠减免部分（优惠在收费环节已扣除）。
  - BR-RPT-001-03：报表数据基于系统中已日结的收费记录（未日结的不纳入正式报表）。
  - BR-RPT-001-04：报表导出文件自动加上水印包含导出人和导出时间信息。
- **优先级**：高
- **关联涉众**：财务人员、机构管理员

##### REQ-RPT-002：课时统计报表

- **功能描述**：统计各课程、各教师的课时消耗情况
- **输入**：统计时间段、校区、教师/课程筛选
- **处理过程**：
  1. 用户选择统计时间段（默认当月），可通过筛选器指定校区范围、教师范围（特定教师或全部）、课程范围（特定课程类型或全部）。
  2. 系统查询选定条件内所有状态为"已完成"的课程安排记录，统计以下维度的数据：按教师维度统计每位教师的授课总课时数（实际授课分钟数÷标准课时分钟数）、按课程类型维度统计各课程的累计授课课时数、按班级维度统计各班级的课时完成进度（已完成课时÷计划总课时）。
  3. 系统计算每位教师的课酬金额预估：各课程类型的课时数 × 对应的课酬单价，汇总得到应发课酬预估值。此数据供财务人员和HR参考，不作为实际发放依据（最终以课时统计确认的REQ-TCH-003数据为准）。
  4. 系统生成课时利用率分析：统计各教室的时间利用率（已排课时长÷可用时长）、各教师的课时饱和度（已授课时数÷可授课时数上限），帮助教务人员优化排课策略。
  5. 报表以可视化形式展示：表格视图（教师课时排行Top N）、柱状图（各课程类型课时分布）、进度条（各班级课时完成率）。
  6. 用户可点击表格中的教师姓名或班级名称下钻查看详细数据：教师详情展示每日授课记录和课时明细，班级详情展示每次课程安排的时间和课时信息。
  7. 支持导出为Excel格式（含详细数据和汇总图表），导出文件名包含统计周期和筛选条件。
- **输出**：课时统计报表
- **业务规则**：
  - BR-RPT-002-01：课时统计仅统计状态为"已完成"的课程安排（已上课并完成考勤登记）。
  - BR-RPT-002-02：课酬金额为预估值，实际发放前需经教师确认和财务审核。
  - BR-RPT-002-03：课时利用率超过100%表示教师实际授课超出可用时长，需教务人员关注排课合理性。
- **优先级**：中
- **关联涉众**：教务人员、机构管理员

##### REQ-RPT-003：学员统计报表

- **功能描述**：统计学员的结构和变化趋势
- **输入**：统计时间段、校区
- **处理过程**：
  1. 用户选择统计时间段和校区范围，系统查询学员信息表生成学员结构统计数据。
  2. 系统计算核心指标：时间段内新增学员数（按日报/周报/月报粒度）、退学学员数、当前在读学员总数、净增长率（新增 - 退学）/期初在读 × 100%。
  3. 系统按多个维度生成学员结构分析：按课程类型统计学员分布（各课程的在读学员数和占比）、按年龄/年级统计学员分布（年龄段分布柱状图）、按来源渠道统计学员获客来源（网络推广、老学员推荐、地推活动等到店渠道的占比饼图）、按校区统计学员分布。
  4. 系统生成学员留存趋势分析：统计各批次学员（按月入学的学员群组）在不同时间点的留存率，以留存曲线图展示学员流失趋势，帮助机构管理层评估教学和服务质量。
  5. 报表以可视化图表和表格结合展示：关键指标卡片（在读总数、本月新增、本月退学、净增长率）、学员来源渠道饼图、学员课程分布柱状图、学员年龄/年级分布直方图、学员留存率折线图。
  6. 用户可下钻到具体数据列表：点击"新增学员"指标可查看新增学员详细名单，点击某课程类型的学员数可查看该课程下所有学员列表。
  7. 支持导出为Excel格式（含统计数据、图表和明细数据）、PDF格式（A4排版打印就绪）。
- **输出**：学员统计报表
- **业务规则**：
  - BR-RPT-003-01：新增学员为统计周期内状态首次变为"在读"的学员（从"待分班"通过分班操作进入"在读"状态）。
  - BR-RPT-003-02：退学学员为统计周期内状态变为"已退学"的学员。
  - BR-RPT-003-03：学员留存率数据按月群组计算，展示第1个月、第3个月、第6个月和第12个月的留存百分比。
  - BR-RPT-003-04：来源渠道数据来源于学员信息录入时选择的来源渠道字段。
- **优先级**：中
- **关联涉众**：机构管理员

##### REQ-RPT-004：欠费统计报表

- **功能描述**：统计当前的欠费情况
- **输入**：校区
- **处理过程**：
  1. 用户选择校区范围（默认全部校区），系统实时查询当前所有在读学员的缴费状态和欠费信息。
  2. 系统统计欠费核心指标：当前欠费总金额（所有在读学员未缴费用的总和）、欠费学员总人数、欠费学员占在读学员的比例、平均欠费金额（欠费总金额÷欠费人数）、与上月同期数据的环比变化。
  3. 系统按欠费天数分级统计欠费学员的分布：欠费1-7天人数和金额、欠费8-15天人数和金额、欠费16-30天人数和金额、欠费30天以上人数和金额（严重欠费）。各级别以渐变色条带展示，从浅黄到深红标识严重程度。
  4. 系统生成欠费Top N学员明细列表（默认Top 20），按欠费金额从高到低排序，每条记录显示学员姓名、所在班级、联系电话、欠费金额、欠费天数、最近催缴日期和催缴状态。
  5. 系统按班级维度和课程维度统计欠费分布：哪些班级/课程的欠费率和欠费金额最高，帮助机构识别可能存在的收费管理薄弱环节。
  6. 报表以可视化形式展示：关键指标卡片、欠费分级分布饼图或仪表盘、欠费Top学员表格、按班级欠费排行柱状图。
  7. 财务人员可点击欠费学员姓名进入学员详情页查看完整的缴费历史和催缴记录，可在报表页面直接批量选择学员触发一键催缴操作（手动发送催缴通知）。
- **输出**：欠费统计报表
- **业务规则**：
  - BR-RPT-004-01：欠费金额计算基于学员应缴费用的汇总与已缴费用的差额，精确到0.01元。
  - BR-RPT-004-02：欠费统计报表数据实时计算（不使用缓存），确保数据的即时准确性。
  - BR-RPT-004-03：被标记为"免催缴"的学员仍在报表中统计，但以特殊颜色（灰色）标识。
- **优先级**：中
- **关联涉众**：财务人员、机构管理员

##### REQ-RPT-005：运营分析仪表盘

- **功能描述**：为机构管理层提供一站式的运营数据概览仪表盘
- **输入**：统计时间范围（今日/本周/本月/自定义）、校区（可选）
- **处理过程**：
  1. 机构管理员或教务主管登录系统后进入"运营仪表盘"页面，系统默认展示今日实时运营数据的概览视图。
  2. 系统聚合多个业务模块的关键数据并在单一页面上以仪表盘卡片形式展示：今日课程安排总数、已上课数、待上课数；今日实收金额（实时更新）；当前在读学员总数；今日出勤率（已签到人数÷应到人数）；当月新增学员数；当前欠费学员数和欠费总金额。
  3. 仪表盘提供实时更新的趋势图表：近30日收入趋势折线图（每日实收金额变化）、近30日学员增长趋势图（柱状图展示每日新增和退学）、当前各校区在读学员分布地图或柱状图、近7日出勤率趋势折线图。
  4. 仪表盘展示待办事项提醒区域：待审批的请假申请数量、待处理的退费申请数量、当日未进行签到的课程数量、即将到期的课程安排提醒。每个待办项均可点击跳转至对应的处理页面。
  5. 仪表盘支持自定义布局：管理员可通过拖拽方式调整仪表盘上各卡片和图表的排列位置和显示大小，系统保存个性化布局配置到用户偏好设置中。
  6. 系统每5分钟自动刷新仪表盘上的实时数据（如当日收费金额、出勤率），历史趋势图表数据每小时更新一次。用户可手动点击"刷新"按钮立即获取最新数据。
  7. 仪表盘支持全屏模式展示（适用于在机构大厅屏幕或管理办公室大屏上实时展示运营数据），全屏模式下自动轮播各数据视图。
- **输出**：运营分析仪表盘页面
- **业务规则**：
  - BR-RPT-005-01：仪表盘数据刷新间隔为5分钟（实时指标）和1小时（趋势图表）。
  - BR-RPT-005-02：仪表盘上的收费金额数据为未日结的实时数据，仅供运营参考，不得作为财务对账依据。
  - BR-RPT-005-03：仪表盘布局配置与用户账号绑定，不同用户可有不同的仪表盘布局。
  - BR-RPT-005-04：仪表盘数据权限与用户角色绑定，教务人员只能看到所辖校区的数据。
- **优先级**：高
- **关联涉众**：机构管理员

### 3.9 外部接口需求

#### 3.9.1 用户界面需求

| 要求项 | 说明 |
|--------|------|
| 响应式设计 | 前端界面需支持PC端（1920×1080、1366×768）和平板（iPad）显示 |
| 操作一致性 | 所有列表页面的操作按钮位置一致，所有表单页面的布局风格一致 |

| 快捷键 | 常用操作支持键盘快捷键（Ctrl+S保存、Ctrl+F搜索等） |
| 通知提示 | 操作成功/失败的消息提示需显著、3秒后自动消失 |

#### 3.9.2 API接口需求

- 所有API接口必须遵循RESTful设计规范
- 接口使用JSON格式进行数据交换
- 接口必须支持分页参数（page、size）和排序参数（sort、order）
- 接口必须返回统一的响应格式：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {},
    "timestamp": "2026-06-22T10:30:00+08:00"
  }
  ```
- 所有接口需进行身份认证（JWT Token）
- 敏感接口（收费、退费等）需进行权限校验
- 接口文档使用OpenAPI 3.0.3规范编写

### 3.10 非功能需求

#### 3.10.1 性能需求

| 需求编号 | 需求描述 | 指标 |
|---------|---------|------|
| NF-PERF-001 | 页面加载时间 | 普通页面≤2秒，报表页面≤5秒 |
| NF-PERF-002 | API响应时间 | 查询接口≤500ms，写入接口≤1s，报表接口≤3s |
| NF-PERF-003 | 并发支持 | 系统需支持至少50个用户同时在线操作 |
| NF-PERF-004 | 数据容量 | 支持10,000+学员、100,000+收费记录的数据量 |
| NF-PERF-005 | 数据库查询 | 大表查询（>10万条）需建立合理索引，确保查询效率 |

性能需求的详细说明：

- **NF-PERF-001页面加载时间**：所有页面的首次内容渲染时间（FCP）不超过2秒，交互就绪时间（TTI）不超过3秒。报表类页面因涉及多表关联查询和图表渲染，允许不超过5秒的加载时间。超过5秒的报表页面需展示加载进度条和预估完成时间。前端通过路由懒加载、组件异步加载和静态资源CDN加速等手段优化页面加载性能。
- **NF-PERF-002 API响应时间**：90%的API请求需在目标时间内完成响应（P90指标）。查询类接口要求数据库查询使用索引覆盖，写入类接口的事务范围尽可能缩小以减少锁等待时间。报表类接口允许最长3秒的响应时间，超过3秒的报表查询需考虑异步处理（将报表任务提交到后台队列，完成后通知用户下载）。
- **NF-PERF-003并发支持**：系统需支持至少50个用户同时在线并发操作，在50并发用户的压测场景下，API的平均响应时间不应超过单用户场景的2倍。并发瓶颈点（如数据库连接池、Redis连接池）需合理配置上限。使用数据库连接池（HikariCP）管理连接，连接池大小按公式（CPU核心数×2+磁盘数）配置。
- **NF-PERF-004数据容量**：系统数据库设计需支持10000+学员记录、100000+收费记录、50000+考勤记录的数据量规模。关键业务表需设计合理的分区策略（如按年份或校区对收费记录表进行水平分区），历史数据归档方案需在数据库设计中预留接口。
- **NF-PERF-005数据库查询**：所有涉及大表的查询必须在对应的WHERE条件字段上建立索引，特别是高频查询字段（学员姓名、联系电话、学员编号、班级编号、日期范围等）。联合索引的顺序需与查询条件的使用频率和选择性匹配。定期执行数据库慢查询分析，对执行时间超过200ms的SQL进行优化。

#### 3.10.2 安全性需求

| 需求编号 | 需求描述 |
|---------|---------|
| NF-SEC-001 | 用户密码使用bcrypt算法加密存储 |
| NF-SEC-002 | 用户登录使用JWT Token认证，Token有效期24小时 |
| NF-SEC-003 | 敏感操作（收费、退费、用户管理）需进行二次权限校验 |
| NF-SEC-004 | SQL注入防护：所有数据库操作使用参数化查询 |
| NF-SEC-005 | XSS防护：所有用户输入进行HTML转义处理 |
| NF-SEC-006 | CSRF防护：状态变更操作需验证CSRF Token |
| NF-SEC-007 | 登录失败限流：同IP连续5次登录失败后锁定15分钟 |

安全性需求的详细说明：

- **NF-SEC-001密码安全**：用户密码在存储前使用bcrypt算法进行哈希处理，盐值随机生成且不可逆。bcrypt的工作因子（cost factor）设置为10或以上，确保暴力破解的计算成本足够高。密码传输过程中使用HTTPS加密，前端绝不缓存或明文存储用户密码。
- **NF-SEC-002 JWT认证**：用户登录成功后，服务端签发JWT Token，Token中包含用户ID、用户名、角色列表等基本声明信息。Token有效期为24小时，支持刷新机制（Refresh Token有效期7天）。Token在每次API请求中通过Authorization Header传递，后端中间件校验Token签名和有效期。Token黑名单机制：用户登出或密码修改后，对应的Token加入Redis黑名单。
- **NF-SEC-003二次权限校验**：对于收费、退费、用户管理、权限变更等高风险操作，系统在每次执行前不仅检查用户的角色权限，还需验证操作是否在当前用户的授权范围内（如退费金额是否超过用户的审批额度、用户管理操作是否在自己的校区范围内），防止权限滥用和越权操作。
- **NF-SEC-004 SQL注入防护**：所有数据库查询操作统一使用ORM框架（MyBatis）的参数化查询机制（#{参数}语法），禁止使用字符串拼接SQL。代码审查中需检查是否存在动态SQL拼接的风险点，特别是ORDER BY、GROUP BY等不支持参数化查询的场景，需使用白名单校验。
- **NF-SEC-005 XSS防护**：前端对所有用户输入数据进行HTML实体编码后再展示，禁止直接使用v-html渲染用户输入内容。后端在存储前进行输入清洗（去除script标签、事件处理器等危险内容），API返回的JSON数据中的字符串字段需进行HTML转义后再返回给前端展示。
- **NF-SEC-006 CSRF防护**：所有状态变更操作（POST、PUT、DELETE方法）需验证请求中的CSRF Token。CSRF Token由服务端在用户登录后签发并与用户Session关联，前端在每次请求中自动附加Token。系统需在响应的Cookie中设置SameSite=Strict属性作为辅助防护。
- **NF-SEC-007登录限流**：系统使用Redis记录每个IP地址的登录失败次数和最近失败时间。同一IP地址在15分钟内连续登录失败5次后，该IP被临时锁定15分钟，期间所有登录请求均返回"账户已锁定，请稍后再试"。锁定期间不影响其他IP的正常登录。支持配置白名单IP（如机构内网IP），白名单IP不受限流策略影响。

| 需求编号 | 需求描述 |
|---------|---------|
| NF-AVAIL-001 | 系统可用性≥99.5%（非教学时段可安排维护窗口） |
| NF-AVAIL-002 | 单点故障恢复时间≤2小时 |
| NF-AVAIL-003 | 数据每日自动备份 |
| NF-AVAIL-004 | 关键错误发生时需通过邮件通知管理员 |

可用性需求的详细说明：

- **NF-AVAIL-001系统可用性**：系统的年化可用性目标为99.5%，即年度计划外停机时间不超过43.8小时（约1.8天）。计划维护窗口安排在非教学高峰时段（如每日凌晨2:00-5:00或寒暑假间隙期），维护窗口内的停机不计入可用性指标。系统部署中需考虑应用服务的健康检查机制和自动重启机制，当检测到服务异常时在30秒内自动恢复。
- **NF-AVAIL-002故障恢复**：单点故障（如应用服务进程崩溃、数据库连接池耗尽等）的检测和恢复时间目标为2小时以内。系统需部署监控告警（如Prometheus + Grafana），在检测到服务不可用、错误率超过5%、响应时间超过阈值的1分钟内发出告警。数据库层面建议配置主从复制（Master-Slave），当主库故障时可快速切换到从库。
- **NF-AVAIL-003数据备份**：系统每日凌晨3:00自动执行全量数据库备份，备份文件保留最近7天版本。同时每小时生成数据库binlog的增量备份，用于在灾难场景下恢复到任意时间点。备份文件需异地存储（上传到云存储OSS或独立于应用服务器的备份服务器），防止服务器硬件故障导致备份和源数据同时丢失。
- **NF-AVAIL-004错误通知**：系统集成邮件通知服务，当发生以下关键错误时自动发送告警邮件给系统管理员和运维人员：应用服务异常崩溃重启、数据库连接失败、第三方支付API调用连续失败3次以上、磁盘空间使用率超过80%、系统错误日志数量在短时间内急剧增长（异常突增超过日常水平5倍）。告警邮件包含错误详情、发生时间、影响范围等关键信息。

其他非功能需求的详细说明：

- **NF-OTH-001时区统一**：系统服务器时区统一设置为Asia/Shanghai（UTC+8，北京时间），数据库时区与服务器保持一致。所有业务时间字段（创建时间、修改时间、业务日期等）均以北京时间存储和展示。API交互中的时间字段使用ISO 8601格式并带+08:00时区偏移量，如"2026-06-22T10:30:00+08:00"。定时任务的cron表达式以服务器时区为准进行配置。
- **NF-OTH-002金额精度**：所有金额字段以分为单位存储在数据库中（BIGINT类型），或使用DECIMAL(10,2)类型精确存储到分。金额计算使用Java的BigDecimal类进行，严禁使用float或double参与金额计算以避免浮点精度误差。金额展示时以元为单位显示，保留两位小数，如"1,280.50元"。
- **NF-OTH-003日期格式**：系统所有用户可见的日期均以"YYYY-MM-DD"格式展示（如"2026-06-22"）。日期选择器组件统一使用该格式进行数据的输入和输出。API交互中的日期字段使用ISO 8601日期格式"YYYY-MM-DD"。日期相关的数据库字段统一使用DATE类型存储。
- **NF-OTH-004数字格式**：所有金额和数量类数字在展示时使用千分位逗号分隔，如学员总数显示为"1,280"人，收费金额显示为"15,800.00元"。千分位格式根据用户的语言偏好自动适配（中文环境使用千分位逗号）。数字输入框中不显示千分位分隔符。
- **NF-OTH-005 Excel导出**：所有报表和数据列表均支持导出为Excel（.xlsx）格式。导出功能使用Apache POI库实现，支持大数据量导出（最多100,000行）。Excel文件包含表头样式（粗体、居中、背景色）、自动列宽和数字格式。导出文件名包含数据内容和导出时间信息，如"收费台账_20260601-20260622.xlsx"。

### 4.3 数据模型概述

#### 4.3.1 核心实体关系描述

EduFeeMS系统的核心数据模型围绕以下主要业务实体构建，实体间的核心关系如下：

- **校区（Campus）** 是数据隔离的顶层实体，一个机构可以有多个校区。教室、教师、班级、学员均隶属于特定校区。
- **课程（Course）** 是教学产品的定义实体，包含课程类型的基本信息、课时配置和费用标准。一个课程类型可以对应多个班级实例。
- **班级（Class）** 是课程的具体开课实例，关联了课程类型、授课教师、教室和校区。一个班级包含多个学员，通过学员-班级关联表建立"多对多"关系（考虑转班历史）。
- **学员（Student）** 是系统的核心实体，学员可以依次属于多个班级（通过分班和转班操作），每次班级归属变化均记录在学员班级历史表中。
- **教师（Teacher）** 关联可教授课程类型（多对多），被分配到班级进行授课。教师通过课程安排记录与具体的上课时间和班级关联。
- **排课（Schedule）** 记录班级的每次课程安排，关联了班级、教室、教师、日期和时间段信息。
- **考勤（Attendance）** 记录了每次课程安排中每个学员的出勤状态（签到/请假/缺勤），关联排课记录和学员。
- **收费记录（FeeRecord）** 记录每一笔收费操作，关联学员、收费项目和支付方式。退费记录（RefundRecord）关联原始收费记录。
- **收据（Receipt）** 与收费记录一对一关联，记录收据编号和打印信息。
- **发票（Invoice）** 与收费记录一对一关联，记录发票编号、类型、抬头和开票状态。

实体间的关键关系图示（文字描述）：
- Campus 1:N Classroom, Campus 1:N Teacher, Campus 1:N Class, Campus 1:N Student
- Course 1:N Class, Course 1:N FeeConfig (versioned)
- Class 1:N Schedule, Class N:M Student (via StudentClass)
- Teacher N:M Course (via TeacherCourse), Teacher 1:N Schedule
- Schedule 1:N Attendance
- Student 1:N Attendance, Student 1:N FeeRecord, Student 1:N LeaveRequest
- FeeRecord 1:1 Receipt, FeeRecord 1:1 Invoice, FeeRecord 1:1 RefundRecord

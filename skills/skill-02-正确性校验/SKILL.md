# 业务正确性校验

## 概述

业务正确性校验子技能用于验证代码实现中的关键业务逻辑是否严格遵循SRS中定义的业务规则。该技能深入分析源代码的计算逻辑、数据处理流程和边界条件处理，与SRS中的业务规则（BR-xxx）逐一对比，标注偏差项并评估偏差的风险等级。特别关注金额计算精度、日期时间处理、事务管理正确性等财务系统关键领域。

该技能是EduFeeMS项目六级逆向校验体系中的第二级（L2-正确性维度），旨在确保关键业务逻辑的实现准确性，防止因计算错误、精度丢失或事务不当导致的财务数据差错。

## 输入要求

| 输入项 | 类型 | 是否必需 | 说明 |
|--------|------|----------|------|
| 源代码路径 | 目录路径 | 是 | 项目源代码根目录 |
| SRS文档路径 | 文件路径 | 是 | 包含所有BR-xxx业务规则的SRS文档 |

| 三层约束文档 | 文件路径 | 否 | TLCD三层约束设计文档路径，用于交叉验证 |

## 处理流程

### 步骤1：业务规则提取

从SRS文档中提取所有业务规则（BR-xxx），解析每条规则的关键要素：

1. **规则编号**：如 BR-STU-004-03、BR-FEE-005-02
2. **规则描述**：业务约束的具体内容
3. **关联需求**：该规则关联的REQ需求编号
4. **校验类型**：
   - **数值计算**：涉及金额、比例、折扣等数值运算
   - **状态流转**：涉及状态机、生命周期管理
   - **数据约束**：涉及字段格式、取值范围、唯一性
   - **流程约束**：涉及操作顺序、前提条件、后置条件
   - **权限控制**：涉及角色权限、操作授权

### 步骤2：代码逻辑提取

在源代码中定位与业务规则相关的实现代码：

1. **Service层扫描**：提取所有`*ServiceImpl.java`中与业务规则相关的public方法
2. **计算逻辑提取**：重点关注以下模式的代码：
   - `BigDecimal`运算（add、subtract、multiply、divide）
   - 条件判断（if-else、switch、三元运算符）
   - 状态变更（setStatus、updateStatus）
   - 事务注解（@Transactional及其属性）
   - 日期处理（LocalDate、LocalDateTime的操作）
3. **方法调用链追踪**：分析跨Service调用时的数据流转和状态变化

### 步骤3：规则-代码对比

逐条对照SRS业务规则与代码实现进行对比校验：

#### 3.1 金额计算校验（优先级最高）

| 检查项 | 校验方法 | 常见问题 |
|--------|----------|----------|
| BigDecimal使用 | 搜索`float`、`double`关键字，确认金额字段全部使用BigDecimal | 混用float/double导致精度丢失 |
| 精度控制 | 检查divide操作是否指定RoundingMode | 未指定导致ArithmeticException |
| 退费金额约束 | 检查退费逻辑中的金额比较（退费≤已收） | 退费金额可能超过已收金额 |
| 优惠计算 | 检查折扣计算顺序和叠加规则 | 优惠叠加后金额异常 |
| 四舍五入 | 确认最终金额使用HALF_UP模式取2位小数 | 舍入模式不一致导致1分钱差异 |

#### 3.2 日期处理校验

| 检查项 | 校验方法 | 常见问题 |
|--------|----------|----------|
| 日期类型 | 确认使用LocalDate/LocalDateTime而非Date | 使用旧Date API导致时区问题 |
| 日期范围校验 | 检查日期比较逻辑（开班≤当前日期、退学≥入班日期） | 日期倒挂逻辑未校验 |
| 时区一致性 | 确认统一使用北京时间（UTC+8） | 服务器时区不一致 |
| 日期格式化 | 确认序列化使用统一格式 | 前端接收格式不一致 |

#### 3.3 事务管理校验

| 检查项 | 校验方法 | 常见问题 |
|--------|----------|----------|
| 写操作事务 | 检查所有写数据的Service方法是否有@Transactional | 数据不一致风险 |
| 事务传播 | 检查跨Service调用的事务传播行为 | 嵌套事务导致数据不一致 |
| 事务隔离 | 检查涉及收费、退费的高并发操作 | 并发收费导致金额错误 |
| 回滚策略 | 检查异常处理中的事务回滚配置 | 异常被吞导致事务未回滚 |

#### 3.4 状态流转校验

检查状态机实现的正确性：
- 学员状态：待分班→在读→已退学（不可逆）
- 班级状态：招生中→开课中→已结课（单向不可逆）
- 缴费状态：未缴→部分缴纳→已缴清/已退费
- 请假审批：待审批→已通过/已驳回

### 步骤4：偏差标注与风险评估

对每个发现的偏差进行分级标注：

| 偏差等级 | 标识 | 定义 | 处理要求 |
|----------|------|------|----------|
| 严重（CRITICAL） | C | 违反强制约束，可能导致财务数据错误或系统崩溃 | 必须立即修复 |
| 高（HIGH） | H | 违反高优先级规则，可能影响核心业务流程 | 应在当前迭代修复 |
| 中（MEDIUM） | M | 违反建议约束，存在潜在风险 | 计划在后续迭代修复 |
| 低（LOW） | L | 可实现性改进，不直接影响功能正确性 | 可延后处理 |

## 使用示例

### 示例1：收费模块正确性校验

```
输入：
  - 源代码路径：src/backend/edu-fee/
  - SRS文档路径：wiki/baselines/BL-20260622-01/SRS-正式版.md
  - 重点检查项：["金额计算", "事务管理", "退费逻辑"]

处理流程：
1. 从SRS中提取所有BR-FEE-xxx业务规则（约20条）
2. 扫描FeeServiceImpl.java提取收费和退费相关方法
3. 检查refund()方法中退费金额是否校验不超过已收金额
4. 检查createRecord()方法中BigDecimal运算是否正确
5. 确认@Transactional注解覆盖所有写操作
6. 验证流水号生成逻辑的格式和唯一性

输出：
  - 发现fee模块2个HIGH级别偏差：
    1. FeeServiceImpl.refund()中退费金额校验使用了double比较（应使用BigDecimal.compareTo）
    2. FeeServiceImpl.createRecord()中优惠计算顺序与SRS规则不一致
```
